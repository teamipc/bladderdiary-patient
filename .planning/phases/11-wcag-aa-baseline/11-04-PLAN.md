<objective>
Extend `e2e/a11y.spec.ts` to verify Phase 11's full WCAG 2.1 AA contract end-to-end. This is the only end-to-end proof that the success criterion 5 from ROADMAP "axe-core sweep across 6 locales × N viewports on diary day 1 / summary / landing / one learn article reports 0 WCAG 2.1 AA violations" holds.

**Why extend the existing spec, not create a new one:**
The existing `e2e/a11y.spec.ts` already runs in a dedicated Playwright project `a11y` (per `playwright.config.ts:128-140`), already loops over all 6 locales, already uses `@axe-core/playwright` with the WCAG tag chain, already seeds a complete diary state via `buildSeedState` for the summary scan, and already writes findings JSON to `test-results/walkthrough/findings/a11y.json`. This plan ADDS 3 new routes (diary day 1, learn topic, learn article) to the existing per-locale loop, plus 4 targeted per-criterion assertions for A11Y-01..04. The output JSON shape is preserved by extending the `byLocale[L]` record from `{ homepage, summary }` to `{ homepage, diaryDay1, summary, learnTopic, learnArticle }`.

**Matrix scope decision (per CONTEXT planning question 5):**
The audit recommended "6 locales × 3 viewports × 5 routes = 90 test cases". The CONTEXT planning recommendation was "6 × 1 viewport × 5 routes = 30 cases for the regression spec". This plan adopts the 30-case sampling: ONE viewport (the existing 390×844 iPhone 14 dimensions already configured in `playwright.config.ts` for the a11y project) × 6 locales × 5 routes. Rationale:
- WCAG violations are almost entirely viewport-independent at the rules axe-core checks (heading structure, ARIA validity, live regions, skip-link presence, focus management, color contrast). The handful of viewport-dependent rules (e.g., "scrollable region has no keyboard accessible content" at narrow widths) are caught by the iPhone-mobile-default that already drives the daily walkthrough.
- The Phase 6 + Phase 8 verification specs already covered the desktop + tablet visual+keyboard matrix. Phase 11's job is WCAG semantic correctness, not responsive behavior.
- A 90-case run would take 5-7× longer than the 30-case run with marginal additional signal. The 30-case run is the canonical Phase 11 regression spec; an "expanded matrix" run can be invoked manually if a future change suggests viewport-specific risk.

**5 routes per locale:**
1. `/` (or `/<locale>` for non-EN) — the landing page (existing in the spec).
2. `/diary/day/1` (or `/<locale>/diary/day/1`) — the most-used surface, A11Y-01 sign-off route. Seeded with diary-started state.
3. `/summary` (or `/<locale>/summary`) — existing, already seeded with a complete 3-day diary.
4. `/learn/voiding` (or `/<locale>/learn/voiding`) — a learn topic pillar page. Coverage for the Learn section axe scan (the audit found I4/I5/I6 in the learn surface as well).
5. `/learn/voiding/<a-real-slug>` — a single learn article page. The slug needs to exist in `content/articles/<locale>/voiding/`. Use `feeling-bladder-is-not-empty.mdx` which is a known-shipping article confirmed in the audit's live-curl spot-check.

If a specific locale × topic combination is not built (e.g., a non-EN locale lacks the voiding topic), the test SHOULD skip that one route with a `test.skip()` plus a console note — the spec must not hard-fail on missing content. The 5-route × 6-locale matrix should produce at least 28 of 30 scans even in worst-case content gaps.

**A11Y-01 targeted assertion (page-has-heading-one):**
For each scan, after running axe-core, assert `page.locator('h1').count() === 1`. axe-core already includes `page-has-heading-one` in its wcag2aa rule set, BUT that rule fires only at "moderate" impact and may be filtered by some default configurations — making it an explicit `expect()` ensures the regression is caught even if axe-core's rule-set evolves.

**A11Y-02 targeted assertion (Toast aria-live):**
The Toast component is only visible WHEN visible. The spec triggers a toast by deep-linking to `/diary/day/1` with seeded state, opening the bedtime form, saving it, then asserts the rendered toast root has `role="status"` + `aria-live="polite"` + `aria-atomic="true"` attributes within a short timeout window. This is a separate test variant added per locale (likely just runs for EN + AR + ZH — 3 of the 6 locales — to keep run time reasonable; the underlying Toast component is locale-independent at the ARIA layer so this sampling is sufficient).

If the toast cannot be triggered programmatically (e.g., the form opening selectors differ from Phase 6's), the test should `test.skip()` and surface a note for the human-verify checkpoint — the audit explicitly notes that toast announcement is hard to fully verify without a screen reader.

**A11Y-03 targeted assertion (skip-link present + functional):**
For each locale × route combination, assert (in a single non-looped test):
1. `page.locator('a[href="#main-content"]').first()` exists in the DOM.
2. The link has the `sr-only` class (or is otherwise invisible per `getComputedStyle.position`).
3. After `page.keyboard.press('Tab')` from a fresh page-load focus, the focused element is the skip link.
4. After `page.keyboard.press('Enter')` on the focused skip link, `document.activeElement` becomes the `<main id="main-content">` element.
5. The `<main>` element has `id="main-content"` and `tabIndex="-1"`.

**A11Y-04 targeted assertion (ConfirmDialog Cancel autoFocus):**
The most realistic way to trigger ConfirmDialog in an automated test is to deep-link to `/diary/day/1` with seeded state, open a log form (FAB → void), make the form dirty (set a value), then dismiss (close button). The dirty-discard ConfirmDialog appears. Assert:
1. After dialog opens, `document.activeElement` is the Cancel button (the button whose text matches the locale's `common.keepEditing` translation).
2. The dialog's button DOM order has the Confirm button (text = `common.discard`) FIRST (DOM-order index 0 inside the `.flex.gap-3` parent) and Cancel SECOND (index 1).

If the seeded state + form opener selectors don't reliably trigger the ConfirmDialog (the Phase 6 spec has form-open patterns we can reuse — see `e2e/phase6-keyboard.spec.ts` if it exists), this test should `test.skip()` for that locale and surface a note for the human-verify checkpoint. The unit-level guard is in Plan 11-03 (the DOM-order grep + the ref-rename assertions); this e2e is an integration-level smoke test.
