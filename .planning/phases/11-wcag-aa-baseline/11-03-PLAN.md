<objective>
Rework `src/components/ui/ConfirmDialog.tsx` so the safe action (Cancel) is the default focus + the right-positioned primary button, and the destructive action (Confirm) sits in the left-positioned secondary slot. The audit-cited orphan `confirmBtnRef` (declared at line 27, never assigned to any element) is replaced with a `cancelBtnRef` correctly wired to the Cancel button + auto-focused on dialog open.

**Why this matters medically:** Two of the three ConfirmDialog callers in this codebase guard clinical record integrity:
- `DayPageClient.tsx:425-434` — dirty-form discard ("Your changes won't be saved"). A patient who taps Enter at dialog-open today loses 30 seconds of input (per UI-REVIEW §A3). After this plan, Enter at dialog-open keeps the form open.
- `TimelineView.tsx:845-854` — delete-entry confirmation. A patient who taps Enter at dialog-open today silently deletes the entry they were trying to inspect. After this plan, Enter keeps the entry.

Plus the third caller (TimelineView's reset-diary overlay, which is NOT this `ConfirmDialog` component — it's an inline pattern at line 856-870 of TimelineView.tsx). That inline overlay is out of scope for this plan because (a) it predates ConfirmDialog and uses a different UI pattern, (b) it has its own already-correct hardcoded button order with Cancel as the primary green button. The audit (UI-REVIEW.md A3) explicitly cites the `ConfirmDialog.tsx` file as the gap; the inline reset overlay in TimelineView is correctly defaulting to Cancel already (a separate Phase 8 finding noted its inline-overlay pattern is a refactor candidate but not a safety bug).

**Visual change is the explicit a11y win.** Swapping the visible position of Confirm (left, secondary-styled) and Cancel (right, primary-styled) is a deliberate, audit-prescribed visual change. The plan does not pretend this is invisible — it IS visible — but it is the canonical medical-grade pattern (safe action right/primary; destructive action left/secondary). Phase 11 budgets exactly this visual diff in A11Y-04's success criterion.

**Variant handling:** `ConfirmDialog` accepts `variant: 'danger' | 'default'`. Today:
- `variant === 'danger'` → Confirm button is red (`bg-danger`)
- `variant === 'default'` → Confirm button is amber (`bg-ipc-500`)

After this plan:
- The PRIMARY (right) button is always Cancel, styled with the SAFE solid look — the existing `bg-ipc-500 text-white` (the amber-warm "good action" pattern already used by the Hero CTA in LandingContent.tsx).
- The SECONDARY (left) button is always Confirm, with destructive variant rendered red (`bg-danger text-white` for danger) or amber (`bg-ipc-500 text-white` for default).
- Wait — for the NON-destructive ('default') variant, both buttons end up `bg-ipc-500` if we follow that mapping literally. That's wrong; the user needs to distinguish which button "does the thing" vs "cancels".
- Correct mapping: Cancel always gets the SAFE neutral style (`bg-ipc-50 text-ipc-800 border border-ipc-200` — the existing Cancel pattern from the current file, line 67-70). The destructive button is red for danger, amber for default. This way, Cancel reads as "neutral / safe", Confirm reads as "the action you came for". The CALLER never needs to know about the swap — the API is unchanged.

So the new visual mapping is:
- LEFT (secondary): the Confirm button.
  - `variant='danger'` → `bg-danger text-white` (unchanged from today's danger style)
  - `variant='default'` → `bg-ipc-500 text-white` (unchanged from today's default style)
- RIGHT (primary, autoFocused): the Cancel button.
  - Style: `bg-ipc-50 text-ipc-800 border border-ipc-200` (unchanged from today's Cancel style)

The styles are unchanged. ONLY the DOM order swaps and the autoFocus shifts from "nothing" to "Cancel".

This is the smallest possible change that delivers the medical-grade safe-default semantics. Visual contrast still cleanly distinguishes the two buttons (red/amber vs neutral cream), but the SAFE one is now where muscle memory + platform convention puts the primary action — on the right (or "end" in RTL).

**Caller compatibility:** No `ConfirmDialog` caller in the repo passes a custom button-order prop. Every caller in `grep -rn "ConfirmDialog"` results provides only `title / message / confirmLabel / cancelLabel / variant / onConfirm / onCancel` — the existing prop shape stays identical. The plan does NOT add a prop API for "which button is primary"; the safe-default is baked into the component. Callers do not change.

Output: A single file (~25 lines net change) in `src/components/ui/ConfirmDialog.tsx`. Two callers' user-visible behavior shifts (visible swap of button position; Cancel autoFocused; Enter at dialog-open activates Cancel) but their API surface to those callers is unchanged.

```tsx
useEffect(() => {
  if (!open) return;
  // Programmatic backup focus — autoFocus on the JSX element handles
  // the common case; this useEffect catches the edge case where React
  // skips autoFocus on portal-style mounts.

    cancelBtnRef.current?.focus();
  }, 0);
  return () => window.clearTimeout(t);
}, [open]);
```

(The `setTimeout(..., 0)` is the canonical pattern for "focus on next tick" — ensures the element is fully attached to the DOM before .focus() is called.)
