# Plan 05-05: Reposition QuickLogFAB to anchor against the centered content column at md+ (Pattern A — anchored float)

<objective>
At viewport ≥768px (`md`), the QuickLogFAB must stop floating at the viewport's bottom-right corner and instead anchor to the right edge of the centered content column. UI-SPEC §"5. `QuickLogFAB` (MODIFIED — repositions at md+)" locks **Pattern A — anchored floating to content column right edge** as the chosen approach (Pattern B, an inline anchored "Log event" button replacing the FAB, is deferred — it requires inserting controls into TimelineView which is Phase 6 work).

The implementation strategy (UI-SPEC lines 543-554):

1. **`right-5` → `end-5`** at the base (mobile) class. This is a CRITICAL latent-RTL fix: today's `right-5` keeps the FAB on the right side in RTL (Arabic), which is the inline-START side. Switching to `end-5` (logical) places the FAB correctly in BOTH directions: in LTR locales (`en`, `fr`, `es`, `pt`, `zh` per project memory `project_i18n_six_locales.md`) `end-5` resolves to `right-5` so the LTR mobile baseline is byte-equivalent (no visual diff in 5 of 6 locales). In Arabic the FAB now correctly sits on the LEFT (the inline-end side in RTL) — UI-SPEC §"Mobile Invariants" point 2 explicitly notes this is a CORRECTNESS FIX, not a regression, and is the documented exception to the "no mobile diff" rule.

2. **`md:end-[max(1.25rem,calc((100vw-768px)/2+1.25rem))]`** — at viewport ≥768px, the FAB's inline-end position is computed so it sits 20px (`1.25rem`) from the right edge of a 768px-wide content column centered in the viewport. The `max()` floor of `1.25rem` ensures it never overlaps the viewport edge if the viewport is narrower than 768px (an unreachable case past the `md:` breakpoint, but defensive).

3. **No `lg:` formula needed.** The `md:end-[max(1.25rem,calc((100vw-768px)/2+1.25rem))]` rule above handles ALL desktop widths (md, lg, xl, 2xl) because the diary content column stays at `max-w-3xl` (768px) across every desktop breakpoint per the locked diary contract. (UI-SPEC §"Note on the assumption that DayPageClient uses `Container variant=\"default\"`" + the corrected position table at lines 568-577 of UI-SPEC.) An earlier draft of this plan had a `lg:end-[max(1.25rem,calc((100vw-1024px)/2+1.25rem))]` rule that anchored to a 1024px column edge — that was wrong because the diary column does NOT widen to 1024px at lg+; it stays at 768px. The wrong formula would have placed the FAB ~108px past the actual content edge at 1280px+ viewports, defeating the entire phase goal. The single `md:` rule above is correct for all widths.

4. **`md:bottom-8`** — at md+, the FAB sits 32px from the bottom of the viewport (no BottomNav to clear; the `pb-24` content spacer collapses via 05-03). Below md, the existing scroll-aware `bottom-6` / `bottom-24` toggle is preserved.

5. **`focus-visible:` rings on the toggle button AND the three speed-dial buttons** per UI-SPEC §"Accessibility" for this component: `focus-visible:ring-2 focus-visible:ring-ipc-500 focus-visible:ring-offset-2 focus-visible:ring-offset-surface`. These are NEW; the existing FAB has `active:scale-[0.95]` only and no explicit focus-visible style. The drink/leak/void speed-dial buttons keep their existing `focus:ring-drink/40` etc. (NO — re-check: those are on the existing `<Button>` primitive variants, not on these `<button>` elements; the speed-dial buttons in QuickLogFAB.tsx are raw `<button>` tags with no focus styling. Add focus-visible rings to all three plus the toggle.)

Mobile invariants (UI-SPEC §"Mobile Invariants" point 2):
- At <768px in LTR locales: `end-5` resolves to `right-5` so visually identical to today.
- At <768px in RTL (Arabic): FAB moves from right-corner to left-corner — DOCUMENTED EXCEPTION (correctness fix).
- The existing scroll-aware `navHidden` logic + `bottom-24`/`bottom-6` toggle is preserved at <768px.
- Existing speed-dial buttons (drink/leak/void) keep their `ps-4 pe-3` logical padding (already correct per UI-SPEC §"Pre-existing physical-CSS leak in expanded action buttons" — these are NOT touched).
- Existing `data-testid` attributes (`fab-toggle`, `fab-action-drink`, `fab-action-leak`, `fab-action-void`) MUST be preserved unchanged — E2E tests depend on them per UI-SPEC §"Accessibility".

RTL invariants (UI-SPEC §"Phase 5 RTL audit" row 4):
- `right-5` → `end-5` is the central RTL fix.
- `md:end-[...]` uses the logical `inset-inline-end` arbitrary value (Tailwind compiles `end-[X]` to `inset-inline-end:X`). No `lg:end-[...]` rule is needed because the single `md:` formula handles all desktop widths (the diary column stays at 768px at every breakpoint).
- The transform `translateX` is NOT used here — the existing `rotate-45` on the toggle when expanded is direction-neutral.

Open questions addressed:
- Q5 (Button.tsx focus-visible migration) — not in this plan. UI-SPEC defers to Phase 6.

This plan has no `depends_on`. It can run in parallel with 05-03 and 05-04 in Wave 2 (no file overlap with either). It does NOT depend on Plan 05-01 (Container is not used by QuickLogFAB).
</objective>
