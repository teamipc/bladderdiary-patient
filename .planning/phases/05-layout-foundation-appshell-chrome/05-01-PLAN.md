# Plan 05-01: Create the `<Container>` layout primitive (server-component-safe)

<objective>
Land the shared `<Container>` React component that consolidates the duplicated `max-w-* mx-auto w-full px-*` pattern currently scattered across `LandingContent.tsx`, `OnboardingFlow.tsx`, `diary/layout.tsx`, and `summary/page.tsx`. This is the foundation Wave-1 deliverable: every other plan in Phase 5 (and the chrome-adoption work in Phase 6/7) depends on this primitive existing and being importable.

The Container is a server-component-safe React component (no `'use client'`, no hooks). It exposes a 4-variant `variant` prop (`narrow | default | wide | full`), an `as` prop for semantic-HTML choice (`div | section | main | article | header | footer | nav`), a `noPadding` prop (boolean — when true, the variant's `px-*` classes are STRIPPED from the emitted class string entirely so callers can supply their own padding without fighting Tailwind's internal canonical class ordering), and a `className` prop that is appended after the variant class string so callers can layer per-page vertical padding without restating the geometry contract.

Component lives at `src/components/layout/Container.tsx` (NOT `src/components/ui/`) — it is a structural primitive, not an interactive UI element. UI-SPEC §"Container component API" locks this rationale.

This plan also adds a minimal vitest unit test (`src/__tests__/container.test.tsx`) verifying each variant emits the locked class string from UI-SPEC §"Class string per variant", that `as` controls the rendered tag, that `className` is appended (not replaced), and that `noPadding` strips the variant's `px-*` classes.

UI-SPEC §"1. `Container` (NEW)" is the source of truth for visual contract; UI-SPEC §"Container component API (the chosen shape)" locks the API; UI-SPEC §"Class strings per variant" locks every class string this plan must emit.

Open questions addressed:
- Q1 (mobile padding regression on landing) and Q2 (mobile padding regression on onboarding) — design the `noPadding` prop so callers in 05-06 can do `<Container variant="narrow" noPadding className="px-6 sm:px-6">` to fully replace padding. Tailwind 4 utility ordering is internal-canonical (NOT class-string order), so the previous "append `px-6 sm:px-6` after `px-4 sm:px-6`" override pattern is fragile. `noPadding` removes the variant padding entirely, making the override deterministic. Document this in a JSDoc on the component.
- Q8 (no `useBreakpoint` hook) — locked: Container has no hooks at all.
</objective>

    Create a new file at `src/components/layout/Container.tsx`. It must follow these specifics, every value taken from UI-SPEC §"Container component API" and §"Class strings per variant":

    1. NO `'use client'` directive at the top — this component is server-component-safe and must work inside both client trees (diary day pages) and server trees (potentially `/learn` later). UI-SPEC §"Why `layout/` not `ui/`" locks this rationale; do not add `'use client'`.

    2. Imports: only `import type { ElementType, ReactNode } from 'react';` — no React runtime import needed for a server-component-safe component that just renders the tag returned by `as`.

    3. Define and EXPORT a `ContainerVariant` type union: `'narrow' | 'default' | 'wide' | 'full'`. Define and EXPORT a `ContainerAs` type union: `'div' | 'section' | 'main' | 'article' | 'header' | 'footer' | 'nav'`. Downstream plans (05-04, 05-06) import `ContainerVariant` to keep call sites typed.

    4. Define a non-exported `interface ContainerProps` with: `variant?: ContainerVariant` (default `'default'`), `as?: ContainerAs` (default `'div'`), `noPadding?: boolean` (default `false`), `className?: string` (default `''`), `children: ReactNode`.

    5. Define a non-exported `const VARIANT_CLASSES: Record<ContainerVariant, string>` with EXACTLY these strings (these are the locked tokens from UI-SPEC §"Class strings per variant" — do not invent your own):
       - `narrow`: `'mx-auto w-full max-w-2xl px-4 sm:px-6'`
       - `default`: `'mx-auto w-full max-w-3xl px-4 sm:px-6'`
       - `wide`: `'mx-auto w-full max-w-5xl px-4 sm:px-6 lg:px-8'`
       - `full`: `'w-full px-4 sm:px-6 lg:px-8 xl:px-10'`

    6. Define a non-exported `const VARIANT_CLASSES_NO_PADDING: Record<ContainerVariant, string>` containing each variant string with ALL `px-*` and `sm:px-*` and `lg:px-*` and `xl:px-*` tokens removed (geometry-only — `mx-auto`, `w-full`, `max-w-*`):
       - `narrow`: `'mx-auto w-full max-w-2xl'`
       - `default`: `'mx-auto w-full max-w-3xl'`
       - `wide`: `'mx-auto w-full max-w-5xl'`
       - `full`: `'w-full'`

    7. Implement `export default function Container({ variant = 'default', as = 'div', noPadding = false, className = '', children }: ContainerProps)`. Inside, build:
       ```ts
       const variantClasses = noPadding ? VARIANT_CLASSES_NO_PADDING[variant] : VARIANT_CLASSES[variant];
       const classes = `${variantClasses} ${className}`.trim();
       const Tag = as as ElementType;
       return <Tag className={classes}>{children}</Tag>;
       ```
       Use `as ElementType` to satisfy strict TS — this is type-narrowing, not type-laundering, so it is acceptable per CLAUDE.md (no `as any`, no `@ts-ignore`).

    8. Add a JSDoc block ABOVE the function with these key facts an executor adopting Container needs to know:
       - Server-component-safe: no hooks, no `'use client'`
       - Geometry only: max-width, horizontal padding, horizontal centering — NOT vertical rhythm
       - `className` is APPENDED to the variant class string, so callers can layer per-page vertical padding (`pt-12 pb-8`)
       - `noPadding` STRIPS the variant's `px-*` classes entirely (returns `mx-auto w-full max-w-*` only). Use this when you need to fully replace horizontal padding without relying on Tailwind class-order tie-breaking — Tailwind 4 utility ordering is internal-canonical, so appending `px-6 sm:px-6` after a variant's `px-4 sm:px-6` does NOT reliably win. Pattern: `<Container variant="narrow" noPadding className="px-6 sm:px-6">` for the documented landing/onboarding mobile-padding edge case (UI-SPEC Open Questions Q1/Q2).
       - Direction-neutral: all properties used (`mx-auto`, `max-w-*`, `px-*`) work in both LTR and RTL; no logical-property substitution needed.
       - Default variant is `'default'` (= `max-w-3xl`); pick `narrow` for single-column form/onboarding/footer content, `wide` for summary metric grids and Header chrome, `full` for chrome bands that span viewport width.

    9. NO additional Tailwind variants beyond those four. NO `tailwind-merge`/`clsx` import (per UI-SPEC §"Tech baseline" — `clsx`/`cva` is NOT in deps and NOT introduced).

    10. NO emoji. NO em-dashes. Comments use `//` or `/* */` only.

    Per CONTEXT.md §"Internationalization invariants": every class string used here is direction-neutral (`mx-auto`, `w-full`, `max-w-*`, `px-*`). Do not introduce `ml-*`/`mr-*`/`pl-*`/`pr-*`/`left-*`/`right-*`. The audit table in UI-SPEC §"Phase 5 RTL audit" row 1 already certifies Container PASS — your job is to keep it PASS.
