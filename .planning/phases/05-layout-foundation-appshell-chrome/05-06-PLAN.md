# Plan 05-06: Adopt `<Container>` in 4 page components (LandingContent, OnboardingFlow, diary/layout, summary/page)

<objective>
Replace the duplicated `max-w-* mx-auto w-full px-*` wrapper pattern in four page components with the new `<Container>` primitive (Plan 05-01). This consolidates ROADMAP Phase 5 success criterion #3 — "A shared container/layout primitive … is in use across `LandingContent.tsx`, `OnboardingFlow.tsx`, `DayPageClient.tsx`, and `summary/page.tsx`" — and lets Phases 6/7 redesign per-page composition without re-deriving max-widths.

CRITICAL CONSTRAINT — **PRESERVE per-page composition exactly**. This plan ONLY swaps the outermost wrapper `<div>` for `<Container variant="...">` plus moves the per-page vertical padding into `className`. It does NOT redesign onboarding step composition, summary metric grids, hero hierarchy, button placements, or any other internal markup. UI-SPEC §"8. `OnboardingFlow` and `summary/page.tsx` (Container adoption — Phase 5 minimum, Phase 7 full redesign)" locks this scope.

**The four files and their locked transformations** (all values from UI-SPEC §"Adoption checklist" and §"7. `LandingContent`", §"8. `OnboardingFlow`", §"9. `diary/layout.tsx`"):

1. **`src/app/[locale]/diary/layout.tsx`** — Replace `<div className="max-w-xl mx-auto w-full px-4 pt-4">` with `<Container variant="default" as="div" className="pt-4">`. Visual delta: content widens from `max-w-xl` (576px) to `max-w-3xl` (768px) at desktop. At mobile (<640px) padding stays `px-4` exactly. (UI-SPEC §"9. `diary/layout.tsx`".)

2. **`src/app/[locale]/LandingContent.tsx`** — Three wrapper divs to swap:
   - Line 87 (hydration-loading state): `<div className="flex items-center justify-center py-24 bg-surface">` → `<Container variant="narrow" as="div" className="flex items-center justify-center py-24 bg-surface">`
   - Line 95 (`diaryStarted` "Welcome back" view): `<div className="flex flex-col items-center px-6 pt-12 md:pt-20 pb-12 max-w-lg md:max-w-xl mx-auto w-full">` → `<Container variant="narrow" as="div" noPadding className="px-6 sm:px-6 pt-12 md:pt-20 pb-12 flex flex-col items-center">`
   - Line 161 (pre-onboarding hero view): `<div className="flex flex-col items-center justify-center px-6 pt-8 md:pt-16 pb-12 max-w-lg md:max-w-2xl mx-auto w-full">` → `<Container variant="narrow" as="div" noPadding className="px-6 sm:px-6 pt-8 md:pt-16 pb-12 flex flex-col items-center justify-center">`

   **Mobile-padding override (CORRECTED — uses `noPadding` prop):** `<Container variant="narrow">` defaults to `px-4 sm:px-6`. The two non-loading wrappers in landing currently use `px-6` at mobile. Per UI-SPEC §"Mobile invariant" for §"7. `LandingContent`" and Open Question Q1, the locked approach is to PRESERVE `px-6` at mobile by passing `noPadding` (Plan 05-01 added this prop) which STRIPS the variant's `px-*` classes entirely, then supplying `px-6 sm:px-6` in `className`. This is DETERMINISTIC. (An earlier draft of this plan tried "append `px-6 sm:px-6` after `px-4 sm:px-6` and rely on Tailwind class-string-order tie-breaking" — that approach was FRAGILE because Tailwind 4 sorts utilities by internal canonical order in the generated CSS, NOT by HTML class-string order, so the override could silently fail. The `noPadding` prop removes the variant padding entirely, making the override deterministic.)

   The hydration-loading state (line 87) does NOT need `noPadding` because it has no explicit `px-*` of its own — the default Container padding (`px-4 sm:px-6`) is fine for a centered spinner.

   The pre-onboarding hero view has `max-w-lg md:max-w-2xl` (mobile 512px / desktop 672px). After Container the effective width becomes `max-w-2xl` (672px) at all widths, since `Container variant="narrow"` is `max-w-2xl`. UI-SPEC §"7. `LandingContent`" §"Visual delta" notes this: at desktop the column widens slightly (the `md:max-w-2xl` was already 672px so no change at md+); at mobile the column widens from 512px to 672px which inside a 375px or 414px iPhone viewport is bounded by the viewport itself anyway (so no visible change at mobile). The inner `<div className="w-full md:max-w-md ...">` at line 174 (CTA wrapper) is intentionally narrower than `narrow`; it stays as-is.

3. **`src/components/onboarding/OnboardingFlow.tsx`** — Line 89 wrapper currently is `<div className="flex flex-col items-center px-6 pt-6 md:pt-12 pb-10 max-w-lg md:max-w-xl mx-auto w-full">`. Replace with `<Container variant="narrow" as="div" noPadding className="px-6 sm:px-6 pt-6 md:pt-12 pb-10 flex flex-col items-center">`. Same deterministic `noPadding + className="px-6 sm:px-6"` override pattern as landing (Q2 — onboarding mobile-padding edge case). Width widens slightly at desktop (`max-w-xl` 576px → `max-w-2xl` 672px) per UI-SPEC §"8. `OnboardingFlow.tsx` line 89".

4. **`src/app/[locale]/summary/page.tsx`** — Two wrappers to swap (UI-SPEC §"8. `summary/page.tsx` lines 78, 89"):
   - Line 78 (locked / not-yet-complete view): `<div className="max-w-2xl mx-auto px-4 pt-12 text-center">` → `<Container variant="default" as="div" className="pt-12 text-center">`
   - Line 89 (full summary view): `<div className="max-w-2xl mx-auto px-4 pt-4 pb-12 space-y-6">` → `<Container variant="default" as="div" className="pt-4 pb-12 space-y-6">`

   Width widens from `max-w-2xl` (672px) to `max-w-3xl` (768px) per UI-SPEC `default` variant. UI-SPEC §"8. `summary/page.tsx` lines 78, 89" notes Phase 7 will widen further to `wide` for the metric grid; Phase 5 adopts `default`.

   Note on `LandingContent.tsx` line 87 (the loading state) and `summary/page.tsx` lines 67-72 (the loading state): both have a `<div className="flex items-center justify-center py-24 bg-surface">` loading spinner. Only the LandingContent one is in scope for this plan (it is the OUTER wrapper of LandingContent's render flow). Summary's lines 67-72 spinner is INSIDE the page render and is fine as a plain `<div>` — UI-SPEC adoption checklist for summary lists only lines 78 and 89, not the loading state. Leave summary's lines 67-72 spinner alone.

Mobile invariants (UI-SPEC §"Mobile Invariants" points 6, 7, 8):
- Diary day page content widens from `max-w-xl` (576px) to `max-w-3xl` (768px) ON DESKTOP; at mobile (<640px) the effective width is the viewport minus `px-4` padding — same as today.
- Landing page mobile padding stays `px-6` because we pass `noPadding` (strips variant's `px-*`) AND `className="px-6 sm:px-6"` (supplies the desired padding). Deterministic — no dependency on Tailwind class-order tie-breaking. Defends invariant Q1.
- Onboarding flow mobile padding stays `px-6` for the same reason — `noPadding + className="px-6 sm:px-6"`. Defends Q2.
- Summary page mobile padding shifts from `px-4` to `px-4 sm:px-6` — at <640px the effective padding is `px-4` (16px) which matches today's `px-4`, so NO mobile diff. At 640px+ on a tablet portrait, padding becomes `px-6` (24px) — slightly more breathing room, which is expected and desirable.

RTL invariants (UI-SPEC §"Phase 5 RTL audit" — Container is direction-neutral):
- All Container variant classes are direction-neutral (`mx-auto`, `max-w-*`, `px-*`).
- `flex flex-col items-center`, `text-center`, `space-y-6`, `bg-surface`, `pt-*`/`pb-*`/`py-*` are all direction-neutral.
- DO NOT introduce any `ml-*`/`mr-*`/`pl-*`/`pr-*`/`left-*`/`right-*` classes in the appended `className` strings.
- Existing inner direction-aware code (e.g., `summary/page.tsx` line 99 `ChevronLeft size={20} className="rtl:scale-x-[-1]"`, `summary/page.tsx` line 151 `ms-1`, `OnboardingFlow.tsx` ChevronRight RTL flips) is INSIDE the Container — those lines are not touched.

Open questions addressed:
- Q1 (landing mobile padding regression) — addressed by `noPadding + className="px-6 sm:px-6"` deterministic override on lines 95 and 161 of LandingContent.
- Q2 (onboarding mobile padding regression) — addressed by `noPadding + className="px-6 sm:px-6"` deterministic override on line 89 of OnboardingFlow.
- Q3 (Header max-w) — N/A this plan, addressed in 05-04.
- Q5 (Button.tsx focus-visible) — N/A, deferred to Phase 6.

This plan depends on Plan 05-01 only. It can run in parallel with Wave 2 (05-03, 05-04, 05-05) because none of those plans modify any of the four page-component files this plan touches. Per the dependency-graph rules, Wave 3 is assigned because file-overlap analysis shows zero conflicts with Wave 2 plans.
