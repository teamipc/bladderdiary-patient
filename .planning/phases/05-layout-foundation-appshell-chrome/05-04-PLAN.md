# Plan 05-04: Add desktop top-bar nav inside Header.tsx (Home / Track / Diary), wrap Header inner with `<Container variant="wide">`

<objective>
Expand `src/components/layout/Header.tsx` so that at viewport ≥768px (`md`) it shows a primary top-bar `<nav>` with three links (`Home`, `Track`, `Diary`) inline between the existing logo (left) and the existing Learn link + locale switcher (right). Below 768px, the Header renders BYTE-FOR-BYTE identically to today (mobile invariant).

Per UI-SPEC §"3. `Header` (MODIFIED — adds top-bar nav)", the locked decisions for this plan are:

1. **Top-bar nav lives INSIDE `Header.tsx` — NOT a new `<TopNav>` component.** Rationale: the Header already hosts primary nav (the existing Learn link is inline there); a separate band would add a second sticky row/border and break the editorial single-band aesthetic. Reduces coupling: 1 file edit instead of 2 + AppShell mount.

2. **`NavLink` is an INLINE helper component defined inside `Header.tsx`** (NOT a separate `src/components/layout/NavLink.tsx` file). It is a one-use component; pulling it into its own file would inflate the layout/ directory without payoff. Defined as a non-exported `function NavLink({...})` below the default `Header` function in the same file.

3. **Header inner uses `<Container variant="wide">`** (= `mx-auto w-full max-w-5xl px-4 sm:px-6 lg:px-8`) instead of the existing `max-w-6xl mx-auto px-4 sm:px-6 lg:px-8`. UI-SPEC §"Header internal max-width change" locks this — `max-w-5xl` (1024px) aligns the Header's content gridline with summary + learn pages.

4. **Three-region flex layout**: left = logo (existing button), center = primary nav (NEW), right = Learn link + locale switcher (existing utility cluster).

5. **Conditional nav-item logic** (UI-SPEC §"Conditional nav-item logic (locked)"):
   - `Home` always shown
   - `Track` shown only when `diaryStarted` (matches existing BottomNav Track-disabled behavior — at desktop we hide instead of locking)
   - `Diary` shown only when `isTrackingComplete` (Day 3 bedtime recorded; matches existing BottomNav locked-Diary behavior — at desktop we hide instead of showing a padlock)
   - `Learn` is the existing right-cluster Link, NOT moved into the new primary nav block. Existing visual treatment preserved.

6. **NavLink visual + interaction (UI-SPEC §"Hover-affordance spec" + §"NavLink helper component")**:
   - Resting: `text-ipc-800` text, transparent bg, NO underline
   - Hover: `text-ipc-950 bg-ipc-50` (two-channel feedback)
   - Active (`aria-current="page"`): `bg-ipc-100 text-ipc-900 underline decoration-ipc-400 underline-offset-4 decoration-2`
   - Focus-visible ring (UI-SPEC §"Focus-visible ring spec"): `focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-ipc-500 focus-visible:ring-offset-2 focus-visible:ring-offset-surface`
   - Padding: `px-3 py-1.5 rounded-lg`, typography: `text-sm font-semibold`, transition: `transition-colors duration-150`

7. **Primary nav is wrapped in `<nav aria-label={tNav('primaryNavAriaLabel')}>`** (the i18n key delivered by Plan 05-02), with classes `hidden md:flex items-center gap-1`. At <768px the entire `<nav>` is `display:none` — mobile DOM unchanged in behavior because nothing reachable changes; the new chrome is gated behind `md:flex`.

This plan depends on Plan 05-01 (Container) AND Plan 05-02 (`nav.primaryNavAriaLabel` i18n key). Both must complete in Wave 1 before this plan starts.

Mobile invariants (UI-SPEC §"Mobile Invariants" point 3):
- The new `<nav className="hidden md:flex">` is `display:none` at <768px → no visual or layout effect.
- The Header's outer `<header>` element keeps its existing `sticky top-0 z-30 bg-surface/80 backdrop-blur-md border-b border-ipc-100` classes.
- Existing logo button (lines 52-67), existing Learn link (lines 70-79), and existing locale-switcher button + dropdown (lines 82-113) are preserved in DOM order and class strings.
- Container wrapping replaces `max-w-6xl mx-auto px-4 sm:px-6 lg:px-8 h-14` with `<Container variant="wide" as="div" className="flex items-center justify-between h-14">`. At <640px (mobile), `Container variant="wide"` resolves to `mx-auto w-full max-w-5xl px-4 sm:px-6 lg:px-8` — only the `max-w-5xl` differs from `max-w-6xl` BUT at mobile widths neither caps anything (viewport <1024px never reaches the cap), so mobile is byte-equivalent.

RTL invariants (UI-SPEC §"Phase 5 RTL audit" row 2):
- All new classes used here are direction-neutral: `gap-1`, `px-3`, `py-1.5`, `rounded-lg`, `bg-*`, `text-*`, `underline`, `decoration-*`, `underline-offset-*`, `transition-colors`, `duration-150`, `focus-visible:*`, `hidden md:flex`, `flex items-center justify-between h-14`, all the `mx-auto`/`max-w-*`/`px-*` used by Container.
- DOM order Home → Track → Diary inside the `<nav>` block; in RTL the parent flex inherits `dir` from `<html>` and naturally flows right-to-left.
- Existing logical `end-0` on the locale dropdown (line 96) preserved.
- NO `ml-*`/`mr-*`/`pl-*`/`pr-*`/`left-*`/`right-*` introduced in any new line.

Open questions addressed (UI-SPEC §"Open Questions / Claude's Discretion"):
- Q3 (Header `max-w-6xl` vs. `max-w-5xl`) — locked default `max-w-5xl` via `Container variant="wide"`. If a follow-up screenshot at 1280px+ shows nav cramping, the planner of a future revision can override with `<Container variant="full" className="max-w-6xl mx-auto">`. Not exercised in this plan.
- Q4 (PT/FR nav-text overflow at 768px) — UI-SPEC §"Long-translation handling" verifies all 6 locales fit; if executor finds overflow during 05-07 visual verification, the documented fallback is `md:px-2 lg:px-3` on NavLink padding. Default `px-3 py-1.5` ships in this plan.
</objective>

    ```tsx
    function NavLink({
      href,
      active,
      children,
    }: {
      href: string;
      active: boolean;
      children: React.ReactNode;
    }) {
      const baseClasses =
        'px-3 py-1.5 rounded-lg text-sm font-semibold transition-colors duration-150 ' +
        'focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-ipc-500 focus-visible:ring-offset-2 focus-visible:ring-offset-surface';
      const stateClasses = active
        ? 'bg-ipc-100 text-ipc-900 underline decoration-ipc-400 underline-offset-4 decoration-2'
        : 'text-ipc-800 hover:text-ipc-950 hover:bg-ipc-50';
      return (
        <Link
          href={href}
          aria-current={active ? 'page' : undefined}
          className={`${baseClasses} ${stateClasses}`}
        >
          {children}
        </Link>
      );
    }
    ```
