**Key design decisions resolved here (locked, NOT for executor to revisit):**

- **D-01 — Night-mode fade is implemented via a CSS `transition` on the body element, NOT a JS-driven class animation.** Per CONTEXT constraint "Tailwind keyframes + CSS variables only" + "No new heavy dependencies". The existing toggle of the `.nighttime-bg` class on the body (in `DayPageClient.tsx:176-184`) is the trigger; adding a CSS transition on the body's color properties means the toggle naturally animates without ANY JS change. This is the simplest possible implementation — zero new code paths, zero new event handlers, zero new state.

- **D-02 — Transition targets ONLY color-related properties, NOT `all`.** Per CONTEXT constraint "Performance: only `transform` + `opacity` in animation properties; avoid layout-triggering properties". `background-color`, `color`, `border-color` are repaint-only (no layout reflow). `transition: all` would also animate margin, padding, font-size, etc. — properties that the cascade may change incidentally and produce jank. The explicit property list scopes the transition to the visual intent.

- **D-03 — 2-second duration is INTENTIONALLY longer than the Boomer-safe 200ms cap.** Per CONTEXT MI-05: "a subtle ~2s fade-in transition into the existing night-mode color palette". The 200ms cap applies to INTERACTION transitions (button presses, step advances, chip selections) where rapid feedback matters. The night-mode shift is an AMBIENT environment change (like the lights dimming in a room) — slow IS the point. 2s gives the patient a moment to register "the app is shifting to night mode" without being abrupt.

- **D-04 — `cubic-bezier(0.4, 0, 0.2, 1)` matches the curve used by Phase 14 + 15-01 + 15-02.** Visual coherence across all motion in Phase 14 and 15. The "emphasized" Material curve eases out gently — appropriate for an ambient transition.

- **D-05 — `prefers-reduced-motion: reduce` handled by the EXISTING global rule at globals.css:94-103.** That rule sets `transition-duration: 0.01ms !important` for all elements. The new body transition collapses to instant for reduced-motion users — they see the night-mode color shift instantly when the class is added. No information lost; only the 2s ease is removed.

- **D-06 — The transition is on the BODY element, NOT on each `.nighttime-bg .child` override individually.** The CSS cascade handles propagation: when the body's background-color changes (because `.nighttime-bg` adds a new `background-color` value), the transition on the body animates that change. The child rules (`.nighttime-bg header`, `.nighttime-bg .bg-surface`, etc.) ALSO need transitions on their respective elements because the cascade applies the body's transition only to the body's own color properties — child elements need their own transitions to animate. To keep the edit minimal, the executor adds transitions to: (a) `body`, (b) `header`, (c) `nav`, (d) `.bg-surface` (or its applied elements), so the full nighttime palette animates coherently. The detailed list is locked in the Action section.

- **D-07 — Verification spec is INERT, NOT executed in this plan.** Per Phase 13/14 precedent: the spec is AUTHORED for future verification but is NOT part of the daily walkthrough and is NOT triggered as a CI gate in this plan. The spec runs ONLY when a developer explicitly invokes it via the `PW_TEST_MATCH` env var. Each MI requirement gets multiple assertions because the surface has multiple observable behaviors (e.g., MI-04 has tooltip-appears + dismissal-persists + Day-1-suppression — 3 distinct assertions for one requirement).

         await page.addInitScript(() => {
           // Replace navigator.vibrate with a spy that counts calls. The
           // production fireSaveHaptic() still calls navigator.vibrate(15),
           // which now hits this spy; the spy's return value (true) keeps
           // the production code path happy.
           const counter = { count: 0, lastArg: null as number | number[] | null };
           (window as unknown as Record<string, unknown>).__hapticSpy = counter;
           const original = navigator.vibrate?.bind(navigator);
           Object.defineProperty(navigator, 'vibrate', {
             value: (arg: number | number[]) => {
               counter.count += 1;
               counter.lastArg = arg;
               return original ? original(arg) : true;
             },
             configurable: true,
           });
         });
       }

         });
       }
