**Key design decisions resolved here (locked, NOT for executor to revisit):**

- **D-01 — Liquid-fill is `transform: scaleY(0) -> scaleY(1)` from bottom, NOT a `height` animation.** Per CONTEXT constraint "Performance: only transform + opacity in animation properties". `transform: scaleY` is compositor-only (GPU-friendly) and avoids the layout/paint cost of `height` animations. `transform-origin: bottom` anchors the scale to the bottom edge, producing the visual of liquid rising from the chip's bottom.

- **D-02 — The fill span sits BEHIND the chip text via z-index ordering (`absolute inset-0 -z-0` on the fill, `relative z-10` on the text wrapper).** This way the chip's text stays continuously visible during the fill — the fill paints UNDER the text. Alternative (text fading in during the fill) was rejected because the text is information-bearing ("250 mL"), not decoration, and should be readable throughout the gesture.

- **D-03 — Animation re-fires on every tap via React `key` prop refresh on the fill `<span>`, NOT useEffect + setTimeout class swap.** The fill span's `key` is composed from `${p.id}-${nonce}` where `nonce` is a monotonic counter that increments on every chip selection. React unmounts the previous fill span and mounts a fresh one — the `animate-liquid-fill` keyframe naturally re-runs on every mount. Preferred over class manipulation (no JS state to manage, no React re-render churn).

- **D-04 — LeakForm has NO volume preset chips and is OUT OF SCOPE for MI-01.** Per CONTEXT "Files: LogVoidForm.tsx + LogDrinkForm.tsx (+ optionally LogLeakForm.tsx — leaks don't have volume chips so MAYBE skipped depending on form structure)". Confirmed via grep: `LogLeakForm.tsx` does not import `VOLUME_PRESETS` or use the chipPop pattern. The leak form's `LeakAmount` picker uses a different visual vocabulary (icon + label tiles) which is OUT OF SCOPE for MI-01. The chip-fill animation is applied to LogVoidForm + LogDrinkForm only.

- **D-05 — Haptic fires from the FORM save handlers, NOT from QuickLogFAB.** Per CONTEXT "src/components/diary/QuickLogFAB.tsx — `navigator.vibrate(15)` on successful save" — but the FAB itself does NOT save; it only opens the relevant form. The actual save happens inside `LogVoidForm`/`LogDrinkForm`/`LogLeakForm` when the form's submit handler calls `addVoid`/`addDrink`/`addLeak`. The haptic must fire AFTER the save succeeds, which means the call lives in the forms. The QuickLogFAB edit is a comment-only annotation explaining this rerouting to future developers.

- **D-06 — `navigator.vibrate(15)` — single 15ms pulse, NOT a pattern.** Per CONTEXT "Single short pulse — not a buzz". The argument is a single integer (milliseconds) rather than an array (which would denote a pattern). 15ms is the lower bound for perceptible vibration; CONTEXT specifies it directly. A 50ms or 100ms pulse would feel jarring; 15ms feels like a soft confirmation tap.

- **D-07 — SSR + capability safety in `haptic.ts`.** The utility is called from client components, but the static-export build still SSR-renders the components into HTML at build time. During SSR, `navigator` is undefined. The guard `typeof navigator !== 'undefined' && typeof navigator.vibrate === 'function'` handles both: SSR safety (no crash during build) AND runtime capability detection (no-ops on iOS Safari + desktop Chrome which don't expose `navigator.vibrate`).

- **D-08 — Default `hapticEnabled = true` for ALL devices, NOT capability-gated.** Per CONTEXT "Haptic default ON for mobile (detect via `navigator.vibrate` presence)". The implementation defaults to `true` for everyone; on devices without the API, `fireSaveHaptic` no-ops (the capability check is inside the utility). The setting persists across sessions; the toggle is hidden on devices without the API (so iOS Safari users don't see a setting that does nothing).

- **D-09 — HapticSettings island is hidden via `null` return when `navigator.vibrate` is absent.** The island is a client component, so it can use `useEffect` to probe `typeof navigator !== 'undefined' && 'vibrate' in navigator` post-mount, then conditionally render or return null. Alternative (rendering a disabled toggle with a "not supported on this device" caption) was rejected — clutters the help page for users who can't use the feature anyway.

- **D-10 — Settings live in `/help` page, NOT a new `/settings` route.** Per CONTEXT key planning question #1 + recommendation: "piggy-back on existing notifications-permission UX. Recommend the latter." There is no existing notifications-permission UX (verified via grep — no `notificationEnabled` or similar in src/), so the next-best surface is the existing `/help` page which is the de-facto settings + info home. Creating a new `/settings` route would require: a new page file, a new layout, a new bottom-nav entry, 6 new locale strings for the route label — over-engineering for a single toggle. Help page is sufficient.

- **D-11 — Persist version bump from `3` to `4`, with `migrateBladderDiaryState` adding the default.** The store's persist middleware handles versioned migrations. The current version is `3` (per `src/lib/store.ts:380`). Bumping to `4` and updating `migrateBladderDiaryState` (already updated for v2 -> v3 transitions at lines 138-166) to add `hapticEnabled: true` for any persisted state lacking it preserves all existing user data without crashing on the new field. Existing users wake up with haptic ON (matching the new-user default) and can toggle it off in /help if they prefer.

- **D-12 — `hapticToggle` is a NEW top-level i18n namespace, NOT nested under `help.*`.** Rationale: the toggle's strings are reusable if we add the same setting to other surfaces (e.g., a future quick-settings sheet from the bottom-nav). Top-level keeps the strings discoverable; `help.*` would dilute the help namespace with settings-flavoured strings. The `i18n-sync` hook handles the auto-mirror to fr/es/pt/zh/ar.

- **D-13 — Liquid-fill timing matches Phase 14 EM-05's `stepPulse` (180ms cubic-bezier(0.4, 0, 0.2, 1)).** Visual coherence: when a patient progresses through onboarding (Phase 14 step-pulse beats) and then logs a void (Phase 15 chip-fill beat), the motion vocabulary is the same. Boomer-safe per CLAUDE.md memory `ux_philosophy` 200ms cap.
