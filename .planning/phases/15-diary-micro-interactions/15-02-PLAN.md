**Key design decisions resolved here (locked, NOT for executor to revisit):**

- **D-01 — Overlay placement is ROUTE-LEVEL inside DayPageClient, NOT between routes.** Per CONTEXT key planning question #2 + recommendation: "Recommend route-level." The overlay mounts inside `DayPageClient.tsx`'s return JSX and is triggered when the URL's `dayNumber` advances (Next.js App Router's `useParams` exposes the new dayNumber on every navigation). This is simpler than a global root-layout overlay because the trigger is scoped to the page that knows about day advances (no need for a global context or a router event listener).

- **D-02 — Advance detection via `useRef` storing the previous dayNumber + a `useEffect` comparing prev vs. current.** A naive `useEffect` keyed on `dayNumber` would fire on EVERY mount (including first paint of Day 1), not just advances. The useRef pattern stores the previous value across re-renders without triggering them; the useEffect compares prev < current and only flips `showOverlay` when an advance is detected. Backward navigation (e.g., 3 -> 2 via the back button) does NOT trigger the overlay.

- **D-03 — Auto-dismiss after exactly 1.5s.** Per CONTEXT: "1.5s auto-dismiss". The setTimeout is `1500ms`. A 200ms fade-out begins at 1500ms (via the `visible` state flipping); the parent unmount happens after the fade completes (the parent's useEffect schedules a follow-up timeout at 1500 + 200 = 1700ms to set `showOverlay` false). For `prefers-reduced-motion: reduce`, the fade-in + fade-out collapse to instant, BUT the 1.5s hold is preserved — the message still needs to be readable.

- **D-04 — Day 3 completion is OUT OF SCOPE for the overlay.** The existing flow at `DayPageClient.tsx:248` already handles Day 3 bedtime via the milestone toast + auto-redirect to `/summary`. Adding a Day 3 -> ??? overlay would be redundant (and there's no Day 4 to navigate to). The overlay covers only Day 1 -> Day 2 and Day 2 -> Day 3.

- **D-05 — FMV tooltip is only meaningful from Day 2 onward.** Per CLAUDE.md memory `ipc-calculations.md`: "Day 1 excluded from 24HV, NPi, AVV (adaptation period)". FMV is still flagged in the data model on Day 1, but it doesn't influence clinical metrics until Day 2. Surfacing the tooltip on Day 1 would teach the patient about a metric that doesn't matter yet. The tooltip is gated by `dayNumber > 1`.

- **D-06 — FMV tooltip dismissal persists via Zustand store action, NOT direct localStorage write.** Per CONTEXT key planning question #3 + recommendation: "Recommend store action for IndexedDB persistence." The store's persist middleware writes to IndexedDB (per `src/lib/store.ts:382`); calling the new `markFmvTooltipShown()` action goes through the canonical persistence path. Direct `localStorage.setItem` would bypass the migration logic and create a parallel persistence channel — fragile and confusing.

- **D-07 — Tooltip dismissal is via close button OR backdrop tap OR ESC keypress.** Three dismissal paths cover the common interactions: (a) explicit "Got it" button for the deliberate user, (b) backdrop tap for the impatient user, (c) ESC key for the keyboard user. All three converge on the same dismiss handler that calls `markFmvTooltipShown()` then unmounts.

- **D-08 — Overlay uses `role=\"status\"` + `aria-live=\"polite\"`, NOT `role=\"alert\"`.** `alert` is for urgent, interrupting messages (e.g., "Your session is expiring"); `status` with `polite` is for informational acknowledgments that don't demand immediate attention. The day-transition overlay is celebratory + informational — `status` is correct.

- **D-09 — Tooltip uses `role=\"dialog\"` + focus trap.** The tooltip is a modal-ish surface (the user must dismiss it before continuing), so `role=\"dialog\"` is appropriate. A simple focus-trap is implemented by setting `tabIndex={-1}` on the tooltip root + focusing it on mount + handling ESC to dismiss. Focus returns to the document body on dismiss (the user can re-tab through the page normally).

- **D-10 — Caret on the FMV tooltip points UP toward the timeline event.** The tooltip is bottom-fixed (above the bottom-nav at `bottom-24`); the morning-pee event in the timeline is somewhere in the middle of the page. The caret is a small CSS triangle on the TOP edge of the tooltip card, centered horizontally. It's a visual hint that the tooltip is "about something above me" — not a precise anchor (precise anchoring would require dynamic positioning logic that's overkill for a once-per-user moment).

- **D-11 — `dayTransition` and `fmvTooltip` are NEW top-level i18n namespaces.** Rationale: both are NEW micro-interaction surfaces with their own copy. Grouping under `diary.*` or `timeline.*` would dilute those existing namespaces; top-level keeps the keys discoverable. The `i18n-sync` hook handles the auto-mirror to fr/es/pt/zh/ar.

- **D-12 — Overlay timing is calibrated for readability, NOT for ambition.** 200ms fade-in + 1.5s hold + 200ms fade-out. The 1.5s hold gives a slow reader enough time to absorb the two-line message (typical reading speed: ~250 words/min; a 5-word message takes ~1.2s to read). Going shorter (e.g., 1s) would feel rushed. Going longer would feel intrusive when the user has already mentally moved on. 1.5s is the CONTEXT-locked value.

- **D-13 — Tooltip is a CONTROLLED, dismissed-on-demand component, NOT timed.** Unlike the day-transition overlay, the FMV tooltip stays open until the user dismisses it. The patient may need to read the body twice (it's clinical content). Auto-dismissing it after N seconds would be condescending — the patient knows when they've read it.

- **D-14 — Persist version bump 4 -> 5; migration extends the v3 -> v4 chain established by 15-01.** The migration function (after both plans land) checks for both `hapticEnabled` AND `fmvTooltipShown` and defaults each idempotently. The persist `version: 5`. Existing users wake up with `fmvTooltipShown = false` (the new-user default) so they get to see the tooltip once. Existing users who have ALREADY seen FMV-flagged voids on prior sessions will still see the tooltip on their next Day 2/3 morning — slightly redundant for them but harmless.
