**Key design decisions resolved here (locked, NOT for executor to revisit):**

- **D-01 — Custom inline SVG, NOT Lottie.** Per CONTEXT key planning question #2: Lottie would add a new dependency (~80KB minified) for a single decorative animation. A hand-rolled inline SVG with CSS keyframe animation is <2KB and adds zero deps. The chosen technique is `stroke-dasharray` + `stroke-dashoffset` animation on the strike-through `<line>`, plus an opacity fade on the cloud `<g>`. Both are widely supported (IE11+ for stroke-dasharray; universal for opacity transitions).

- **D-02 — `prefers-reduced-motion` honored via the EXISTING global rule at `src/app/globals.css:94-103`.** That rule already collapses ANY `animation-duration` to `0.01ms !important` and `transition-duration` to `0.01ms !important` for reduced-motion users. We do NOT add a second `@media (prefers-reduced-motion: reduce)` block for the new keyframes; the existing global rule handles it. Verification: a reduced-motion user opens the page, sees the final state (cloud at opacity 1, strike line fully drawn) on first paint with no perceptible animation.

- **D-03 — Disclosure widget: native `<details>/<summary>`, NOT a custom collapsible.** Per the canonical pattern at `src/app/[locale]/help/page.tsx:69-87`. Mirrored verbatim:
  - Tailwind classes: `group rounded-2xl bg-white border border-ipc-100 overflow-hidden`
  - Summary: `flex items-center justify-between px-5 py-4 cursor-pointer list-none text-base font-semibold text-ipc-950 [&::-webkit-details-marker]:hidden`
  - Chevron rotation: `transition-transform -rotate-90 group-open:rotate-[-270deg] shrink-0 ml-2 rtl:scale-x-[-1]`
  - Body: `px-5 pb-4 text-base text-ipc-700 leading-relaxed`

  Why native: codebase consistency (Phase 13 also uses this pattern at `ExportActions.tsx`'s More options disclosure), SSR-safe (closed default = static HTML), free a11y (browser provides aria-expanded), free RTL (chevron flip via `rtl:scale-x-[-1]`).

- **D-04 — REUSE existing `landing.privacyBody` for the disclosure body, do NOT introduce a new key.** Per the existing copy at `messages/en.json:83`:

  > "Your data never leaves your phone. No account needed, no sign-up, nothing stored on our servers. When you're done, you choose what to share with your health professional."

  This is already 6-locale parity, already register-checked, already audited by past `naturalize-prose` passes. Introducing a new disclosure-body key would mean 6 new translations + a new register pass. Reuse saves that work and preserves voice consistency. The disclosure SUMMARY ("How is my data protected?") IS a new key (`welcome.privacyDisclosureLabel`) — that's a header question, not a body restatement.

- **D-05 — Welcome panel REPLACES the existing hero block, NOT added above it.** Per CONTEXT EM-01 ("Welcome panel above the existing wizard"): the welcome panel IS the new hero. The existing hero block at `LandingContent.tsx:180-208` (logo container + heroTitle + heroSubtitle + Start Tracking Button) is REMOVED. The WelcomePanel's headline takes the h1 role; its sticky CTA takes the start-tracking role. The visual outcome: first paint shows the value framing + privacy graphic + disclosure + sticky CTA, no separate "Your 3-day bladder diary" preamble above. This is the additive empathy beat the CONTEXT envisions — the existing hero was the form; the welcome panel IS the empathy beat that replaces it.

- **D-06 — Inline privacy card at LandingContent.tsx:262-272 is REMOVED.** That block (Lock icon + privacyTitle + privacyBody) is now redundant — its content is reachable via the new disclosure inside WelcomePanel. Rendering both would be a duplicate-content UX smell. The existing footer at lines 274-295 stays (disclaimer + IPC link + privacy/terms links) — only the inline privacy CARD between the install prompt and the footer is removed.

- **D-07 — Welcome panel's sticky CTA mounts the existing OnboardingFlow via the existing `setShowOnboarding(true)` toggle.** The handler:

  ```tsx
  <WelcomePanel
    onStart={() => {
      track('start_tracking');  // existing analytics event, preserve verbatim
      setShowOnboarding(true);
    }}
  />
  ```

  The `track('start_tracking')` call MUST be preserved — it's an existing analytics event used by the conversion-funnel dashboard. No new analytics events are introduced in 14-01 (welcome impression isn't its own event; the `start_tracking` event already signals "user converted from welcome to onboarding").

- **D-08 — Sticky CTA implementation: standard `<Button variant="hero">`, NOT a position:sticky element.** "Sticky" in CONTEXT is rhetorical (the CTA stays in view as the patient scrolls the privacy graphic + disclosure). The actual implementation: place the Button INSIDE the WelcomePanel above the PrivacyGraphic. The panel is short enough (headline + body + button + graphic + disclosure ≈ 600px tall on mobile) that scrolling isn't the dominant interaction. If post-ship analytics show scroll abandonment, we can convert it to `position: sticky` in a focused follow-up. For now: standard inline placement, no sticky positioning, no new wrapping div.

- **D-09 — Step transition animation timing.** The cloud-fade is 200ms (matches the existing Boomer-safe cap from memory `ux_philosophy`). The strike-line-draw is 300ms (slightly above the cap, but the 200ms cap is for INTERACTION transitions; this is a first-paint DECORATIVE reveal where 300ms reads as a deliberate gesture rather than a snap). Total animation budget: 500ms (cloud fade 0-200ms, strike draw 200-500ms, sequenced via `animation-delay: 200ms` on the strike line). After 500ms the user has the full visual at rest.

- **D-10 — i18n: new `"welcome"` top-level namespace, NOT nested under `landing.*` or `onboarding.*`.** Per CLAUDE.md project-skills convention + existing namespace structure (`messages/en.json` has top-level `common`, `landing`, `onboarding`, `timeline`, `summary`, etc.). The welcome panel is a NEW surface that mounts before onboarding starts; giving it its own namespace makes the keys easier to grep + easier to evolve. The PostToolUse `i18n-sync` hook handles the auto-mirror to fr/es/pt/zh/ar when `messages/en.json` is edited.
