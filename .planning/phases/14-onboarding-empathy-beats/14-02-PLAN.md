**Key design decisions resolved here (locked, NOT for executor to revisit):**

- **D-01 — Inline custom SVG, NOT a PNG screenshot.** Per CONTEXT key planning question #1: an SVG scales cleanly at retina + zoom + screen-reader-zoom, can be locale-aware (header text re-renders per locale via `useTranslations`), adds zero bytes beyond the component itself, and survives Next.js static export without an image-loader workaround. A PNG would: (a) require generating 5 locale-specific images (en/fr/es/pt/zh/ar) to avoid locale mismatches, (b) consume disk space in `public/`, (c) require alt-text plumbing, (d) lose sharpness at large zoom. Inline SVG is unambiguously the right call.

- **D-02 — Boomer-safe: SVG is fully STATIC.** Per CONTEXT ("Static, no animation. Boomer-safe.") and CLAUDE.md memory `ux_philosophy`. No `<animate>`, no `<animateTransform>`, no CSS keyframes inside the SVG. No hover state. No motion. The preview's job is to show the deliverable, not to entertain. `prefers-reduced-motion` is a non-issue because there's nothing to reduce.

- **D-03 — Locale-aware text INSIDE the SVG via inline `<text>` elements bound to next-intl, NOT a single locale-agnostic image.** The inline SVG renders `<text>` tags whose `children` are the result of `t(key)`. This works in static export because next-intl's RSC-safe `useTranslations` hook (client component variant) reads the locale from React context at render time. No build-time string baking required; each locale's static-export HTML embeds the locale-correct strings into the SVG's `<text>` tags.

- **D-04 — `text-ipc-300` for ALL SVG strokes + light-grey fills.** The preview is a SUBTLE document silhouette, not a vivid hero. The job is to communicate "you'll produce a clinical-looking deliverable", not to upstage the WelcomePanel above. `text-ipc-300` matches the footer's text shade — quiet, low-contrast against the cream surface, but legible. The caption below the SVG uses `text-ipc-600` (one shade darker) so the caption-as-narration is more readable than the silhouette-as-illustration.

- **D-05 — Per-step time-estimate text is `text-xs text-ipc-500 mt-1`.** Per CONTEXT EM-04 ("subtle copy near the step indicator"). The size + colour matches the existing `tc('stepOf', { current: step, total: 3 })` rendering at OnboardingFlow.tsx:99-101 EXCEPT the time estimate is one size smaller (`text-xs` instead of the existing `text-[11px]`). This visual hierarchy makes the step-of label the primary wayfinding signal and the time estimate the secondary expectation-setting signal.

- **D-06 — Time-estimate values come from a single ternary, NOT three separate JSX blocks.** Each step (1/2/3) has its own `onboarding.timeEstimate.stepN` key, and the OnboardingFlow renders the current step's estimate via `t(\`timeEstimate.step${step}\` as const)`. This keeps the wizard's step rendering symmetric — there's one shared step-indicator block at the top, and the time estimate within it is computed from `step`. No duplication; no per-step JSX surgery.

- **D-07 — Time-estimate copy values: step1=\"~10 seconds\", step2=\"~10 seconds\", step3=\"~30 seconds\".** Per CONTEXT EM-04 (\"`~10 seconds` / `~30 seconds`\" anchor). Rationale for the split: step 1 is single numeric input (age), step 2 is one-of-two button tap (mL or oz), step 3 has both the date picker + the timezone picker + the 3-day preview, so the realistic time-to-complete is roughly 3x the simpler steps. The `~` (tilde) before the value softens the claim — \"about 10 seconds\", not \"exactly 10 seconds\".

- **D-08 — `sampleExport` is a NEW top-level i18n namespace, NOT nested under `landing.*` or `onboarding.*`.** Rationale: the sample-export preview is a NEW surface that gets its own keys; grouping under `landing.*` would dilute landing's already-busy namespace. The `timeEstimate.*` sub-namespace IS nested under `onboarding.*` because the time-estimate copy is wizard-scoped — it has no meaning outside the OnboardingFlow component.

- **D-09 — i18n auto-mirror via the existing PostToolUse hook.** Per CLAUDE.md `## Configuration` + memory `project_translation_workflow`, every edit to `messages/en.json` auto-triggers `i18n-sync`, which mirrors new keys into the 5 non-EN locale files. The executor (this plan's runtime) ADDS the new keys to `messages/en.json` ONLY; the hook fires automatically and produces the 5 non-EN files. After the hook runs, the executor verifies parity via the `article-i18n-completeness.sh` checks. If `i18n-sync` registers a stiff rendering, a `naturalize-prose` follow-up may refine — but that's a separate pass, NOT part of this plan's RED→GREEN.

- **D-10 — Insertion point in `LandingContent.tsx` is BELOW the WelcomePanel + ABOVE the PWA install prompt.** After 14-01 ships, the `!diaryStarted && !showOnboarding` branch renders: `<WelcomePanel onStart={...} />` followed by a `<Container>` that wraps the PWA install prompt + footer (per 14-01's restructure). 14-02 inserts `<SampleExportPreview />` as the FIRST child inside that `<Container>` (i.e., directly above the PWA install prompt). The composition becomes: WelcomePanel → SampleExportPreview → PWA install prompt → footer. The PWA install prompt + footer remain UNTOUCHED.

- **D-11 — `SampleExportPreview` does NOT depend on `WelcomePanel` directly.** The component is freestanding; it could in principle be dropped anywhere. The LandingContent.tsx edit just happens to place it below WelcomePanel. This decoupling means a future phase could surface the preview elsewhere (e.g., the summary page completion celebration in Phase 16) without rewiring.

- **D-12 — SVG dimensions: 280px × 360px ratio.** US-Letter at 8.5 × 11 = 0.773 aspect ratio. A landscape mini-preview at small scale: 280px wide × 360px tall = 0.778 aspect ratio (close enough; 280px is a nice round value that fits inside the editorial reading column on both mobile and md+ desktop). `width=\"280\" height=\"360\"` attributes, `viewBox=\"0 0 280 360\"`. The component itself is centered via `mx-auto`.
