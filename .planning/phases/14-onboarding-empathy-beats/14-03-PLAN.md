**Key design decisions resolved here (locked, NOT for executor to revisit):**

- **D-01 — \"Checkmark glow\" expressed as a ring halo on the active step-dot, NOT a literal check icon.** Per CONTEXT EM-05 (\"subtle visual beat (checkmark glow + step-indicator advance with motion) acknowledges progress\"). The existing OnboardingFlow uses CIRCULAR dots, not check icons, for the step indicator. Introducing a checkmark glyph would require restructuring the step indicator visual vocabulary (and possibly displacing the existing dots). The empathy beat is adapted to fit: the active dot gets a subtle ring halo, which reads as \"glow on the current step\" without breaking the existing dot affordance. This adheres to CONTEXT's \"Boomer-safe: NO confetti, NO sound, NO overlay — just refinement of the existing step transition\" — the ring is a refinement, not a replacement.

- **D-02 — Pulse via React `key` prop refresh, NOT useEffect + setTimeout class swap.** The `animate-step-pulse` keyframe fires on EVERY render of the dot element. If the dot uses `key={\`dot-${step}\`}`, React unmounts the dot on every step change and remounts it — the keyframe runs once per step change. This is preferable to useEffect + setTimeout class-add-then-remove because: (a) no JS state to manage, (b) no React re-render churn from class manipulation, (c) the keyframe `both` fill mode handles the final state correctly even if React re-renders mid-animation.

- **D-03 — Existing `animate-step-in-left` / `animate-step-in-right` keyframes tuned to 180ms.** Per CONTEXT success criterion #4 (\"Step transitions animate (180ms cubic-bezier, under the Boomer-safe 200ms cap)\"). The existing values at globals.css:651-657 are `250ms ease-out` — above the 200ms cap. Replacing 250ms → 180ms and `ease-out` → `cubic-bezier(0.4, 0, 0.2, 1)` (the standard \"emphasized\" Material easing) brings them inside the cap AND aligns with the new `animate-step-pulse` curve, producing a coherent step-transition gesture (dot pulses while the new step content fades in, both completing within 180ms).

- **D-04 — Animation timing budget: 180ms total per step transition.** stepPulse (180ms) + stepInLeft/Right (180ms) run in PARALLEL, both starting at the step change. Both complete at 180ms. Total user-perceived motion duration: 180ms. Well under the Boomer-safe 200ms cap.

- **D-05 — `prefers-reduced-motion: reduce` handled by the EXISTING global rule at globals.css:94-103.** That rule already collapses ALL `animation-duration: 0.01ms !important` and `transition-duration: 0.01ms !important` for reduced-motion users. We do NOT add a second `@media (prefers-reduced-motion: reduce)` block for the new keyframes. Verification: a reduced-motion user advances steps in the wizard and sees the new step content appear INSTANTLY (no pulse, no fade) with the active dot ring already in place. The visual state is the same; the motion is removed.
