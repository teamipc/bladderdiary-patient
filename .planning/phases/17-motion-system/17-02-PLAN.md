This plan delivers MOT-03 (page-to-page transitions for the diary flow) via a thin layer of new layouts + a client wrapper component + a Tailwind keyframe fallback + a future-proof useViewTransition hook:

The hook is CANONICAL API surface for FUTURE explicit View Transitions API triggers; not invoked in this plan because Next.js App Router does not expose a navigation-completion hook to attach to.

**Key design decisions resolved here (locked, NOT for executor to revisit):**

- **D-01 — PageTransitionWrapper, NOT inline layout logic.** Per CLAUDE.md `## Conventions > Module Design` (one component per file). Inlining the wrapper into the day layout would duplicate the logic in summary/layout.tsx. A shared component keeps the behavior coherent.

- **D-02 — `key={pathname}` re-mount strategy, NOT JS-managed animation restart.** The simplest cross-browser way to re-fire a CSS keyframe is to mount a new DOM node — `key={pathname}` on the wrapper div forces React to unmount the old wrapper subtree and mount a fresh one when the pathname changes. CSS animations fire on element creation by default. Alternative: imperatively reset the `animation` property via `el.style.animation = 'none'; void el.offsetHeight; el.style.animation = '';` (the classic forced-reflow hack). The key-based approach is cleaner, idiomatic React, and avoids the imperative DOM mutation.

- **D-03 — Subtle 4px upward slide + cross-fade, NOT horizontal slide.** Per CONTEXT MOT-03 the transition is a "slide / cross-fade between /diary/day/1 → /diary/day/2 → /diary/day/3 → /summary". A horizontal slide would feel like a swipe gesture (and would conflict with the existing horizontal scroll on the diary timeline cards). A vertical 4px slide is a HINT of physical continuity without being a directional gesture. Cross-fade is the primary effect (opacity), slide is the accent (transform).

- **D-04 — PageTransitionWrapper lives at `src/components/layout/`, NOT co-located with the layout.** Per the existing src/components/layout/ pattern (AppShell, Container, Footer, BottomNav, etc.). The wrapper is a layout primitive; layout-folder is its natural home. Reusable from both the diary day layout AND the summary layout.

- **D-05 — View Transitions API is OPPORTUNISTIC, NOT required.** Per CONTEXT MOT-03 constraint: "Uses View Transitions API where supported with a Tailwind keyframe fallback." The keyframe IS the primary user-visible mechanism; the View Transitions API integration is a future-proof API surface (`useViewTransition` hook) for explicit transitions that don't depend on Next.js's navigation-completion lifecycle. The `useViewTransition` hook is created NOW so future consumers (e.g., a tab component, an explicit modal transition) can call it without inventing parallel patterns.

- **D-06 — Next.js 16 App Router does NOT have built-in View Transitions API integration.** The user-supplied CONTEXT mentions View Transitions API where supported. The honest reading: Next.js's router does not expose a 'before navigation completes' hook that the wrapper could attach to. The KEYFRAME mechanism (via `key={pathname}`) IS the cross-browser, Next.js-version-agnostic, View-Transitions-API-or-not mechanism. The View Transitions API in the CONTEXT is the FUTURE roadmap; the immediate deliverable is the keyframe.

- **D-07 — `useViewTransition` hook is created EVEN THOUGH no consumer in this plan calls it.** Forward-compatibility. Future plans / phases that explicitly trigger a transition (e.g., 'animate the metric block expanding') can call `useViewTransition()` and receive a callback that wraps their state-change in `startViewTransition` if supported. Creating the canonical API now prevents future developers from inventing parallel matchMedia + feature-detect inline code.

- **D-08 — The hook's feature-detection is at MODULE LOAD, NOT inside the returned callback.** Module-level `const SUPPORTS_VIEW_TRANSITIONS = typeof document !== 'undefined' && 'startViewTransition' in document;` is computed once. The callback closes over the constant. Avoids per-call feature-detection overhead. SSR-safe: during server-render `document` is undefined, the constant is false, the callback degrades to direct invocation.

- **D-09 — Cross-fade + 4px upward slide is the keyframe; NO `scale` transform.** Scale transforms feel like zoom/click responses; the page transition should NOT feel like a zoom. The Y-axis 4px slide is subtle physical continuity; opacity is the primary visual.

- **D-10 — Duration `var(--duration-normal)` = 180ms, NOT --duration-slow.** Page transitions are interactions (the user clicked a Link); they fall under the Boomer-safe 200ms cap. The 180ms `normal` token is the canonical interaction duration from 17-01. Using `--duration-slow` (300ms) for page transitions would feel sluggish.

- **D-11 — Easing `var(--ease-decelerated)` (entrance curve), NOT --ease-emphasized.** The new page is ENTERING the viewport — decelerated easing slows it into rest, which is the canonical entrance feel. Emphasized (the symmetric Material curve) is for elements that both enter AND exit; decelerated is the entrance specialist.

- **D-12 — Wrap the children at the LAYOUT level, NOT at the page level.** Layouts persist across route changes within the same segment, so the wrapper's children change but the wrapper component itself is preserved. Using `key={pathname}` on a layout-level wrapper still re-mounts the inner subtree on pathname changes (Next.js App Router preserves the layout BUT the children prop changes; the wrapper component itself receives a new `key` value derived from the new `pathname`, which forces React's reconciler to unmount + remount). This is the canonical pattern for route-change-triggered animations in Next.js App Router.

- **D-13 — Day layout creation: NEW file at `src/app/[locale]/diary/day/[dayNumber]/layout.tsx`.** Next.js App Router permits layouts at any segment. The new file is a thin SERVER component (no `'use client'`) that imports the client PageTransitionWrapper and renders it around children. Server-component layout + client-component wrapper is the canonical Next.js pattern — keeps the layout statically generatable and SEO-friendly while enabling client-side interactivity inside the wrapper.

- **D-14 — Summary layout modification: ADD the wrapper.** The existing `src/app/[locale]/summary/layout.tsx` is a minimal layout that sets metadata. Wrapping `{children}` in `<PageTransitionWrapper>` is a pure ADDITIVE change — metadata still sets correctly, children still render, the wrapper adds the keyframe class.

- **D-15 — NO modification of `src/app/[locale]/diary/layout.tsx`.** The diary parent layout owns the Container chrome (per existing code at `src/app/[locale]/diary/layout.tsx:9-19`). Wrapping AT THE PARENT level would put the keyframe on the Container itself, animating the back link / chrome AND the page content together. We want ONLY the inner page content to animate. Wrapping at the [dayNumber] / summary level (one layer DEEPER) keeps the Container static across navigations and animates only the page-specific content.

- **D-16 — Continue button + Back link are UNCHANGED.** The existing `<Link>` components in DayPageClient.tsx and summary/page.tsx are unchanged. The navigation triggers a pathname change; the wrapper handles the visual transition via the keyframe re-fire. Zero per-Link instrumentation.

- **D-17 — The 200ms / 1500ms timings in DayTransitionOverlay.tsx are UNCHANGED.** That component is a SEPARATE feature (Phase 15 MI-03 — Day 1→2→3 transition acknowledgment OVERLAY). It is a one-shot in-page overlay, not a route transition. The new PageTransitionWrapper supplements it: when the user dismisses the overlay → tap Continue → navigate to Day 2, BOTH the overlay-dismiss-fade AND the page-transition keyframe fire (sequentially). The combined feel is: overlay fades out, new page slides up. Acceptable per CONTEXT MOT-03 (the wrapper does not conflict with existing overlays).

- **D-18 — RTL safety: Y-axis-only transforms.** The 4px upward slide is direction-neutral. The keyframe does NOT use translateX which would differ in LTR vs RTL.

The new `useReducedMotion` hook at `src/lib/hooks/useReducedMotion.ts` (from 17-01):

```tsx
'use client';
import { useEffect, useState } from 'react';

export default function useReducedMotion(): boolean {
  const [reduced, setReduced] = useState(false);
  useEffect(() => {
    if (typeof window === 'undefined') return;
    const mq = window.matchMedia('(prefers-reduced-motion: reduce)');
    setReduced(mq.matches);
    const onChange = (e: MediaQueryListEvent) => setReduced(e.matches);
    mq.addEventListener('change', onChange);
    return () => mq.removeEventListener('change', onChange);
  }, []);
  return reduced;
}
```

     * Computed once at module load (NOT per call) for performance.
     * SSR-safe: returns false during server-render because `document` is

    /**

     *
     * Returns a stable callback `runWithTransition` that accepts a callback
     * `cb` and either:
     *   (a) wraps `cb` in `document.startViewTransition(() => cb())` if the
     *       browser supports the API AND the user has NOT requested

     *   (b) calls `cb()` directly (synchronous, no transition).

     * way to handle route-level transitions.
     *
     * Phase 17 MOT-03.
     */
    export default function useViewTransition(): (callback: () => void) => void {
      const reducedMotion = useReducedMotion();

      return useCallback(

          if (!SUPPORTS_VIEW_TRANSITIONS || reducedMotion) {

          // can be removed.
          const doc = document as Document & {
            startViewTransition: (cb: () => void) => unknown;
          };
          doc.startViewTransition(callback);
        },
        [reducedMotion],
      );
