**Key design decisions resolved here (locked, NOT for executor to revisit):**

- **D-01 — BottomSheet refinement is TOKENS-ONLY, not a visual retune.** Per CONTEXT key planning question #3 recommendation: "just consume tokens, or also visually retune (e.g. tighter spring feel)? Recommend tokens-only — visual is already good per Phase 6 review." The migration changes the easing curves (a side-effect of pointing utilities at token values) but does NOT change the keyframe shapes, durations, or BottomSheet component code.

- **D-02 — .animate-modal-in easing migrates from `cubic-bezier(0.16, 1, 0.3, 1)` to `--ease-emphasized` (`cubic-bezier(0.4, 0, 0.2, 1)`).** The out-expo curve was Phase 6's choice for a "confident decelerated finish at short duration"; the Material emphasized curve is the Phase 17 canonical token. Visual diff: slightly less dramatic deceleration; smoother settle. Both curves end at the same point (1, 1) so the destination is identical; the rate of approach differs. Sub-perceptual at 180ms duration.

- **D-03 — .animate-slide-up easing migrates from CSS `ease-out` keyword to `--ease-decelerated` (`cubic-bezier(0.0, 0.0, 0.2, 1)`).** CSS `ease-out` is equivalent to `cubic-bezier(0, 0, 0.58, 1)` (per CSS spec). Our `--ease-decelerated` is `cubic-bezier(0.0, 0.0, 0.2, 1)` — slightly more pronounced deceleration (the control point at 0.2 is closer to the start, so the curve "front-loads" the motion). Sub-perceptual at 300ms duration.

- **D-04 — .animate-slide-up duration stays at 300ms (semantic-token reference).** The previous declaration was `0.3s` (300ms numeric); the new declaration is `var(--duration-slow)` (also 300ms semantic). No visual change in duration; just locks the value to the token.

- **D-05 — Shimmer implementation: linear-gradient background-position shift, NOT opacity pulse.** The canonical CSS shimmer pattern (used by GitHub, Stripe, and many design systems) uses a linear-gradient with shifted background-position. The keyframe shifts `background-position: -200% 0` to `200% 0` over 1500ms, with `background-size: 200% 100%` so the gradient is twice as wide as the element. The gradient direction (`90deg` = left-to-right) is consistent regardless of LTR/RTL — shimmer effects don't have semantic direction. Alternative (opacity pulse) would feel like a heart-beat, less polished.

- **D-06 — Shimmer iteration 1500ms is intentional out-of-token.** Ambient animations (the shimmer continues indefinitely until the real content replaces the placeholder) don't fit the 200ms cap. 1500ms is a gentle, non-distracting rhythm — fast enough to communicate "loading" without feeling slow, slow enough not to feel flickery. Per D-12 in 17-01 there is no `--motion-duration-shimmer` token (overkill for one usage). Annotate as intentional out-of-token in the comment.

- **D-07 — Reduced-motion override: SOLID COLOR, NOT collapsed animation.** The global `prefers-reduced-motion: reduce` rule would collapse the shimmer's `animation-duration` to 0.01ms — but because the keyframe shifts background-position, collapsing the duration freezes the gradient in whatever position it happened to land at (could be off-center, looks broken). To handle this gracefully, the `.animate-skeleton-shimmer` utility includes an explicit `@media (prefers-reduced-motion: reduce)` override that sets `background: var(--color-ipc-100); animation: none;` — completely eliminating the gradient and the animation. Reduced-motion users see a solid `--color-ipc-100` rectangle as the placeholder.

- **D-08 — Three variants: text / card / metric.** Per CONTEXT key planning question #4 recommendation: "Recommend new <Skeleton> component with variants (text, card, metric)". Variant CSS:
  - `text`: `h-4 w-32 rounded` (single short line; common label/value placeholder)
  - `card`: `h-32 w-full rounded-2xl` (full-width block; article hero or large card)
  - `metric`: `h-16 w-full rounded-2xl` (sized to match an IpcMetricsBlock tile)
  Future variants (`avatar`, `icon`, etc.) can be added as needed without breaking the existing API (the prop is a string union, additive).

- **D-09 — `className` prop is optional, merged via simple template literal.** No `clsx` or `tailwind-merge` dependency added (none in package.json currently). The variant base class + the user's className are joined with a space. For 99% of use cases the variant + className covers everything; if a consumer needs sophisticated merging they can pass already-merged classes.

- **D-10 — `aria-label` falls through to localized 'Loading' default.** Per the canonical loading-state a11y pattern: `role='status'` + `aria-busy='true'` + a visually-hidden span with the localized label. SR users hear 'Loading' (or the custom label). Sighted users see the shimmer placeholder. Both modalities communicate the loading state.

- **D-11 — Visually-hidden label uses Tailwind's `sr-only` utility.** The `sr-only` class hides content visually while keeping it accessible to screen readers. Standard Tailwind utility; no custom CSS needed.

- **D-12 — Summary page integration: hydration-gated SWAP, NOT additive overlay.** The existing IpcMetricsBlock (from 16-02) is conditionally rendered on `isComplete`. The Phase 17 enhancement: when `!hydrated`, show a 2x2 grid of Skeleton placeholders FIRST; when `hydrated`, swap to the IpcMetricsBlock (if `isComplete`) or render nothing (if incomplete diary). This eliminates the previous "blank space then sudden metric block appearance" hydration FOUC (flash of unrendered content).

- **D-13 — Summary skeleton uses `variant='metric'` × 4 in the same grid layout.** Layout-stability: the placeholder grid uses `grid grid-cols-2 md:grid-cols-4 gap-2` (identical to IpcMetricsBlock's grid). Each Skeleton with `variant='metric'` has `h-16 w-full` so the placeholder tile height matches the real tile height (the real tile renders `text-2xl text-ipc-950 tabular-nums leading-none` for the number plus a small label below — approximately 64px / h-16 total). No layout shift on hydration → no Cumulative Layout Shift (CLS) score impact.

- **D-14 — Learn article integration is COMMENT-ONLY in this plan.** `src/lib/mdx.tsx` uses `next-mdx-remote/rsc` which is a server-component rendering pattern — the MDX is fully resolved at build time for static-export. There is no client-side loading state to wrap. The Skeleton component is the canonical placeholder for FUTURE dynamic loading (search results, async glossary entries, etc.). Annotate this in mdx.tsx as a forward-looking note.

- **D-15 — One new i18n key under `common` namespace, NOT a new namespace.** `common.loading` is the canonical home for cross-surface generic copy (per the existing `common.close`, `common.cancel`, etc. patterns at `messages/en.json` common namespace). The Skeleton component imports `useTranslations('common')` and uses `t('loading')`.

- **D-16 — Boomer-safe: shimmer is subtle (low-contrast brand-palette gradient).** The gradient stops are `var(--color-ipc-50)` (very light) and `var(--color-ipc-100)` (slightly darker). Both are decorative tints, not high-contrast accent colors. The shimmer is gentle motion across a soft amber wash. No high-contrast strobing, no rainbow gradient, no eye-catching colors.

- **D-17 — No new dependencies.** The Skeleton component uses only React + lucide-react (already a dependency for the optional icon variant if added later — not used in this plan). The shimmer uses CSS only.

- **D-18 — RTL safety: shimmer gradient direction is LEFT-TO-RIGHT in both LTR and RTL.** Shimmer effects don't have semantic direction — they're visual placeholders, not reading-direction indicators. The 90deg linear-gradient (CSS direction conventions: 90deg = left-to-right) renders identically in both directions. The Skeleton variant CSS uses direction-neutral classes (`w-full`, `w-32`, `h-4`, `rounded`).

**Out of scope for 17-03 (deferred to later plans):**

- Tactile retune of BottomSheet motion (e.g., tighter spring feel, longer rest pause) — per D-01 + CONTEXT recommendation, tokens-only refinement.
- Migrating OTHER globals.css keyframes (e.g., `fadeSlideUp`, `scaleIn`, `stepInLeft`, etc.) to use tokens — only the BottomSheet-relevant keyframes migrate in this plan; other keyframes stay as-is per 17-01 D-13.

- Wiring Skeleton into a third consumer beyond the summary page + the mdx.tsx comment annotation (e.g., diary day pages, onboarding) — not currently needed; the existing diary day pages already have a useStoreHydrated() gate with `null` return (no FOUC visible).
- INERT verification spec covering all 5 MOT requirements (17-04 owns it).

This plan does NOT modify the diary forms, the diary day pages, the onboarding flow, the store, the types, or any non-summary surface beyond the Skeleton component itself + the mdx.tsx comment + the 6 locale files. It does NOT add new dependencies.
