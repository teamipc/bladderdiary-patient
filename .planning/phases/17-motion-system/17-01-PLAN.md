**Key design decisions resolved here (locked, NOT for executor to revisit):**

- **D-01 — Tailwind 4 `@theme inline` for token install, NOT a separate config file.** Per CONTEXT key planning question #2: "MOT-01 token naming: Tailwind's existing duration-* utilities collide with our intended names. Recommend Tailwind extend (less friction)." Tailwind 4 does not use a `tailwind.config.ts` file — it uses the `@theme` directive inside CSS. The install site is `src/app/globals.css` lines 17-53 (the existing `@theme inline` block). Adding `--duration-fast/normal/slow` + `--ease-emphasized/decelerated/accelerated` tokens THERE makes them available as both Tailwind utilities (`duration-fast`, `ease-emphasized`, etc.) AND as CSS custom properties (`var(--duration-fast)`, `var(--ease-emphasized)`).

- **D-02 — Token names: `duration-fast/normal/slow` + `ease-emphasized/decelerated/accelerated`.** Per CONTEXT MOT-01 + the question #2 recommendation: extend rather than namespace. Built-in Tailwind utilities like `duration-200` (numeric) do NOT collide with our named tokens (`duration-fast`); the Tailwind utility namespace permits both. No `mfc-` prefix needed.

- **D-03 — 3 duration tokens: fast=120ms, normal=180ms, slow=300ms.** Per CONTEXT MOT-01 spec. The 180ms `normal` aligns with the existing `modalIn` keyframe (globals.css:194) and the Phase 14/15 motion vocabulary. The 120ms `fast` covers micro-interactions (chip taps, button hovers). The 300ms `slow` covers slightly longer reveals (skeleton shimmer iteration, page transitions, observation cards). These are SOFT targets — the Boomer-safe 200ms cap remains for INTERACTIONS, but `slow=300ms` is acceptable for AMBIENT reveals.

- **D-04 — 3 easing tokens: emphasized (Material standard), decelerated (entrance), accelerated (exit).** The `emphasized = cubic-bezier(0.4, 0, 0.2, 1)` is THE curve used throughout Phases 14/15/16; locking it as the canonical token. The `decelerated = cubic-bezier(0.0, 0.0, 0.2, 1)` is the iOS HIG / Material "ease-out" curve — used for elements entering the screen. The `accelerated = cubic-bezier(0.4, 0.0, 1, 1)` is the iOS HIG / Material "ease-in" curve — used for elements leaving the screen.

- **D-05 — `useReducedMotion()` SSR-safe pattern.** Returns `false` on initial render (server + first client paint pre-effect). On client mount, the useEffect reads `mq.matches` synchronously and calls setState — so the second client render reflects the real OS preference. Updates LIVE on `change` events for users who toggle their preference mid-visit (e.g., enable reduced-motion in Accessibility settings without reloading). The `change` event is the modern API (iOS 14+, Safari 14+, Chrome 84+); the deprecated `addListener` fallback is NOT included because the production target is iOS 14+ (per Phase 8 walkthrough matrix).

- **D-06 — Hook returns a boolean, NOT an object.** Per the simplest API surface: a `boolean` return is the cleanest call site (`const reducedMotion = useReducedMotion();`). Comparable to `useStoreHydrated()` (existing pattern in this codebase) which also returns a primitive. Future enhancement (returning richer metadata like `prefers-color-scheme` etc.) can be a sibling hook, not an overload of this one.

- **D-07 — Default export, NOT named.** Per the `Button.tsx` precedent in the codebase. The hook is self-contained, no co-located exports, no namespace needed. Import line is cleaner: `import useReducedMotion from '@/lib/hooks/useReducedMotion'`.

- **D-08 — Migration of AnimatedMetric.tsx is 1:1 in behavior.** The private `usePrefersReducedMotion` hook at lines 6-25 has the IDENTICAL contract (returns boolean; subscribes to matchMedia + change events; SSR-safe). The migration is purely a code-DRY pass: delete the private hook, import the shared one, change the variable name from `usePrefersReducedMotion()` to `useReducedMotion()` at the call site. ZERO behavioral change for users.

- **D-09 — Migration of ObservationCardReveal.tsx changes WHEN matchMedia is consulted.** Previously the matchMedia check ran INSIDE the useEffect (lines 60-64). With the hook, the check moves to the component TOP LEVEL (rule of hooks: hooks at top, not inside useEffect). The useEffect now reads the hook's `reducedMotion` value from closure and branches early. Behaviorally identical, but the React reconciler can now re-render the component when the user toggles their OS preference (the hook subscribes to live changes and updates state). This is an UPGRADE — previously the component only checked at mount; now it responds to mid-visit toggles.

- **D-10 — 350ms→300ms downshift on ObservationCardReveal.tsx line 99 is accepted.** The inline transition was `350ms cubic-bezier(0.4, 0, 0.2, 1)` (Phase 16-03 D-06). Migrating to `var(--duration-slow) var(--ease-emphasized)` resolves to `300ms cubic-bezier(0.4, 0, 0.2, 1)`. Visual diff: 50ms shorter reveal per card. Sub-perceptual at this scale (sequential cards with 150ms stagger; the 50ms is masked by the stagger rhythm). Locking the token vocabulary > preserving the exact prior duration.

  - The 1500ms hold in DayTransitionOverlay is the "let the user see this for a moment" timer — not an animation duration; semantically distinct from token vocabulary.
  - The 240ms dismiss-fade in CompletionHero is a one-shot exit; close to `--duration-slow` (300ms) but the original 240ms was tuned to feel "just barely longer than normal" without dragging. Migrating to `--duration-slow` would be a 60ms perceptual change. Keep as-is + annotate as intentional.

Migrating CSS keyframes to use `var(--duration-*)` is technically possible (CSS custom properties resolve in animation declarations in modern browsers), but the value is marginal because the keyframes are co-located in globals.css and reading the file shows the timing at a glance. 17-03 (BottomSheet refinement) DOES touch globals.css and migrates the BottomSheet-relevant keyframes (`slideUp`, `modalIn`) to consume tokens; other keyframes remain in their current form.
