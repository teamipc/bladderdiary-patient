**Key design decisions resolved here (locked, NOT for executor to revisit):**

- **D-01 — Count-up implementation: custom rAF hook, NOT CountUp.js.** Per CONTEXT constraint "No new heavy dependencies. No CountUp.js — write a small custom hook (~20 lines)." The internal `useCountUp` hook is ~25 LOC, uses only `requestAnimationFrame` + `useState` + `useEffect`, zero npm dependencies added. The function-name signature (target, durationMs, delayMs, prefersReducedMotion) -> number is the smallest possible surface.

- **D-02 — 4 metrics, NOT 5.** Per Phase 13 D-02 locked decision: NBC was dropped from the summary surface. The actual `DiaryMetrics` interface (src/lib/calculations.ts:43-65) exposes `periods[].twentyFourHV`, `periods[].nPi`, `periods[].avv`, and top-level `mvv` — that's 4 distinct value paths. The CONTEXT writes "5 IPC clinical metrics (24HV / NPi / AVV / MVV / NBC)" as a sketch but per Phase 13's locked decision NBC is not surfaced. This plan implements 4 tiles. The stagger math (150ms × 4 tiles = 450ms + 800ms count-up = 1250ms total) reflects 4 tiles.

- **D-03 — Period 0, not Period 1.** The `DiaryMetrics.periods` array has 2 elements (Night 1 / Day 2, then Night 2 / Day 3). Per Phase 13 D-02 + the standing IPC-calculations memory: Day 1 is the adaptation period and excluded from 24HV / NPi / AVV. So the "first scoreable period" is `periods[0]`, which represents Night 1 / Day 2. The summary surface uses `periods[0]?.twentyFourHV ?? 0` (with optional chaining + nullish coalescing because `periods` may be empty if the diary is incomplete — though we mount the block on `isComplete`). The MVV is top-level (`metrics.mvv`) because it's a cross-day max, not a per-period value.

- **D-04 — Count-up duration 800ms, stagger 150ms.** Per CONTEXT constraint: "Animation budget: count-up ~800ms total, 150ms stagger per metric". With 4 tiles, the total reveal is 0 + (3 × 150ms stagger) + 800ms = 1250ms from first paint to last settle. Acceptable per Boomer-safe per the rationale in the truths list above (the count-up IS the reward, not a one-off interaction).

- **D-05 — Easing: `easeOutCubic` = `1 - (1 - t)^3`.** Smooth deceleration; matches the character of `cubic-bezier(0.4, 0, 0.2, 1)` (Material emphasized curve) without requiring a CSS keyframe (the hook is JS-driven so it computes the eased value on every rAF tick directly). The cubic curve makes the final values feel like they "settle" rather than abruptly stop.

- **D-06 — `prefers-reduced-motion` handled INSIDE the AnimatedMetric component, NOT via a CSS media query.** Because the count-up is JS-driven (rAF + setState), the CSS `animation-duration: 0.01ms` global rule does NOT affect it (that rule targets CSS animations, not JS-controlled state updates). The component's internal `usePrefersReducedMotion` hook reads the media query directly and bypasses the rAF loop when true — the tile renders the final value instantly on first paint. The wrapper section's `animate-fade-slide-up` IS handled by the global CSS rule, so the section's fade-in collapses to instant for reduced-motion users; only the count-up needs the JS-side guard.

- **D-07 — `aria-live="polite"` on the wrapper section, `aria-busy={anyAnimating}` for the count-up phase.** Per CONTEXT constraint "aria-live=polite on the metric container — SR users hear final values once they settle, not the count-up". `aria-live="polite"` means SRs queue the change announcement until the user is idle; `aria-busy=true` while counting suppresses any interim announcement. When all 4 tiles finish, `aria-busy=false` and the live region announces the final state. Canonical Web a11y pattern for animated value reveals.

- **D-08 — `tabular-nums` on the number span.** The number changes 60 times per second during count-up; without `tabular-nums`, each digit's character width could shift slightly per frame causing visible layout jitter. The CSS feature `font-variant-numeric: tabular-nums` (which Tailwind's `tabular-nums` class applies) ensures every digit is the same width, eliminating jitter. The existing effort-stat tiles at summary/page.tsx:144 / 150 / 156 already use this class — we're just maintaining the convention.

- **D-09 — IPC metrics block sits BELOW the top-CTA, ABOVE the data-warning / story / observations.** Specifically: between lines 201 (close of top-CTA section) and 204 (open of data-warning block). Rationale: (a) the top CTA is the FIRST action the patient should take (share with clinician), so it stays maximally close to the top, (b) the metrics are the SECOND-most-important reward (the clinical achievement made visible), so they immediately follow the CTA, (c) the longer narrative content (story, observations, reflection) follows after the metrics — the patient who scrolls past the metrics is the deeply-engaged reader who wants the full narrative.

- **D-10 — Keys nest under EXISTING `summary` namespace as `summary.ipcMetrics.*`, NOT a new top-level namespace.** Per the same rationale as 16-01 D-08: IPC-metrics strings are bound to the summary surface; nesting keeps the namespace tree coherent.

- **D-11 — Locale-aware number formatting via `toLocaleString(locale)`.** Numbers in EN/FR/ES/PT use Western digits; ZH uses Western digits by default; AR may use Eastern Arabic numerals (٠١٢٣٤٥٦٧٨٩) depending on `next-intl`'s locale tag. This is the canonical behavior of `Intl.NumberFormat`; we accept it as-is (matches how the existing effort-stat tile at summary/page.tsx:144 uses `toLocaleString()` without explicit locale). If walkthrough QA later flags AR numerals as undesirable, a separate post-launch tweak passing `ar-EG-u-nu-latn` can force Western digits — out of scope for this plan.

- **D-12 — Mount on `isComplete` only.** Per the existing summary-page pattern: the locked + hero + effort-stats / top-CTA / etc. ALL gate on `isComplete`. The IPC metrics block is meaningless before Day 3 bedtime is set (the calculations return null AVV / nPi for incomplete periods). Conditional mount inside `<Container>` is `{isComplete && <IpcMetricsBlock />}` (matches the existing pattern at line 170 for `topObservation && <section>`).

- **D-13 — NO modification of the existing ExportActions component or its mount.** Per CEL-04: the existing `<ExportActions pdfOnly shimmer />` at summary/page.tsx:200 ALREADY positions the Phase 13 hero CTA at the top of the page. The CEL-04 'pinned above the fold' requirement is satisfied by VERTICAL STACKING (CTA above metrics), not by CSS `position: sticky` (which would harm scroll experience). This plan does NOT add `sticky` positioning, does NOT add a duplicate CTA, does NOT modify the shimmer or pdfOnly props.

- **D-14 — IpcMetricsBlock is a SEPARATE component from AnimatedMetric.** Rationale: (a) IpcMetricsBlock has data-fetching responsibilities (computeMetrics, volumeUnit conversions, labels) that AnimatedMetric should not know about, (b) AnimatedMetric is potentially reusable (future phases might animate other numeric reveals), (c) the file-per-component convention (per CLAUDE.md `## Conventions > Module Design`) keeps each file focused. Two small files > one big file.

- **D-15 — Number rendering uses `Math.round(target * eased)` (integer for all 4 metrics).** Reasonable for 24HV, AVV, MVV which are mL volumes (integer mL is the existing rendering convention). For NPi which IS a percentage with one decimal: the AnimatedMetric `precision={1}` prop scales the count-up internally — the hook returns `Math.round(target * eased * 10) / 10` when precision=1, so the rendered value goes 0 -> 0.5 -> 1.2 -> 2.7 -> 3.8 (settling at the final value with one decimal place). The display formatter uses `(value).toFixed(precision)` to ensure a stable decimal-point rendering even at the value 0.

- **D-16 — Boomer-safe + Performance.** No bouncy easing; only `easeOutCubic`. Total reveal time (1250ms) is at the upper edge of acceptable per CLAUDE.md memory `ux_philosophy.md` — justified because this IS the reward moment, not a routine interaction. Zero new keyframes; zero new dependencies; zero new external network requests.

    4. **Private hook `useCountUp`:**

       ```tsx
       /**
        * Count up from 0 to `target` over `durationMs`, starting after
        * `delayMs`. Uses requestAnimationFrame for smooth 60fps updates.
        * easeOutCubic easing for a smooth-deceleration feel (matches the
        * Material emphasized curve character used elsewhere in Phase 14/15).
        * Returns the current numeric value scaled to `precision` decimals.
        *
        * If `reduced` (prefers-reduced-motion) is true, returns `target`
        * immediately and skips the rAF loop — instant settled value, no
        * animation, no SR jitter.
        */
       function useCountUp(
         target: number,
         durationMs: number,
         delayMs: number,
         precision: 0 | 1,
         reduced: boolean,
       ): number {
         const [current, setCurrent] = useState<number>(reduced ? target : 0);
         const rafRef = useRef<number | null>(null);
         const startTimeoutRef = useRef<ReturnType<typeof setTimeout> | null>(null);

         useEffect(() => {
           // If reduced-motion is preferred, snap to target immediately.
           if (reduced) {
             setCurrent(target);
             return;
           }

           // Reset to 0 each time target changes (rare in practice; defensive).
           setCurrent(0);

           const startTick = () => {
             const startTs = performance.now();
             const tick = (now: number) => {
               const elapsed = now - startTs;
               const progress = Math.min(Math.max(elapsed / durationMs, 0), 1);
               // easeOutCubic: 1 - (1 - t)^3 — smooth deceleration
               const eased = 1 - Math.pow(1 - progress, 3);
               const scale = precision === 1 ? 10 : 1;
               const value = Math.round(target * eased * scale) / scale;
               setCurrent(value);
               if (progress < 1) {
                 rafRef.current = window.requestAnimationFrame(tick);
               }
             };
             rafRef.current = window.requestAnimationFrame(tick);
           };

           startTimeoutRef.current = setTimeout(startTick, delayMs);

           return () => {
             if (startTimeoutRef.current !== null) {
               clearTimeout(startTimeoutRef.current);
             }
             if (rafRef.current !== null) {
               window.cancelAnimationFrame(rafRef.current);
             }
           };
         }, [target, durationMs, delayMs, precision, reduced]);

         return current;
       }
       ```
