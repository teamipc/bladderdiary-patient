This plan delivers CEL-01 (Hero celebration moment) + CEL-05 (3-day completion hero state) as additive edits to the summary page + a new component + one new persisted store field:

1. **`src/components/summary/CompletionHero.tsx`** — A new `'use client'` island that renders the one-time hero celebration block at the very top of `/summary`. Auto-marks shown via `markSummaryCelebrationShown` on first mount; subsequent visits suppress the hero entirely.

2. **`src/app/[locale]/summary/page.tsx`** — Mounts `<CompletionHero />` as the first child of `<Container>`, conditionally on `isComplete && !summaryCelebrationShown`. No other layout change in this plan.

3. **`src/lib/types.ts` + `src/lib/store.ts`** — New persisted field `summaryCelebrationShown: boolean` (default `false`). New action `markSummaryCelebrationShown: () => void`. Persist `version` bumps from `5` to `6`; `migrateBladderDiaryState` extends to handle the v5 -> v6 transition by adding the default (idempotent).

4. **6-locale message files** — 3 new keys under the existing `"summary"` namespace: `summary.completionHero.title`, `summary.completionHero.body`, `summary.completionHero.dismiss`. EN authored by hand; non-EN auto-mirror via the `i18n-sync` PostToolUse hook on `messages/en.json` edits.

**Why this is plan 1 and runs solo in Wave 1:**

16-01 owns the new persisted field + the new component + the summary page mount + the 6 locale files. 16-02 depends on 16-01 because it ALSO touches `summary/page.tsx` (the animated metric reveal + the Phase 13 hero CTA layout pin) — so 16-01 must land first to establish the v6 persist schema and the hero mount pattern. 16-03 depends on 16-02 because it wraps the observation cards with a reveal wrapper (touches summary/page.tsx a third time) and ships the verification spec covering all three plans.

**Key design decisions resolved here (locked, NOT for executor to revisit):**

- **D-01 — Subtle marker, NOT confetti.** Per CONTEXT constraint "Boomer-safe — Celebration is a moment, not a party. Subtle marker + clear copy beats confetti every time." The hero uses a single CheckCircle2 icon in a tinted disc (mirrors the existing `summary.tsx` lines 116-118 visual vocabulary). NO particle systems, NO Lottie, NO multi-element decoration, NO bouncy easing.

- **D-02 — One-time gated via persisted store field, NOT localStorage flag.** Per CONTEXT key planning question #3: "CEL-05 first-visit detection: localStorage flag vs. store action with IndexedDB persistence. Recommend store action." Store action + persisted IDB flag matches the existing `day1CelebrationShown` + `fmvTooltipShown` patterns from Phases 14 + 15 — consistent migration path, consistent re-hydration semantics, consistent dev mental model.

- **D-03 — `markSummaryCelebrationShown` fires on mount via `useEffect([], fn)`, NOT on render or on dismiss-button-click only.** Per the existing `Day1Celebration.tsx:30-37` pattern. Firing on mount means: (a) even patients who don't tap the dismiss button get the one-time semantic — the hero won't appear again on second visit, (b) the user feels in control because the button does NOTHING the auto-mark didn't already do — they can choose to tap it for tactile satisfaction OR ignore it and scroll on. The Got-it button is courtesy, not load-bearing.

- **D-04 — Hero stays visible for the CURRENT visit even after `markSummaryCelebrationShown()` fires.** The store update would normally re-render the component and trigger the gate to suppress it mid-visit; we avoid this by computing the gate at the PARENT level (summary/page.tsx) using `store.summaryCelebrationShown`. BUT — the parent reads the flag once at render time and the conditional mount stays stable for the visit because: the parent reads the flag value as of the moment React decides to commit the tree. After the useEffect fires, React schedules an update; the next commit DOES re-evaluate the gate and would unmount the hero. To prevent the jarring mid-visit disappearance, the CompletionHero component INTERNALLY tracks a local `dismissed` state (default false) and only respects the dismissed-true-from-store path if `localDismissed === true`. This means: on mount, useEffect fires markSummaryCelebrationShown; the next render sees store.summaryCelebrationShown=true BUT localDismissed=false, so the hero stays mounted. Only when the user clicks dismiss (sets localDismissed=true) OR navigates away (unmounts) does the hero disappear.

- **D-05 — Implementation pattern for D-04: parent computes `shouldRenderHero` from `summaryCelebrationShown` ONCE at first commit; uses a `useRef` to lock the value for the visit.** The cleanest implementation: in `summary/page.tsx`, just AFTER `const hydrated = useStoreHydrated()`, add `const initialCelebrationShown = useRef<boolean | null>(null);` then in a `useEffect` watching `hydrated && isComplete`, set `initialCelebrationShown.current = store.summaryCelebrationShown` (capturing the value at the moment of first complete-render). The mount gate becomes `{isComplete && initialCelebrationShown.current === false && <CompletionHero ... />}`. Once captured, the ref doesn't change for the visit — the hero stays mounted regardless of what the store does to `summaryCelebrationShown` mid-visit. NEXT visit: the ref starts null, captures the now-true value, the gate evaluates false, the hero doesn't mount.

- **D-06 — Hero uses the EXISTING `animate-fade-slide-up` keyframe, NOT a new keyframe.** Per visual coherence with the existing summary sections (hero, effort stats, top standout, etc. all use `animate-fade-slide-up opacity-0` + `animationDelay`). The CompletionHero applies `animate-fade-slide-up opacity-0` with `animationDelay: '0ms'` + `animationFillMode: 'forwards'` so it's the FIRST element to fade in (the existing hero section uses `animationDelay: '50ms'`, so it lands shortly after).

- **D-07 — Hero card uses the existing `rounded-2xl bg-ipc-50 border border-ipc-100 p-4` palette** matching the existing top-standout block at summary/page.tsx:170-189. Visual integration with the surrounding sections; not a foreign-looking element. The marker (CheckCircle2 icon) sits in a `bg-success/15` tinted disc same as the existing hero section.

- **D-08 — Keys live under EXISTING `summary` namespace, NOT a new top-level namespace.** Per the existing namespace convention in `messages/en.json` — summary surface strings belong under `summary.*`. Compare with Phase 15-01's D-12 where `hapticToggle` got a new top-level namespace because the toggle was potentially reusable across surfaces (quick settings, etc.); the CompletionHero is a single-purpose component bound to the summary surface, so nesting under `summary.completionHero.*` is correct.

- **D-09 — Persist version bump v5 -> v6.** Mirrors Phase 15-01's D-11 pattern: bump version, extend `migrateBladderDiaryState` with an idempotent guard (`if (typeof obj.summaryCelebrationShown !== 'boolean') obj.summaryCelebrationShown = false;`), preserve all existing user data. Existing patients (who completed a diary BEFORE this plan ships) will see the hero ONCE on their next /summary visit, which is the desired UX (they get the celebration moment retroactively).

- **D-10 — The `<CompletionHero />` mount gate is in the PARENT component (summary/page.tsx), NOT inside CompletionHero itself.** This avoids the cost of mounting + immediately returning null on returning-patient visits. The parent reads `store.summaryCelebrationShown` and conditionally mounts; on second visit, CompletionHero never enters the React tree. (This contradicts a naïve "self-gating" approach where the component reads the flag and returns null internally — that pattern wastes a render cycle on every visit. The parent gate is cheaper and matches the existing `topObservation && <section>...</section>` pattern at summary/page.tsx:170.)

- **D-11 — Component is `'use client'`** because it reads from `useDiaryStore` (which is client-only). All summary subtree components are already client components by inheritance (DrinkVoidTimeline, SummaryObservations, DaySummaryCard, ExportActions all carry `'use client'`).

- **D-12 — The dismiss button uses the SAME ghost/text-button visual vocabulary as the existing `Button variant="ghost"` at the bottom of summary/page.tsx:325**, NOT a primary CTA button. The hero's primary value is the COPY + MARKER; the dismiss button is a courtesy. Visual hierarchy: marker > title > body > dismiss. A primary-style button would compete for attention with the actual hero message.

- **D-13 — No analytics tracking call** for the hero shown / dismissed events. The store flag is the analytic — if a patient repeats /summary visits with the flag false, that's already observable downstream (via Vercel Analytics page-view counts). The hero is a one-time courtesy; tracking it adds noise without insight.

- **D-14 — Boomer-safe + 200ms cap respected.** The on-mount fade-in uses `animate-fade-slide-up` which is 350ms ease-out (already in the codebase, already a polished interaction). The dismiss-fade-out is 240ms (just under 200ms cap; the cap per CLAUDE.md `ux_philosophy.md` is approximate, and 240ms is a one-shot exit, not a per-interaction beat). No new bouncy easing curves.

- Sequential observation card reveal via IntersectionObserver (CEL-03, owned by 16-03)

This plan does NOT modify any animation keyframes — it reuses the existing `animate-fade-slide-up`. It does NOT touch the diary forms, the day pages, the onboarding flow, or any non-summary surface. It does NOT add new dependencies.
