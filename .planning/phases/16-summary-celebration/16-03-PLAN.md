**Key design decisions resolved here (locked, NOT for executor to revisit):**

- **D-01 — IntersectionObserver, NOT immediate stagger.** Per CONTEXT key planning question #2: "CEL-03 observation reveal trigger: IntersectionObserver (reveals as scrolled into view) vs. immediate stagger. Recommend IntersectionObserver — more polished, doesn't waste motion on cards the patient hasn't seen." If all observation cards happen to be in the viewport at first paint (tall desktop screen, short observations list), the reveal still respects the index-ordered stagger via the `setTimeout(reveal, index * staggerMs)` — IntersectionObserver fires near-simultaneously for all visible cards, but the staggered timeouts ensure visual ordering.

- **D-02 — Wrap each `<li>`, NOT the `<ul>`.** Per the desired behavior: cards reveal SEQUENTIALLY. Wrapping the `<ul>` would reveal the whole block at once (or with no stagger between children). Wrapping each `<li>` with `index={i}` lets each child time its own reveal, which is the intent.

- **D-03 — One-shot observer disconnect after reveal.** Per CONTEXT performance constraint + the cleanest IntersectionObserver pattern. Once the wrapper has revealed, the observer is no longer needed; calling `observer.disconnect()` inside the callback frees up the browser's intersection-tracking work. No re-trigger on scroll.

- **D-04 — Reduced-motion check INSIDE the component, in ADDITION to the global CSS rule.** The global CSS rule at globals.css:124-133 sets `transition-duration: 0.01ms !important` for all elements — sufficient to collapse the transition. BUT the IntersectionObserver setup is still wasteful for reduced-motion users (they don't want the observer-driven reveal; they want instant final state). The component's internal `prefers-reduced-motion` check on mount skips the entire observer flow and renders children at final state immediately. Belt-and-suspenders: CSS handles the transition collapse, JS handles the observer-setup skip.

- **D-05 — Initial state: `opacity: 0` + `transform: translateY(8px)`.** Mirrors the existing `animate-fade-slide-up` keyframe (`opacity 0 + translateY(12px) -> opacity 1 + translateY(0)`, from globals.css:136-139). 8px instead of 12px to slightly differentiate the per-card reveal from the section's overall fade-slide-up entrance — the SECTION fades up first, then individual CARDS fade up afterwards with a smaller distance. Visual ordering: section > cards > text within cards.

- **D-06 — 350ms transition duration, `cubic-bezier(0.4, 0, 0.2, 1)` curve.** Consistent with the Phase 14/15/16-02 motion vocabulary. 350ms is at the upper acceptable boundary for an interaction-adjacent animation — justified because each card's reveal is a one-shot beat the patient sees ONCE per visit, not a repeated interaction.

- **D-07 — `staggerMs: 150` default.** Per CONTEXT constraint "Sequential observation cards animate-in one at a time (~150ms stagger)". With ~3-5 observation cards typically rendered, the total reveal time is 0 + (4 × 150) + 350 = 950ms — under 1 second, acceptable per Boomer-safe.

- **D-08 — SSR safety via `typeof window === 'undefined'` check.** The static-export build pre-renders components at build time; `window` and `IntersectionObserver` are undefined. The component must not crash. The pattern: check `typeof window === 'undefined'` at the top of the useEffect; if true, return early; the SSR HTML renders children at the INITIAL (hidden) state, then on client-hydration the useEffect fires and the reveal begins. To avoid a flash of invisible content on hydration, the component renders children at the VISIBLE state during SSR (opacity 1) and then on client-mount switches to invisible (opacity 0) + scheduled reveal. Two-state hydration: SSR = visible; client = invisible-then-reveal. The cleanest implementation: use a `mounted` state defaulted false; render children at final state when `!mounted`; switch to invisible + observer setup when `mounted` flips true via useEffect. On client mount, useEffect flips mounted=true, then the observer setup begins. SSR users (e.g., crawlers, RSS-style readers) see the final content; client users see the staged reveal.

- **D-09 — Wrap the `<li>` content INSIDE the `<li>`, NOT the `<li>` itself.** The semantic structure is preserved: SummaryObservations.tsx renders `<ul><li>{text}</li><li>{text}</li>...</ul>`. The new wrapper sits inside the `<li>` around the text content: `<ul><li><ObservationCardReveal index={0}>{text}</ObservationCardReveal></li>...</ul>`. Why: wrapping the `<li>` from the OUTSIDE would inject a `<div>` between `<ul>` and `<li>` — invalid HTML (a `<ul>`'s direct child must be a `<li>` or `<li>`-styled element). Wrapping the CONTENT preserves `<ul>` > `<li>` parentage; the reveal `<div>` lives INSIDE the `<li>`. Per the existing SummaryObservations.tsx:38 markup `<li ...>{keyToCopy(o, t)}</li>`, the wrap is around `{keyToCopy(o, t)}`.

- **D-10 — Spec mirrors Phase 14/15 pattern verbatim.** Same imports (`@playwright/test`, `e2e/helpers/fixtures`), same `clearDiary` / `seedDiary` helpers, same `localePath` helper, same screenshot directory layout, same `test.describe` block per CEL requirement.
