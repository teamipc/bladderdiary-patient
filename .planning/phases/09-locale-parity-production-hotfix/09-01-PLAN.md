# Plan 09-01: Fix ArticleCard locale-stripping regex to cover all 6 locales (driven from `i18n/config.ts`)

<objective>
LP-01 (UI-REVIEW finding I1, Critical) — `src/components/learn/ArticleCard.tsx:36` calls `article.urlPath.replace(/^\/(en|fr|es)/, '')`. The regex strips only 3 of 6 locale prefixes. For `pt`/`zh`/`ar` articles the prefix survives, and `<Link>` from `@/i18n/navigation` then re-prepends the current locale, producing `/pt/pt/learn/...` URLs that 404 in production. **Verified live by audit on 2026-05-18:** `curl https://myflowcheck.com/pt/learn/voiding/feeling-bladder-is-not-empty` → 404.

This plan replaces the hardcoded `en|fr|es` regex with one driven from the `locales` tuple in `src/i18n/config.ts`, so the bug can never silently regress when a new locale is added later. **Approach (a) from CONTEXT § "Key planning questions" — fix the regex, not the source `urlPath`.**

**Rationale for choosing (a) over (b):** `article.urlPath` is used in 19 places across the codebase. Only `ArticleCard.tsx:36` is inside a `<Link>` from `@/i18n/navigation` (which re-prepends the locale). The other 18 callers (sitemap, BreadcrumbJsonLd, hreflang map, `buildAbsoluteUrl`, `feed.xml`, JSON-LD `Article.url`, sibling article `ArticleCard` `key={a.urlPath}` props) all need the locale-prefixed canonical form. Refactoring `content.ts:buildUrlPath` to strip the prefix at source would silently break 18 SEO surfaces. Per-call stripping in the one offending site, driven from a single source of truth, is the surgical fix.

**Pre-compile a constant regex at module scope** rather than constructing a new `RegExp` per render — `ArticleCard` is rendered ~12 times on `/learn` and ~6+ times on every article page; allocating a regex on every render is wasteful and the locale list is static at build time.

UI-REVIEW.md §I1 + §"Top wins by impact/effort #1" identify this as the highest impact ÷ effort fix in Milestone 3 (literally 1-line regex change restores half the locale base). This plan also adds a vitest unit test that asserts the constructed regex covers all 6 locales — preventing the same hardcoded-subset drift from creeping back in.

Open questions addressed:
- Q2 from CONTEXT (urlPath refactor scope) — RESOLVED in favor of (a). Documented in this plan.
- Q3 from CONTEXT (plan splitting) — LP-01 lives in Wave 1 as a same-day hotfix.
</objective>

  <action>
    Make a minimal, surgical edit to `src/components/learn/ArticleCard.tsx`:

    1. Add an import for `locales` from `@/i18n/config` to the existing `@/i18n/config` import line (currently only `type Locale` is imported there):
       ```
       import { locales, type Locale } from '@/i18n/config';
       ```
       Use the Edit tool — DO NOT introduce a separate import statement; merge into the existing import.

    2. ABOVE the `export default async function ArticleCard(...)` declaration (after the imports block, before the function), add a module-scope constant:
       ```
       // Strip ANY supported locale prefix from urlPath so <Link> from @/i18n/navigation
       // can re-prepend the current locale without producing /pt/pt/... double-prefixed URLs.
       // Driven from i18n/config so this can never silently drift when a locale is added.
       const LOCALE_PREFIX_RE = new RegExp(`^/(${locales.join('|')})(?=/|$)`);
       ```
       The `(?=/|$)` non-capturing lookahead is REQUIRED — without it, a future locale like `enabled-locale-foo` would be matched by a lookalike prefix; with it, only a true path segment boundary terminates the match. Also defends against a hypothetical slug literally starting with a locale code (e.g. a topic folder named `en-something`).

    3. Replace the existing line:
       ```
       href={article.urlPath.replace(/^\/(en|fr|es)/, '')}
       ```
       with:
       ```
       href={article.urlPath.replace(LOCALE_PREFIX_RE, '')}
       ```

    4. Do NOT change anything else in the file. No formatting changes, no other refactors. The blast radius of this commit must be exactly two lines of meaningful change (the import edit + the regex constant) plus the one-line `href` swap.

    Constraints per CLAUDE.md + project conventions:
    - No emoji, no em-dashes in comments
    - `const` not `let`
    - No `as any` / `@ts-ignore`
    - Import order respects the existing convention in this file (framework → third-party → @/components → @/lib)

  <files>src/__tests__/article-card-locale-routing.test.tsx</files>

  <action>
    Create `src/__tests__/article-card-locale-routing.test.tsx`. This test verifies the locale-stripping logic without needing to render the full `ArticleCard` (which is an async server component that depends on `next-intl/server`'s `getLocale`, awkward to mount in jsdom). Instead, the test re-derives the same regex from `locales` and asserts behavior across all 6 prefixes plus negative cases.

    Structure:

    ```ts
    import { describe, it, expect } from 'vitest';
    import { locales } from '@/i18n/config';

    // This regex MUST be kept identical to the one in src/components/learn/ArticleCard.tsx.
    // The test re-derives it from the same `locales` source to catch any drift between
    // the component's regex shape and the contract this test enforces.
    const LOCALE_PREFIX_RE = new RegExp(`^/(${locales.join('|')})(?=/|$)`);

    describe('ArticleCard locale-stripping regex (LP-01)', () => {
      // ... cases below
    });
    ```

    Cases (each its own `it()` block with descriptive English title):

    1. `it('strips each of the 6 supported locale prefixes')` — for each locale `L` in `locales`, assert:
       `'/' + L + '/learn/voiding/feeling-bladder-is-not-empty'.replace(LOCALE_PREFIX_RE, '') === '/learn/voiding/feeling-bladder-is-not-empty'`.
       Loop over `locales` so adding a new locale auto-extends coverage.

    2. `it('strips the locale prefix on a pillar path with no trailing segments')` — for each locale, assert:
       `'/' + L + '/learn/voiding'.replace(LOCALE_PREFIX_RE, '') === '/learn/voiding'`.

    3. `it('strips the locale prefix on a bare-locale-root path')` — for each locale, assert:
       `'/' + L`.replace(LOCALE_PREFIX_RE, '') === ''.
       (The lookahead `(?=/|$)` allows end-of-string as the boundary, so `/pt` strips to `''`. Not used in production for article cards but documents the contract.)

    4. `it('does NOT strip non-locale path segments that resemble a locale')` — assert these three negative cases:
       - `'/learn/voiding/...'.replace(LOCALE_PREFIX_RE, '')` === `'/learn/voiding/...'` (no leading locale → unchanged)
       - `'/english-learning/article'.replace(LOCALE_PREFIX_RE, '')` === `'/english-learning/article'` (`en` is at the head but followed by `glish-` not `/` or end → lookahead fails → unchanged)
       - `'/arctic-research'.replace(LOCALE_PREFIX_RE, '')` === `'/arctic-research'` (`ar` followed by `ctic` → unchanged)

    5. `it('keeps the resulting path as a valid Link href that next-intl will re-prepend the locale to')` — for each locale, assert that the stripped path starts with `'/'` (so `<Link href={...}>` with `localePrefix: 'as-needed'` resolves to `/<locale>/<rest>` for non-default locales and `/<rest>` for `en`).

    6. `it('protects against the live production bug: PT/ZH/AR cards no longer produce /<L>/<L>/learn/... URLs')` — explicit case from UI-REVIEW.md §I1:
       - Input: `/pt/learn/voiding/feeling-bladder-is-not-empty`
       - After regex strip: `/learn/voiding/feeling-bladder-is-not-empty`
       - When `<Link>` re-prepends `pt`, final URL: `/pt/learn/voiding/feeling-bladder-is-not-empty` (NOT `/pt/pt/learn/...`)
       Simulate the `<Link>` re-prepend by manually concatenating `'/' + L + stripped` and asserting the result has exactly one `/<L>/` segment (use a regex like `new RegExp('^/' + L + '/').exec(rebuilt)` returns one match; `new RegExp('^/' + L + '/' + L + '/').exec(rebuilt)` returns null).

    Imports:
    ```
    import { describe, it, expect } from 'vitest';
    import { locales } from '@/i18n/config';
    ```

    NO snapshot tests. NO emoji. NO `it.skip`.
  </action>
