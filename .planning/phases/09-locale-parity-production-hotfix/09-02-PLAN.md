# Plan 09-02: Localize TimePicker bedtime preset chips (LP-04) + Breadcrumb landmark aria-label (LP-05)

<objective>
Two tiny string-leak fixes that share one constraint: both write into `messages/en.json` and rely on the `i18n-sync` PostToolUse hook to auto-mirror new keys to the other 5 locale files. Bundling them prevents two simultaneous hook fires racing against each other, and lets a single i18n verification step at the end of the plan cover both keys.

**LP-04 — TimePicker bedtime preset chips (UI-REVIEW.md §C1, High severity, audit's #4 highest-impact fix):**
`src/components/ui/TimePicker.tsx:159,166,173` hardcodes `'10 PM'` / `'11 PM'` / `'12 AM'` as English string literals injected into the translated wrapper `lastNightAt` (`{time} last night` / `{time} hier soir` / `昨晚 {time}` / `{time} ليلة أمس`). FR users see "10 PM hier soir" (English-PM inside French); AR users see "Latin PM" embedded in a right-to-left line — a literal LTR-script fragment violating bidi flow.

The fix: replace each hardcoded literal with a call to `formatTime()` from `src/lib/utils.ts:239` (the canonical locale-aware time formatter, used everywhere else in the app). Specifically: compute the chip's reference ISO timestamp via `buildIsoForClockTimeInTz(value, hour24, 0, timeZone)`, then format it via `formatTime(iso, locale, timeZone)`. This is the exact pattern UI-REVIEW.md §C1 recommends and the same pattern other parts of TimePicker.tsx already use elsewhere (e.g. `getClockTimeInTz` for the input's displayed time).

**LP-05 — Breadcrumb landmark aria-label (UI-REVIEW.md §I6, Low severity but in scope for Phase 9):**
`src/components/learn/Breadcrumbs.tsx:11` hardcodes `aria-label="Breadcrumb"` in English. Adds `learn.breadcrumbs.ariaLabel` to `messages/en.json` (already a namespace — `learn.breadcrumbs.home`/`learn`/`glossary`/`authors` exist), wires Breadcrumbs to use `useTranslations('learn.breadcrumbs')` for the landmark label, and lets `i18n-sync` mirror to the 5 non-en locale files. Reuses the established pattern from `src/components/layout/Footer.tsx:33` (`t('navAriaLabel')`).

**Why bundle these two requirements:**
- Both add a single new i18n key under an EXISTING namespace
- Both touch `messages/en.json` which fires the `i18n-sync` PostToolUse hook
- Running them as one atomic plan means one hook fire, one verification step covering both keys, one commit
- Neither touches the other's code surface — risk of merge conflict between the tasks is zero

**Order of operations is deliberate** (project memory `project_translation_workflow.md`): the Stop hook + pre-commit hook block partially-translated state. The plan adds BOTH new EN keys first (one Edit call), lets `i18n-sync` fire once for both, verifies parity across all 6 locales, THEN edits the two components to consume the new keys. If the hook missed a locale, Task 2 backfills manually with the documented translations BEFORE the component edit, so an executor never lands in a state where Breadcrumbs references a key not present in `messages/ar.json`.

Open questions addressed:
- Q3 (plan splitting / Wave 1 hotfix bundle) — RESOLVED: LP-04 + LP-05 ship in Wave 1 alongside LP-01 (09-01) as same-day fixes.
- Q5 (i18n-sync interaction) — RESOLVED: single atomic en.json edit → single hook fire → single verification step → manual fallback documented per locale.
