# Plan 09-05: PDF Unicode font registration + ZH/AR locale tables (LP-02 full)

<objective>
LP-02 (UI-REVIEW.md §I2, Critical) — the clinical PDF export renders strings via `getPdfStrings(locale)` from the table extended in plan 09-03, BUT every page calls `doc.setFont('helvetica', ...)`. Helvetica has zero glyph coverage for CJK (Chinese, Japanese, Korean) or Arabic. Even if 09-03 extends the translation table with ZH/AR entries, the rendered PDF would show boxes where CJK or Arabic characters belong. A Mandarin-speaking PFPT receiving a patient's PDF cannot read the section headers.

This plan finishes LP-02 by:
1. **Building a font subsetting pipeline** at `scripts/build-pdf-font-subsets.mjs` that takes a Noto Sans SC TTF and a Noto Sans Arabic TTF, subsets each to ONLY the glyphs the PDF strings table actually emits (the per-locale subset script reads `strings.ts` PT_STRINGS/ZH_STRINGS/AR_STRINGS values + digit/punctuation/whitespace ranges + the small fixed `0-9`/`:`/`/`/`,`/`.`/` ` character class for numeric labels), and emits two base64-encoded `.ts` modules under `src/lib/exportPdf/fonts/`.
2. **Adding ZH and AR locale entries** to the `PDF_STRINGS` table in `strings.ts` (the table extended by 09-03; this plan adds the missing two locales).
3. **Lazy-loading the font module** via dynamic `import()` inside `generatePdfBlob`, so EN/FR/ES/PT generate PDFs without paying the bundle cost. The `generatePdfBlob` signature becomes async (`Promise<{ blob; filename }>`) and the single caller `ExportActions.tsx:handlePdf` already awaits its result, so signature ripple is contained.
4. **Centralizing the font-family selection** via a helper `currentFontFamily(locale)` exported from `src/lib/exportPdf/fonts/index.ts`. Every existing `doc.setFont('helvetica', style)` call in the 5 PDF page modules is replaced with `doc.setFont(currentFontFamily(locale), style)`. For EN/FR/ES/PT this resolves to `'helvetica'` (no behavior change); for ZH this resolves to `'NotoSansSC'`; for AR this resolves to `'NotoSansArabic'`.
5. **Setting the document text direction for AR** via `doc.setR2L(true)` per jsPDF API, applied as part of the font-registration call for AR locale.

**Font sourcing decision (locked in this plan, NOT deferred to checkpoint):**

Following CONTEXT.md §LP-02 evidence:
- **Option (b) "subset + lazy-load" is chosen.** Confirmed: jsPDF's `addFileToVFS(filename, base64string)` + `addFont(filename, fontname, style)` API accepts a base64-encoded TTF at any point before the relevant `setFont` call. The fonts can be registered mid-document via dynamic import.
- **Tooling: `subset-font` npm package** (pure-JS, no native binding) for build-time subsetting. Added to `devDependencies` (NOT a runtime dependency — the subsetting runs at build time in `scripts/`; the output is a checked-in base64 module).
- **Source font: Google Noto Sans family** (SIL OFL 1.1 license — license-clean for redistribution). Specifically:
  - ZH: **Noto Sans SC Regular + Bold** (downloaded from `https://fonts.google.com/noto/specimen/Noto+Sans+SC` or via `@fontsource/noto-sans-sc` npm package).
  - AR: **Noto Sans Arabic Regular + Bold** (downloaded similarly).
- **Subset size budget:** ~150-400KB per locale per weight after subsetting to only the glyphs in the strings table + digit/punctuation/whitespace. Total static-asset cost: ~600KB-1.5MB for both locales × both weights, lazy-loaded. PDF file size impact: only the per-locale font is embedded in the PDF blob (jsPDF VFS), so a ZH PDF carries ~250-500KB of ZH font, well under the 5MB PDF size budget from CONTEXT.md.

**Bundle-cost protection:** the EN/FR/ES/PT bundle stays the same — `import()` is a dynamic boundary, Next.js code-splits the font module out of the main bundle. The ZH/AR font modules are ONLY loaded on `generatePdfBlob(state, 'zh')` or `generatePdfBlob(state, 'ar')` calls. Static-export PWA shipped to Vercel does not include the font bytes in the initial app shell.

**Why this plan depends on 09-03:** 09-03 extends the `PdfStrings` interface with `structuredDataTitle` / `eventsTitle` / `structuredDataSubtitle` AND adds the PT locale entry. This plan inherits the extended interface (so the ZH and AR entries it adds are structurally complete in one shot) and inherits the locale-neutral 24hr axis labels (so this plan does not have to think about per-locale time-axis formatting).

**Why this plan is in Wave 3, not Wave 2:** 09-04 (author photos) runs Wave 2 parallel to 09-03 (PDF strings). 09-05 lands in Wave 3 because (a) it depends on 09-03's extended interface, (b) it is the largest single piece of work in Phase 9, (c) putting it in its own wave keeps the parallelism manageable. 09-05 has zero file overlap with 09-04 — author photo work and PDF font work touch entirely different surfaces.

Open questions addressed:
- Q1 (font strategy decision) — LOCKED to subset + lazy-load using `subset-font` package + Noto Sans family.
- Q4 (test strategy for LP-02) — vitest test (`pdf-font-registry.test.ts`) asserts (a) the font registry returns the correct family for each locale, (b) the ZH/AR font modules export the expected `registerFont` function shape, (c) base64 length is in the expected range (catches build-time subsetting failures), (d) PT remains helvetica (no font registration triggered).
- Q5 (i18n-sync collisions) — N/A; this plan adds NO i18n keys (the PDF strings table is its own private string store, not next-intl).
- Q6 (PDF regression testing) — partial: this plan adds a unit test that the font registry is wired correctly; full PDF-blob assertion (does the generated blob contain the registered font?) is plan 09-06's job and uses `pdf-parse` (already in devDependencies) to introspect the rendered PDF.

    /**
     * Build subsetted Noto fonts for the clinical PDF export's ZH and AR locales.
     *
     * Input: full Noto Sans SC + Noto Sans Arabic TTFs from @fontsource packages.
     * Output: src/lib/exportPdf/fonts/zh.ts + ar.ts with base64-encoded subsetted TTFs.
     *
     * Re-run this whenever the PDF strings table gains new characters in ZH or AR.
     * Run: node scripts/build-pdf-font-subsets.mjs
     */
    import { readFile, writeFile } from 'node:fs/promises';
    import { resolve } from 'node:path';
    import subsetFont from 'subset-font';

    // The character superset MUST cover all glyphs the PDF will render in this locale.
    // Update these strings if new keys are added to the strings.ts table that
    // introduce characters not already in the subset. CI test pdf-font-registry.test.ts
    // exercises a representative sample and will fail if any glyph is missing.

    const COMMON = '0123456789:/.,-· |()% \n\t';

    const ZH_GLYPHS = COMMON + (
      // Every value from PDF_STRINGS.zh in strings.ts — flatten all the keys and
      // labels into one string. Maintain this list in sync with strings.ts.
      // The script imports strings.ts dynamically below to avoid duplication.
      ''
    );

    const AR_GLYPHS = COMMON + (
      ''
    );

    async function loadStringsCoverage() {
      // Dynamic import the strings module (using a tsx loader in dev or by reading it raw).
      // Since strings.ts is a TS file, we read it as text and extract all string literals
      // assigned to keys inside the PDF_STRINGS.zh and PDF_STRINGS.ar blocks.
      const txt = await readFile(resolve('src/lib/exportPdf/strings.ts'), 'utf8');
      const extractBlock = (locale) => {
        const start = txt.indexOf(`  ${locale}: {`);
        if (start < 0) return '';
        let depth = 0;
        let end = start;
        for (let i = start; i < txt.length; i++) {
          if (txt[i] === '{') depth++;
          if (txt[i] === '}') {
            depth--;
            if (depth === 0) { end = i + 1; break; }
          }
        }
        return txt.slice(start, end);
      };
      const extractStrings = (block) => {
        // Match both single-quoted and double-quoted string literals; also template literals
        const re = /(?:'([^'\\]*(?:\\.[^'\\]*)*)'|"([^"\\]*(?:\\.[^"\\]*)*)"|`([^`]*?)`)/g;
        let out = '';
        let m;
        while ((m = re.exec(block)) !== null) {
          out += (m[1] || m[2] || m[3] || '');
        }
        return out;
      };
      return {
        zh: extractStrings(extractBlock('zh')),
        ar: extractStrings(extractBlock('ar')),
      };
    }

    async function buildSubset(sourceFontPath, glyphs, outPath, fontVarName, jspdfName) {
      const buf = await readFile(sourceFontPath);
      const subset = await subsetFont(buf, glyphs, { targetFormat: 'truetype' });
      const base64 = subset.toString('base64');
      const moduleSrc = `// AUTO-GENERATED by scripts/build-pdf-font-subsets.mjs. Do NOT hand-edit.\n// Source: ${sourceFontPath}\n// Subsetted glyphs: ${glyphs.length} unique chars\n// Output size: ${(subset.length / 1024).toFixed(1)} KB\n\nexport const ${fontVarName}_BASE64 = '${base64}';\nexport const ${fontVarName}_JSPDF_NAME = '${jspdfName}';\n`;
      await writeFile(outPath, moduleSrc, 'utf8');
      console.log(`Wrote ${outPath} (${(subset.length / 1024).toFixed(1)} KB)`);
    }

      const cov = await loadStringsCoverage();
      const zhGlyphs = [...new Set(COMMON + cov.zh)].join('');
      const arGlyphs = [...new Set(COMMON + cov.ar)].join('');

    **Step C — Write `src/lib/exportPdf/fonts/index.ts`** as the public registry:

    ```ts
    // Public registry of Unicode fonts for the clinical PDF.
    //
    // Locales using helvetica (Latin script): en, fr, es, pt.
    // Locales requiring Unicode fonts via lazy load:
    //   - zh: Noto Sans SC (Simplified Chinese)
    //   - ar: Noto Sans Arabic (Arabic script, RTL)
    //
    // EN/FR/ES/PT pay zero bundle cost — the per-locale modules are only loaded
    // when generatePdfBlob() is invoked with that locale.

    import type { jsPDF } from 'jspdf';

    export type PdfFontFamily = 'helvetica' | 'NotoSansSC' | 'NotoSansArabic';

    export function currentFontFamily(locale: string): PdfFontFamily {
      if (locale === 'zh') return 'NotoSansSC';
      if (locale === 'ar') return 'NotoSansArabic';
      return 'helvetica';
    }

    /**
     * Register the locale-specific Unicode font with the jsPDF document.
     * No-op for locales that use the default helvetica font.
     *
     * Lazy-loads the font's base64 module via dynamic import so EN/FR/ES/PT
     * generation does not bring CJK/Arabic font bytes into the bundle.
     *
     * For AR, also enables right-to-left rendering on the document via doc.setR2L(true).
     */
    export async function ensureLocaleFontRegistered(doc: jsPDF, locale: string): Promise<void> {
      if (locale === 'zh') {
        const { registerZhFont } = await import('./zh');
        registerZhFont(doc);
      } else if (locale === 'ar') {
        const { registerArFont } = await import('./ar');
        registerArFont(doc);
      }
      // EN/FR/ES/PT: helvetica is built into jsPDF; nothing to register.
    }
    ```
