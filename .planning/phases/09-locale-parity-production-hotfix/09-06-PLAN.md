# Plan 09-06: Phase 9 cross-locale verification spec — Playwright + PDF-blob introspection + human-verify checkpoint

<objective>
Verify the 6 Phase 9 success criteria (LP-01..LP-06) hold against the actual built site across the 6-locale matrix. This plan is the canonical sign-off for Phase 9. It does NOT modify any source code — it only adds a new Playwright spec + a vitest test that exercises a real PDF blob + a human-verify checkpoint that pauses for the project owner to eyeball the locale-rendered PDF outputs.

**Three verification surfaces, three test artifacts:**

1. **Playwright spec `e2e/phase9-locale-parity.spec.ts`** — runs against a LOCAL static-export build (`out/`) served by `npx serve out -l 4173`, exercises LP-01 (article-card 404 regression on PT/ZH/AR), LP-04 (TimePicker bedtime chip render in 6 locales), LP-05 (Breadcrumb aria-label in 6 locales), LP-06 (author photo render on `/learn/authors/*` in 6 locales). Follows the established Phase 5 pattern (`e2e/phase5-chrome.spec.ts`).
2. **Vitest test `src/__tests__/pdf-blob-content.test.ts`** — generates a 3-day-diary PDF blob in each of the 6 locales using `generatePdfBlob` + a synthetic `DiaryState`, then uses `pdf-parse` (already in devDependencies — confirmed in CONTEXT.md tech stack) to extract text content. Asserts (a) localized strings are present, (b) hardcoded English literals are NOT present except on the machine-data page, (c) the file size is under the 5MB budget, (d) per-locale font registration produced a valid PDF (the blob is parseable without errors).
3. **Human-verify checkpoint** — the project owner runs `generatePdfBlob` in each locale via the existing `scripts/generate-test-exports.test.ts` (already in the repo per CONTEXT.md), opens each PDF in a viewer, and confirms (a) CJK / Arabic glyphs render (no tofu boxes), (b) AR text flows right-to-left, (c) author photos load on `/learn/authors/*` across locales, (d) the daily-walkthrough remains green. Unicode glyph correctness CANNOT be fully automated — even with `pdf-parse` text extraction, font-rendering bugs (missing-glyph boxes, wrong baselines, RTL bidi issues) only surface in a real PDF viewer. The human checkpoint is the medical-grade quality gate.

**Why wave 4 + depends on all 5 prior plans:** every LP-* fix lands first. This spec then verifies the integrated behavior. If any prior plan introduced a regression on another LP, this spec catches it.

    import { describe, it, expect } from 'vitest';
    import { generatePdfBlob } from '@/lib/exportPdf';
    import { getPdfStrings } from '@/lib/exportPdf/strings';
    import { locales } from '@/i18n/config';
    import type { DiaryState } from '@/lib/types';

    // pdf-parse v2 exposes a class-based API: `new PDFParse({data}).getText()`.
    // Mirror the adapter from e2e/deep-flow.spec.ts:30-47 so the API stays consistent across the repo.
    // eslint-disable-next-line @typescript-eslint/no-require-imports
    const { PDFParse } = require('pdf-parse');
    async function extractPdfText(buf: Buffer): Promise<{ text: string; numpages: number }> {
      const inst = new PDFParse({ data: new Uint8Array(buf) });
      try {
        const info = await inst.getInfo();
        const result = await inst.getText();
        const text: string = result.text ?? (Array.isArray(result.pages) ? result.pages.map((p: { text?: string }) => p.text ?? '').join('\n') : '');
        const numpages: number = info?.numPages ?? (Array.isArray(result.pages) ? result.pages.length : 0);
        return { text, numpages };
      } finally {
        await inst.destroy().catch(() => {});
      }
    }
