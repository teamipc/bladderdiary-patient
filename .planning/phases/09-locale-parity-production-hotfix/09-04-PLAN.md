# Plan 09-04: Source + wire author profile photos (LP-06)

<objective>
LP-06 — both author JSON files (`content/authors/dr-di-wu.json`, `content/authors/dr-steven-tijerina.json`) have empty/absent `photoUrl` fields; `public/authors/` directory does not exist; author profile pages render without photos; `Author` JSON-LD `image`/`photoUrl` field is empty. For a YMYL (Your Money or Your Life) medical site, real photos are a documented requirement for E-E-A-T signal (Experience, Expertise, Authoritativeness, Trust) — Google's medical-content quality guidelines treat author identifiability as a primary signal.

**This plan has two parts that share no code surface with the LP-01..LP-05 work:**
1. **Asset sourcing** — get real photos for both IPC clinicians. The `image-source` skill (already installed at `.claude/skills/image-source/SKILL.md`) is the standard tool for this in the repo. **HOWEVER** the skill is designed for stock-photography sourcing (Unsplash), which is NOT appropriate for real-clinician profile photos. The fix: this plan declares a **checkpoint** for the user to provide actual photos OR explicitly authorize a documented fallback path.
2. **Wire-up** — once assets exist on disk, populate `photoUrl` in the JSON, render `<img>` with translated alt text on the author page, and add `image` to the `Person` JSON-LD.

**Why this is `autonomous: false`:** Real clinician portraits cannot be sourced by Claude — they require the human (project owner) to either (a) supply photos they have rights to publish or (b) explicitly authorize a documented stand-in (e.g. an IPC-branded silhouette placeholder, a Gravatar pull from the LinkedIn URL already in `dr-di-wu.json`/`dr-steven-tijerina.json`, or initials on a colored circle). The Wave-2 checkpoint pauses execution for that decision before Tasks 2-5 run.

**Why this runs in Wave 2 in parallel with 09-03 (PDF strings):** zero code-surface overlap with the PDF work. The `files_modified` for 09-03 is `src/lib/exportPdf/**` + 1 test file; the `files_modified` for this plan is `public/authors/**` + `content/authors/**` + `src/app/[locale]/learn/authors/[slug]/page.tsx` + `messages/*.json` + `src/components/seo/JsonLd.tsx`. No file appears in both. Wave-2 parallelism is safe.

**i18n collision risk with 09-02 (LP-04+LP-05 hotfix bundle):** 09-02 is Wave 1 and adds `learn.breadcrumbs.ariaLabel` to `messages/en.json`. This plan is Wave 2 and adds `learn.author.photoAlt` to `messages/en.json`. Because they are in different waves AND both keys live under `learn.*`, there is no overlap; 09-02 must complete (PostToolUse hook fires, ariaLabel mirrored) BEFORE this plan touches en.json a second time. Wave dependencies enforce this ordering already.

Open questions addressed:
- Q3 (plan splitting; author work in parallel with PDF) — RESOLVED: 09-04 = author photos, runs Wave 2 parallel to 09-03.
- Q5 (i18n-sync collisions) — RESOLVED: 09-04 fires the hook AFTER 09-02's hook has completed (Wave 1 → Wave 2 dependency).

Open questions DEFERRED to checkpoint:
- C1: Photo sourcing strategy (real photos vs documented placeholder vs Gravatar-from-LinkedIn). The checkpoint surfaces these three options to the user.
</objective>
