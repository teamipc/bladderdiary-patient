# Plan 09-03: Eliminate hardcoded English strings in PDFs across all locales + add PT to PDF_STRINGS table

<objective>
LP-03 (UI-REVIEW.md §I3 + §I2 partial) — PDFs generated in **EN/FR/ES** currently contain several hardcoded English strings that bypass the `PDF_STRINGS` table entirely. Even for the three locales whose translation tables exist, the PDF is half-translated. Specific leaks from CONTEXT.md §LP-03 evidence:

- `src/lib/exportPdf/dailyDiary.ts:55` — `head: [['Time', …]]` (hardcoded `'Time'`)
- `src/lib/exportPdf/slots.ts:44` and `slots.ts:142` — `'AM'` / `'PM'` labels in `buildHourSlots` and `buildHalfHourSlots` (`hour === 0 ? '12 AM' : hour < 12 ? \`${hour} AM\`` …)
- `src/lib/exportPdf/machineData.ts:17` — `doc.text('Structured Data', …)` (page title)
- `src/lib/exportPdf/machineData.ts:21` — `'For clinical software ingestion. This page may be scanned or parsed electronically.'`
- `src/lib/exportPdf/machineData.ts:55` — `head: [['Field', 'Value']]`
- `src/lib/exportPdf/machineData.ts:70` — `doc.text('Events', …)`
- `src/lib/exportPdf/graphs.ts:194` — `const timeLabels = ['6am', '8am', '10am', '12pm', '2pm', '4pm', '6pm', '8pm', '10pm', '12am', '2am', '4am'];` (hardcoded English lowercase, used across all locales)
- `src/lib/exportPdf/combinedDiary.ts:45` — `const ampm = hour === 0 ? '12 AM' : …` (same AM/PM construction as slots.ts)

**Decision on `machineData.ts`:** the audit (UI-REVIEW.md §I2) and the existing `index.ts:44` comment (`// Page 7: Machine-readable data (always English for clinical software)`) explicitly mark this page as English-only by design. The two human-readable header strings (`'Structured Data'`, `'Events'`) are the only items in `machineData.ts` that get localized in this plan — the page sub-title (`'For clinical software ingestion...'`), table column heads (`'Field'`, `'Value'`, `'type'`, `'timestamp'`, …), and field NAMES (`'patient_age'`, `'start_date'`, …) STAY English because they are machine-parsing schema, not human prose. UI-REVIEW.md §I2 endorses this split.

**Decision on time-axis labels (`graphs.ts:194`):** UI-REVIEW.md §I3 recommends moving to 24hr-format labels (`['06:00', '08:00', ...]`) because (a) 24hr is the standard medical convention in EU/clinical contexts, (b) it reads cleanly in every locale without per-locale logic, (c) it is locale-neutral and matches the convention already used inside the daily-grid `hourStr` field (`'06:00'`, `'06:30'`, …) elsewhere in `slots.ts`. **This plan adopts the 24hr-label fix.** Patients in EN-locale already see 24hr clock labels on the SAME page's daily-diary grid (the `hourStr` field in `slots.ts:43,139` reads `'06:00'`, NOT `'6 AM'`), so 24hr on the scatter-plot axis is consistent rather than discordant.

**Decision on AM/PM labels inside slot grid labels (`slots.ts:44, 142` + `combinedDiary.ts:45`):** these are the secondary line under the 24hr time in each row label. The current EN form is `'06:00\n6 AM'`. For locale-correct rendering across FR/ES (and the upcoming PT/ZH/AR in 09-05), this plan removes the AM/PM secondary line entirely — the 24hr line `'06:00'` is unambiguous, locale-neutral, and already in the same cell. Rationale: (1) 24hr clock is universal medical convention, (2) no locale-specific AM/PM equivalent needs invented, (3) reduces row label height in the PDF table (a benefit for the dense 48-row half-hour grid), (4) consistent with the new graph-axis 24hr decision above.

**Why this is Wave 2 and depends on nothing:** This plan extends `PDF_STRINGS` interface + EN/FR/ES tables AND removes the hardcoded English literals from the 5 PDF-page modules. It does NOT add new locales (PT/ZH/AR translation tables — that is plan 09-05's job; PT additions are bundled here because PT is Latin-script and works with helvetica without font registration). Plan 09-05 will EXTEND the table further with ZH/AR translations on top of this plan's keys; 09-05 also adds PT-specific values when the schema is fully locked.

Wait — re-read CONTEXT carefully: "(1) Add PT entries to PDF_STRINGS (Helvetica handles Latin-1 ext OK, Portuguese is feasible without font change). ... (2) For ZH and AR: register a Unicode-capable font with jsPDF". Per the audit, PT is grouped with the font work conceptually but is technically separable (helvetica-safe). This plan ships the PT table alongside EN/FR/ES extensions so 09-05 only needs to add ZH/AR translations + font registration. That gives the cleanest dependency: 09-03 = "everything that does not need a Unicode font"; 09-05 = "everything that needs Unicode fonts".
