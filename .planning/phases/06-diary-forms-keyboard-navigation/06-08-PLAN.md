<objective>
Per 06-UI-SPEC §"Form-by-Form Specs" → "4. `SetBedtimeForm` (122 lines)" + "5. `SetWakeTimeForm` (111 lines)" (locked), these are the TWO smallest forms in scope: single-step, slider-only-via-TimePicker, no preset chips, no sub-pickers. They share the same Phase 6 modification pattern:

1. **Form-level `onKeyDown` handler** on the outer `<div className="space-y-5">` wrapper — Enter triggers `handleSave()` when valid. Per UI-SPEC §"Form-by-Form Specs" → SetBedtimeForm → "Enter-advance" and SetWakeTimeForm → "Enter-advance".

2. **Dirty-state `useMemo` + `onDirtyChange` callback** — single-field comparison: `time !== initialTimeSnapshot`. Per UI-SPEC dirty-state table:
   - SetBedtimeForm: dirty if `time !== resolve(smartDefault())` — i.e., the user moved the time off the resolved smart default. Captured via `initialTimeSnapshotRef`.
   - SetWakeTimeForm: dirty if `time !== smartDefault()` — same pattern. Captured via `initialTimeSnapshotRef`.

3. **NO VolumeInput wrap** (these forms have no VolumeInput; the BottomSheet primitive sets `max-w-2xl` for these forms via Plan 06-10's `maxWidth='narrow'` prop, NOT via per-form wrapping).

4. **NO step indicator changes** (single-step forms; no step dots, no back pill, no sticky Next).

5. **Modal `max-w-2xl` at md+ is owned by Plan 06-10**, which sets `maxWidth='narrow'` on `<BottomSheet>` when `sheetMode === 'bedtime'` or `'wakeup'`. This plan does NOT need to do anything about modal max-width.

**Critical mobile invariant:** Both forms gain only `onKeyDown` + `useMemo`/`useEffect` — pure-keyboard behavior + dirty-state callback. Zero pixel diff at <768px.

**Critical data-testid preservation:**
- `bedtime-save` on SetBedtimeForm line 116 — PRESERVE
- `wake-save` on SetWakeTimeForm line 105 — PRESERVE

**Critical Enter-handling nuance** per UI-SPEC §"Form-by-Form Specs" → SetBedtimeForm → "Enter-advance":
The outer onKeyDown handler must allow Enter inside the TimePicker's text input to commit-edit BEFORE saving. The exemption pattern:
```ts
if (target.tagName === 'INPUT' && (target as HTMLInputElement).type === 'text') return;
```
This is identical to Plans 06-05/06/07's pattern. The TimePicker text input handles Enter internally (commit-edit-value), then the user can press Enter AGAIN on a different focused element (or after Tab moves focus off the input) to trigger save.

**`isInvalid` gating in handler:** both forms have an `isInvalid` boolean computed from time-constraint checks. Enter on these forms triggers `handleSave()` only when `!isInvalid`. The handler reads `isInvalid` from the closure. Pattern:
```ts
if (!isInvalid) handleSave();
```

Why combine into ONE plan: both forms are tiny (122 + 111 lines), follow the identical pattern, touch zero overlapping files, and bundling lets the executor batch the two near-identical edits within one ~25% context cost. Splitting would be wasteful overhead.
