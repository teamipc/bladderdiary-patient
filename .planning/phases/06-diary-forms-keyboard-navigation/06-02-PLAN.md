# Plan 06-02: Add 4 new i18n keys for ConfirmDialog (reset-on-cancel) — `common.discardEntryTitle`, `common.discardEntryMessage`, `common.discard`, `common.keepEditing` — and verify 5-locale mirroring

<objective>
Phase 6's reset-on-cancel pattern (per Streamlined Cognition P7 — confirmation for destructive actions) introduces a `ConfirmDialog` invocation when the user attempts to dismiss a form sheet AFTER they have interacted beyond defaults. The ConfirmDialog (`src/components/ui/ConfirmDialog.tsx` — NOT modified by Phase 6) consumes 4 props that need new i18n keys, locked in 06-UI-SPEC §"Reset-on-Cancel Pattern" → "i18n keys (NEW for reset-on-cancel)":

| Key | EN value | Rationale |
|---|---|---|
| `common.discardEntryTitle` | `"Discard this entry?"` | Period replaced with question mark; no em-dashes; collaborative-question framing (per `feedback_collaborative_tone.md` and `feedback_no_em_dashes.md`) |
| `common.discardEntryMessage` | `"Your changes won't be saved."` | Plain language (Streamlined Cognition P6 — no medical jargon); collaborative "your" peer-tone (`feedback_collaborative_tone.md`); period only, no em-dash |
| `common.discard` | `"Discard"` | Action verb on the confirm button; ConfirmDialog `variant="danger"` styles it red |
| `common.keepEditing` | `"Keep editing"` | Recovery action on the cancel button (per UX_PHILOSOPHY §6 corollary "recoverable progress") |

These are added to `messages/en.json` under the existing `common` namespace (verified present at line 8, currently containing 33 keys including `back`, `cancel`, `next`, `previous`, `previousStep`, `nextStep`, `now`, `save`, `update`, `yes`, `no`, `done`, `log`, `appName`, `close`, `help`, `delete`, etc.). The PostToolUse `i18n-sync` hook auto-fires on `messages/en.json` edits and mirrors the new keys to `messages/{fr,es,pt,zh,ar}.json` with locale-natural translations.

This plan is Wave 1, blocks 06-10 (DayPageClient — consumes these 4 keys via `tc('discardEntryTitle')` etc.). It has no `depends_on` and runs in parallel with 06-01 and 06-03.

The plan deliberately follows the EXACT structural pattern of Plan 05-02 (which added `nav.primaryNavAriaLabel`) — Task 1 edits `messages/en.json`, Task 2 verifies the 5 non-English locales received the mirror and adds manually if any did not. This pattern was checker-PASSED in Phase 5.

Existing key reuse REJECTED: `landing.areYouSure` ("Are you sure?", line 64 of en.json) is about resetting the WHOLE diary — destructive at a different scale than discarding ONE entry. The wording for entry-discard MUST be specific ("Discard this entry?") to avoid scaring users into thinking they're about to lose all their data. Per UI-SPEC.
