# Plan 06-01: Add `@keyframes modalIn` + `.animate-modal-in` + `.safe-bottom-reset` utilities to globals.css

<objective>
Phase 6 introduces ONE new animation for the desktop modal slide-in transformation of `BottomSheet.tsx` (locked in 06-UI-SPEC §"BottomSheet — desktop modal transformation" → "Animation spec (NEW — `animate-modal-in`)"). This plan lands the supporting CSS in `src/app/globals.css` BEFORE the BottomSheet primitive is modified in Plan 06-04, so the executor of 06-04 only needs to add the `md:animate-modal-in` and `md:safe-bottom-reset` Tailwind class strings on the modal card without simultaneously editing globals.css.

The 3 additions:

1. `@keyframes modalIn` — opacity 0→1 + transform `translateY(8px) scale(0.985)` → `translateY(0) scale(1)`. 180ms cubic-bezier(0.16, 1, 0.3, 1) out-expo easing. Per Boomer-safe override 3 (animation ≤ 200ms cap; 180ms is comfortably under).
2. `.animate-modal-in` utility class that applies the keyframe.
3. `.safe-bottom-reset` utility class that strips `padding-bottom: env(safe-area-inset-bottom)` at md+ (since the modal at md+ is no longer pinned to the viewport bottom, the iOS home-indicator safe-area padding is irrelevant).

Wave 1 — no dependencies. Blocks Plan 06-04 (BottomSheet primitive consumes `animate-modal-in` and `safe-bottom-reset`).

This is a CSS-only change. Zero TypeScript impact. Zero pixel diff at mobile (the new classes are only applied via `md:` prefixes in Plan 06-04, not in this plan).

The codebase audit confirms (Bash grep):
- `globals.css` line 17 uses `@theme inline { ... }` (Tailwind 4 native theme directive)
- `globals.css` line 76 defines `.safe-bottom` as a PLAIN CSS class (not `@utility` syntax — the project does NOT use `@utility`)
- Existing animations are defined as plain CSS rules: `.animate-slide-up { animation: slideUp 0.3s ease-out both; }` (lines 139-141)
- The UI-SPEC suggested `@utility animate-modal-in { ... }` but the project convention is plain `.classname { animation: ... }` — Plan 06-01 uses the project convention, NOT the UI-SPEC's suggested `@utility` syntax (this is the planner refining the UI-SPEC's recommendation to match the actual codebase convention).
- `prefers-reduced-motion` block (lines 95-103) already collapses ALL animations to 0.01ms, so the new `modalIn` keyframe automatically respects the user's reduced-motion preference without explicit handling.

Open questions addressed:
- Q1 (UI-SPEC) — animation duration locked at 180ms (under 200ms cap); the executor of Plan 06-04 may tune to 200ms during verification if 180ms feels too snappy, but MUST NOT exceed 200ms.

Constraints from CLAUDE.md + memory:
- `feedback_lightning_css_grouped_selectors.md` — DO NOT group selectors with `,` if any selector has alpha-suffix or `::after`; this plan uses single-class selectors (no grouped rules), so this gotcha does not apply but is honored.
- Lightning CSS gotcha (file footer notes lines 940-980) — single-class rules with single keyframes are safe; this plan uses that exact pattern.
</objective>

Insert the new keyframe definition immediately AFTER the existing `@keyframes fadeIn { ... }` block (currently ending around line 128), and insert the new utility classes immediately AFTER the existing `.animate-fade-in { ... }` rule (currently around line 149).

    Insertion 1 — new keyframe (place AFTER `@keyframes fadeIn { from { opacity: 0; } to { opacity: 1; } }`):

    ```css
    @keyframes modalIn {
      from {
        opacity: 0;
        transform: translateY(8px) scale(0.985);
      }
      to {
        opacity: 1;
        transform: translateY(0) scale(1);
      }
    }
    ```

    Insertion 2 — new utility classes (place AFTER `.animate-fade-in { animation: fadeIn 0.2s ease-out both; }`):

    ```css
    /* Desktop modal slide-in — used by BottomSheet at md+ via md:animate-modal-in.
       Per Boomer-safe override 3 (animation cap 200ms): 180ms stays safely under
       the cap; cubic-bezier(0.16, 1, 0.3, 1) is an "out-expo"-style decelerated
       finish that feels confident at short duration. Mobile keeps the existing
       animate-slide-up; this utility is gated to md+ usage by callers. */
    .animate-modal-in {
      animation: modalIn 180ms cubic-bezier(0.16, 1, 0.3, 1) both;
    }

    /* Strip the iOS home-indicator safe-area padding at md+ where the modal
       is no longer pinned to the viewport bottom. Mobile's .safe-bottom keeps
       the env() padding; .safe-bottom-reset overrides it inside an @media gate. */
    .safe-bottom-reset {
      /* No effect at <768px — only matters at md+ where it strips safe-area. */
    }
    @media (min-width: 768px) {
      .safe-bottom-reset {
        padding-bottom: 0 !important;
      }
    }
    ```

    Notes on the choices:
    - Use plain CSS class syntax (`.animate-modal-in { ... }`) NOT `@utility` — the existing project convention (verify with grep of existing `.animate-*` rules) is plain classes. UI-SPEC suggested `@utility` but the project does not use that Tailwind 4 directive; the planner aligns to actual codebase convention.
    - `animation: ... both;` — `both` keeps the final keyframe state applied after animation ends (matches the existing `.animate-fade-slide-up`, `.animate-scale-in`, `.animate-slide-up`, `.animate-fade-in` pattern in this file).
    - `safe-bottom-reset` is split into a base no-op rule + an `@media` override. This keeps the class string `safe-bottom-reset` valid at all viewports (Tailwind can target it as a utility) while ONLY actually doing anything at md+. The `!important` overrides the inline `padding-bottom: max(env(safe-area-inset-bottom, 0px), 8px)` from `.safe-bottom` at line 76-82 when both classes are present (which is the BottomSheet card scenario at md+).
    - Do NOT modify the existing `prefers-reduced-motion` block (lines 95-103) — it already collapses ALL `animation-duration` to 0.01ms via universal selector, so the new `modalIn` keyframe automatically respects reduced-motion preference without explicit handling.

    Do NOT alter any other section of `globals.css`. Do NOT renumber/reorder existing keyframes. Do NOT touch the `@theme inline` block, the `.nighttime-bg` overrides (lines 973-980), or any of the night-mode CSS rules.

    Lightning CSS safety check: the two new utility classes (`.animate-modal-in` and `.safe-bottom-reset`) are SINGLE-class selectors (no `,` grouping, no alpha-suffix selectors, no `::after` combinations). Per `feedback_lightning_css_grouped_selectors.md` the failure modes do not apply.
