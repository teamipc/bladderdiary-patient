# Plan 06-03: Migrate `Button.tsx` from `focus:` to `focus-visible:` and bump ring color from `ipc-400` to `ipc-500` (Phase 5 Q5 carry-over)

<objective>
The shared `Button` primitive (`src/components/ui/Button.tsx`, 52 lines) currently uses Tailwind's `focus:` modifier for its ring styling. Per 06-UI-SPEC §"Button.tsx focus-visible migration" (locked) and §"Focus-visible ring spec (REUSED from Phase 5)":

- Migrate `focus:` → `focus-visible:` on ALL focus-ring classes
- Bump ring color from `ipc-400` to `ipc-500` for consistency with Phase 5's chrome focus spec (Plan 05-04 NavLink)
- Migrate variant-specific overrides (`drink`, `bedtime`, `leak`, `night`, `hero`) from `focus:ring-*` to `focus-visible:ring-*`

The mobile effect of `focus:` → `focus-visible:` migration is a UX **improvement**, not a regression: `focus-visible` triggers ONLY on keyboard focus, not on touch/click. After this change, mobile users no longer see a lingering focus ring after a tap (which the current `focus:` pattern produces). Per 06-UI-SPEC §"Mobile Invariants" table row for `Button.tsx`:
> "Net mobile diff: LESS focus ring after taps (focus-visible doesn't trigger on touch like focus does). This is a UX IMPROVEMENT, not a regression. Per CONTEXT.md the change is explicitly aligned with Phase 5's NavLink pattern. NOT a violation of mobile-pristine."

Phase 5's CONTEXT.md §"Claude's Discretion" (Q5) deferred this exact migration to Phase 6: *"Whether to introduce a `useBreakpoint` hook for any conditional rendering, or rely entirely on Tailwind's responsive classes (`md:hidden` etc.)"* — focus-visible falls under the same "Q5 (focus-visible token expansion)" notation in 06-CONTEXT.md §"Surfaces in scope" point 10: *"`src/components/ui/Button.tsx` — focus-visible token expansion if needed (Phase 5 Q5 deferred to Phase 6)"*.

Why Wave 1: this primitive is consumed by ALL 5 diary forms (`LogDrinkForm`, `LogVoidForm`, `LogLeakForm`, `SetBedtimeForm`, `SetWakeTimeForm`) via the sticky Next button (`fullWidth size="lg" variant={variant}`) and the Save button. Phase 6 wants the focus-visible behavior to be present BEFORE the forms add their own form-level focus-trap and Enter-advance handlers (Waves 3–5) so the executor can rely on Button having the correct keyboard-only focus ring.

This plan is also Wave 1 — no dependencies. Runs in parallel with 06-01 and 06-02. Blocks Waves 3–5.
