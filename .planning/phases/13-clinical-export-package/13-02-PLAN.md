**Key design decisions resolved here (locked, NOT for executor to revisit):**

- **D-01 — Clinical metrics live in QuestionnaireResponse, NOT as Observation resources.** Per RESEARCH §'Clinical metrics' (the answer to CONTEXT.md Open Question #1): FHIR semantics put computed values in QR (assessment data), direct measurements in Observation. Per memory `ipc-calculations.md`: 24HV/NPi/AVV/MVV/NBC are computed from raw void+drink data, not direct measurements — they belong in QR.

  This deliberately diverges from "everything is an Observation" approaches. The result: Epic readers see a coherent narrative QR section for clinical metrics, while per-event Observations populate the I&O flowsheet for the raw data.

- **D-02 — QR.linkId naming convention: `qr-<category>-<scope>` kebab-case.** Per FHIR R4 spec: `QuestionnaireResponse.item[].linkId` is an opaque identifier that the corresponding `Questionnaire` would have defined. Since we are NOT shipping the Questionnaire (out of scope per CONTEXT — we ship the response only), our linkIds are self-describing kebab-case.

  **Locked catalog (16 items, source-aligned to what `computeMetrics` actually produces):**
  - `qr-diary-startdate` (the patient's Day 1 ISO date)
  - `qr-diary-timezone` (IANA tz string)
  - `qr-diary-age` (patient-reported age — duplicates `Patient.birthDate` year derivation but in narrative QR context for clinician convenience)
  - `qr-metric-24hv-period1`, `qr-metric-24hv-period2` (read from `metrics.periods[0/1]?.twentyFourHV`)
  - `qr-metric-npi-period1`, `qr-metric-npi-period2` (read from `metrics.periods[0/1]?.nPi`)
  - `qr-metric-avv-period1`, `qr-metric-avv-period2` (read from `metrics.periods[0/1]?.avv`. AVV is computed PER-PERIOD on `PeriodMetrics`, NOT per-day on `DayMetrics`. The 13-02 draft incorrectly assumed per-day; corrected here against `src/lib/calculations.ts:43-50`.)
  - `qr-metric-mvv` (read from `metrics.mvv`. MVV is a SINGLE top-level number on `DiaryMetrics`, NOT per-day. The 13-02 draft incorrectly assumed per-day mvv-day1/2/3; corrected here against `src/lib/calculations.ts:52-65`.)
  - `qr-bedtime-day-1`, `qr-bedtime-day-2`, `qr-bedtime-day-3`
  - `qr-wake-day-1`, `qr-wake-day-2`, `qr-wake-day-3`

  **DROPPED from earlier 18-item draft:**
  - `qr-metric-nbc` — Nocturnal Bladder Capacity is NOT computed by `calculations.ts` today. Per plan-checker BLOCKER B3 + the user-prompt NBC decision: drop the linkId entirely (Option A). Clinicians can derive NBC from `metrics.nights[i].nocturnalVolumeMl` and the FMV timestamp in `metrics.nights[i].fmvIso` if needed. Adding NBC to `calculations.ts` is intentional scope expansion deferred to a future phase. The 13-02 draft also listed per-day AVV and per-day MVV, which the corrected catalog collapses to per-period AVV and a single top-level MVV to match the source.

  The Questionnaire that would define these linkIds is out of scope — clinicians treat the QR as a self-documenting narrative.

- **D-03 — null-safe metric encoding.** Per `computeMetrics` semantics in `src/lib/calculations.ts`: 24HV/NPi/AVV/MVV/NBC may be `null` when data is insufficient (e.g. Day 1 is excluded; partial day-3 data). When a metric is null, the QR item is still emitted but with `answer: []` (empty array) AND a `text` field set to "Insufficient data for this metric". Per FHIR R4: `QuestionnaireResponse.item[].answer` is optional; an item without answer signals "question asked, no answer given" — which is exactly the semantic.

  Alternative considered: omit the item entirely when null. REJECTED because the clinician then can't distinguish "this metric was insufficient" from "this metric was not computed at all" — both look the same. Explicit empty answer + text gives the clinical signal.

- **D-04 — `valueDecimal` for numeric metrics, NOT `valueQuantity`.** Per FHIR R4 QuestionnaireResponse spec: numeric answers use `valueDecimal` (a plain number). The UCUM unit is implicit in the linkId semantic (24HV is mL, NPi is unitless ratio, AVV/MVV are mL, NBC is mL). This is the FHIR-idiomatic choice for assessment-questionnaire numeric responses; valueQuantity is for direct-measurement Observations.

- **D-05 — `valueDateTime` (ISO 8601 UTC string) for bedtime + wake markers.** Per FHIR R4 spec: `dateTime` answer type. The `timestampIso` field on `BedtimeEntry`/`WakeTimeEntry` is already canonical UTC ISO 8601 — pass through unchanged.

- **D-06 — AJV strategy: devDependency only, vitest CI only, NEVER bundled.** Per RESEARCH §'AJV + FHIR R4 Schema Validation Strategy' and §'Architectural Responsibility Map':
  - `npm install --save-dev ajv ajv-formats` — both go in devDependencies, never dependencies.
  - The FHIR R4 schema (~3-5MB unpacked) lives at `test-fixtures/fhir/fhir.schema.json` per RESEARCH. NOT in `node_modules`, NOT in `src/`.
  - `src/lib/exportFhir/validate.ts` is intentionally NOT re-exported from `src/lib/exportFhir/index.ts` — the barrel only exposes builders. To use the validator, test code does `import { validateFhirBundle } from '@/lib/exportFhir/validate'` directly. The static-grep guard in the verify block enforces that no production source under `src/app/`, `src/components/`, `src/lib/exportPackage/`, or `src/lib/exportPdf/` imports from `@/lib/exportFhir/validate`.

  Rationale: AJV's compiled validator + the FHIR R4 schema together are ~3-5MB. Bundling into the client violates the 5MB total export-size budget (CONTEXT.md Constraints) and adds CSP `unsafe-eval` requirements (RESEARCH §'Why not runtime validation in the browser').

- **D-07 — AJV options: `{ strict: false, allErrors: true }`.** Per RESEARCH:
  - `strict: false` — FHIR R4 schema uses several JSON Schema features (oneOf, draft-04 cardinality) that AJV's strict mode rejects. Disabling strict is the standard pattern for FHIR validation.
  - `allErrors: true` — collect all violations, not just the first. Helps debugging when a Bundle has multiple shape problems.

  Add `ajv-formats` (handles `date-time`, `uri`, `regex` formats per the FHIR R4 schema's `format:` annotations).

- **D-08 — Schema fetch strategy: vendor the schema, do NOT fetch at test time.** Two options:
  - (a) Check `test-fixtures/fhir/fhir.schema.json` into git (~3-5MB). Tests load it via `readFileSync`. Simple, deterministic, offline-runnable. **CHOSEN.**
  - (b) Fetch from hl7.org at test setup. Fails offline / in CI sandboxes; adds network flake. REJECTED.

  The 3-5MB file size is well within reasonable git tolerance — no Git LFS needed. The file changes only when FHIR R4 itself revises (rare — R4 has been stable since 2019).

  **Note on schema acquisition:** the executor downloads from `https://hl7.org/fhir/R4/fhir.schema.json` via `curl` during Task 3 (NOT via a Node fetch in test code). See Task 3 for the exact `curl` invocation.

- **D-09 — Two separate test files: shape (no AJV) + validate (AJV).** Per RESEARCH §AJV CI cost (~1-3s first compile): splitting the tests means CI runs them in parallel, and the AJV-free shape tests fail fast even when the schema fixture is missing.

  - `src/__tests__/fhir-bundle-shape.test.ts` — runs in <1s, no AJV dependency, asserts the LOCKED contract (Patient PHI audit, LOINC codes, UCUM canonical mL, ID prefix scheme, QR linkIds present).
  - `src/__tests__/fhir-validate.test.ts` — runs in ~3-5s on first AJV compile, asserts the official FHIR R4 schema is satisfied AND that intentionally-malformed Bundles get rejected.

  Both tests use the same `buildMinimalDiaryState()` fixture (cloned from `src/__tests__/pdf-blob-content.test.ts:86-161` per PATTERNS.md §5(a)) so the contract is identical.

- **D-10 — `Observation.encounter` remains omitted.** Per RESEARCH §Open Questions #3: patient self-report at home has no encounter to reference; clinicians fill on import. Lock-in via the negative test case.

- **D-11 — Per-day data structure inside QR.** Beyond clinical metrics, the QR also encodes the per-day diary structure — start date, timezone, age, bedtime, wake time for each of Day 1/2/3. This gives the clinician a single coherent "diary summary" resource without having to derive day boundaries from raw Observations. The full linkId list per D-02.

- **D-12 — QR insertion at Bundle.entry index 1 (between Patient at 0 and Observations at 2..N).** Per the 13-01 Task 5 locked extension point. The orchestrator change is surgical: 13-01 emitted `[patientEntry, ...observationEntries]`; 13-02 changes it to `[patientEntry, qrEntry, ...observationEntries]`. No other restructuring.

    import Ajv from 'ajv';
    import type { ErrorObject } from 'ajv';
    import addFormats from 'ajv-formats';
    import { readFileSync } from 'node:fs';
    import { resolve } from 'node:path';
    import type { FhirBundle } from './types';

    /** Path to the vendored FHIR R4 JSON Schema. Resolved from process.cwd(). */
    const SCHEMA_PATH = resolve(process.cwd(), 'test-fixtures', 'fhir', 'fhir.schema.json');

    let cachedValidate: ((data: unknown) => boolean) | null = null;
    let cachedAjv: Ajv | null = null;

    /** Lazy AJV setup — compiles once per process; subsequent calls reuse the compiled validator. */
    function getValidator() {
      if (cachedValidate && cachedAjv) {
        return { ajv: cachedAjv, validate: cachedValidate };
      }
      const schema = JSON.parse(readFileSync(SCHEMA_PATH, 'utf8')) as Record<string, unknown>;
      const ajv = new Ajv({ strict: false, allErrors: true });
      addFormats(ajv);
      ajv.addSchema(schema, 'fhir-r4');
      // FHIR's $ref-based schema means we compile the Bundle reference, not the top-level schema.
      const validate = ajv.compile({ $ref: 'fhir-r4#/definitions/Bundle' });
      cachedAjv = ajv;
      cachedValidate = validate as (data: unknown) => boolean;
      return { ajv, validate: cachedValidate };
    }

    /** Validate a FHIR Bundle against the R4 JSON Schema. Returns { valid, errors }. */
    export function validateFhirBundle(bundle: FhirBundle | unknown): { valid: boolean; errors: ErrorObject[] } {
      const { validate } = getValidator();
      const valid = validate(bundle);
      const errors = ((validate as unknown as { errors?: ErrorObject[] }).errors ?? []);
      return { valid, errors };
    }
