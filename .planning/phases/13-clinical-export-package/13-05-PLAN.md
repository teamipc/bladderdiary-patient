**Key design decisions (resolved here):**

- **D-01 — Local build + `npx serve` pattern, mirrored from Phase 12-04.** Per `e2e/phase12-seo.spec.ts`: the spec runs against `http://localhost:4173` served from `out/`. The verification harness command sequence:
  1. `npm run build` (which now includes the post-build copy step from Phase 12 + the new exportPackage chunk from Phase 13).
  2. `npx --yes serve out -l 4173 &` (background).
  3. `PW_TEST_MATCH='phase13-export-package\.spec\.ts' npx playwright test e2e/phase13-export-package.spec.ts`.

  The `--test-match` (via env var) is REQUIRED to override `playwright.config.ts`'s `testMatch: /(walkthrough|deep-flow|a11y)\.spec\.ts/`. Per the Phase 5-07 + Phase 12-04 lesson, `--grep` filters test TITLES not file paths.

- **D-02 — Zip generation harness: page.evaluate over real-click.**

  Two approaches considered:
  - **(a) Real click + download interception via `page.on('download', ...)`.** Browser-natural; matches what the user does. BUT: the share-supported codepath does NOT trigger a download; it triggers `navigator.share` which Playwright cannot intercept cleanly without mocking. And on desktop where share is unavailable, the download IS triggered but the resulting file lives in Playwright's temp dir, harder to introspect.

  - **(b) Direct invocation via `page.evaluate(() => {...})` harness.** The spec runs `page.evaluate` to seed the diary store via localStorage, then runs another `page.evaluate` to dynamically import `@/lib/exportPackage`, generate the blob, and return the bytes as a base64 string. Playwright code then decodes and unzips. Faster, deterministic, no temp-file mgmt, and works for BOTH share-supported and share-unavailable codepaths.

  **CHOSEN: (b).** This is the same approach used by `pdf-blob-content.test.ts` for PDF byte introspection — generate in-page, return bytes, introspect from test code.

  The trade-off: (b) doesn't exercise the actual `<Button onClick={handleHeroPackage}>` codepath, so PKG-01's "the button renders and is clickable" check happens via DOM assertion (presence + locator), not via real click. PKG-04's share mock test is a SEPARATE test case that does click the button (with `navigator.share` mocked).

- **D-03 — Locale-natural README substring assertions.** For each of the 6 locales, after unzipping the README, assert:
  - EN: contains `'My Flow Check'`, `'Send to your healthcare team'` (no — that's the button label; the README has `'My Flow Check: 3-Day Bladder Diary'`), `'Epic'`, `'FHIR R4 Bundle'`.
  - FR: contains `'Profil patient'` or `'Patient'` (depending on the final FR translation from 13-03 Task 3), `'Epic'` (preserved English), `'FHIR'` (preserved English).
  - ES: contains `'Perfil del paciente'` or `'Paciente'`, `'Epic'`, `'FHIR'`.
  - PT: contains `'Perfil do paciente'` or `'Paciente'`, `'Epic'`, `'FHIR'`.
  - ZH: contains `'病历号'` or some Mandarin patient-profile term, `'Epic'`, `'FHIR'`.
  - AR: contains `'ملف المريض'` or some Arabic patient-profile term, `'Epic'`, `'FHIR'`.

  The exact assertions depend on the final translations from 13-03 Task 3. The spec uses LOOSE substring matches that survive minor wording variations. Tight per-locale word-level assertions would couple the test too tightly to the translation choices.

- **D-05 — Mock navigator.share + navigator.canShare for the PKG-04 share-path test.**

  ```ts
  await page.addInitScript(() => {
    const shareCalls: Array<{ filesCount: number; firstFileType: string; firstFileName: string }> = [];
    const canShareCalls: number[] = [];
    (window as Window & { __shareCalls?: typeof shareCalls }).__shareCalls = shareCalls;
    (window as Window & { __canShareCalls?: typeof canShareCalls }).__canShareCalls = canShareCalls;
    Object.defineProperty(navigator, 'canShare', {
      configurable: true,
      value: (data: { files?: File[] }) => {
        canShareCalls.push(canShareCalls.length);
        return Array.isArray(data?.files) && data.files.length > 0;
      },
    });
    Object.defineProperty(navigator, 'share', {
      configurable: true,
      value: async (data: { files?: File[]; title?: string }) => {
        const f = data?.files?.[0];
        shareCalls.push({ filesCount: data?.files?.length ?? 0, firstFileType: f?.type ?? '', firstFileName: f?.name ?? '' });
      },
    });
  });
  ```
