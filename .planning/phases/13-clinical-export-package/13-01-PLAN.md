Phase 13 — Plan 1 of 5. **Wave 1 (alone)** — sets up the FHIR module that 13-02 (QuestionnaireResponse + validation) and 13-03 (package zip) both depend on.

This plan creates the **FHIR R4 Bundle generator core** as a new module `src/lib/exportFhir/` containing 5 files: `types.ts`, `loinc.ts`, `patient.ts`, `observations.ts`, and `index.ts`. The public entry is `generateFhirBundle(state: DiaryState): FhirBundle` — a pure synchronous transform that produces a fully-shaped FHIR R4 collection Bundle from a `DiaryState` snapshot.

**Why this is plan 1 and stands alone in Wave 1:**

The Bundle is the central data structure that:
- 13-02 will validate against the official FHIR R4 schema (AJV) AND extend with `QuestionnaireResponse` for IPC clinical metrics
- 13-03 will JSON-stringify and zip alongside the PDF/CSV/README
- 13-04 will dynamically import via `generatePackageBlob` from 13-03

If 13-01 is wrong in shape, every downstream plan misroutes. Wave 1 isolates this — no parallel work on top of an unstable foundation.

**Key design decisions resolved here (locked, NOT for executor to revisit):**

- **D-01 — LOINC code selection: 9187-6 / 8999-5 / SNOMED 162172004 + LOINC 28232-7.** Per RESEARCH §'LOINC Code Decision Matrix', three codes are LOCKED:
  - **Voids**: LOINC `9187-6` "Urine output" (Property=Volume, Time=Point in time, Class=IO_OUTPUT). This is the Epic-flowsheet-compatible code. The CONTEXT.md guesses of `19153-6` (specimen-class, WRONG) and `9192-6` (24-hour aggregate, WRONG) are explicitly rejected per RESEARCH §Pitfall 1.
  - **Drinks**: LOINC `8999-5` "Fluid intake oral Estimated" (point-in-time, estimated qualifier matching patient self-report). The CONTEXT.md guess of `8657-8` does NOT exist in the LOINC catalog per RESEARCH §Assumption A7. Fall back to `9108-2` is documented but `8999-5` is primary.
  - **Leaks**: SNOMED `162172004` "Urinary incontinence" PRIMARY (event-grain finding code) + LOINC `28232-7` "Total urinary incontinence" SECONDARY (both in the `code.coding[]` array per RESEARCH Assumption A3 mitigation — EHR picks whichever it understands).

- **D-02 — Patient.identifier shape: tracking-coded, NOT MRN-shaped.** Per RESEARCH §'FHIR Patient.identifier Decision' + §Pitfall 4: when `clinicCode` is set AND passes the existing `[A-Za-z0-9-]{1,32}` regex (defense in depth — store already validates, builder re-validates), emit:
  ```
  identifier: [{
    use: 'secondary',
    type: { coding: [{
      system: 'https://myflowcheck.com/codesystem/identifier-type',
      code: 'TRACKING',
      display: 'Clinic-assigned tracking code, not a medical record identifier'
    }]},
    system: 'https://myflowcheck.com/clinic-code',
    value: clinicCode
  }]
  ```
  When clinicCode is null OR fails regex, omit `identifier` entirely (do NOT emit an empty array). This avoids the Epic "duplicate patient" trap where a tracking code masquerades as an MRN.

- **D-03 — Birthdate fuzzed to YEAR ONLY.** Per RESEARCH §Pitfall 3 + FHIR-EX-03: `state.age` is the patient's reported age. Compute `year = new Date().getFullYear() - state.age` and emit `birthDate: String(year)` (e.g. `"1971"`). Do NOT use `new Date(year, 0, 1).toISOString().slice(0, 10)` — that emits day-precision `"1971-01-01"` which is HIPAA Safe Harbor full-DOB-equivalent PHI. When `state.age === null`, omit `birthDate` entirely.

- **D-04 — No locale arg in `generateFhirBundle`.** Per RESEARCH §'Architectural Responsibility Map' + §Architecture Patterns and PATTERNS.md §'Why this diverges from exportPdf' point 1: FHIR clinical terminology stays English. The display strings on LOINC/SNOMED codes (`'Urine output'`, `'Urinary incontinence'`) are part of the code system, not user-facing UI copy. Patient's UI locale (`messages/<locale>.json`) does NOT influence FHIR output. Only the README inside the zip (built by 13-03) is locale-aware.

- **D-05 — Volume is canonical mL, ALWAYS.** Per RESEARCH §Anti-Patterns + §Assumption A9 + §Pitfall 2: regardless of `state.volumeUnit: 'mL' | 'oz'` (which is a UI display preference), every FHIR `valueQuantity` emits `{ value: <mL>, unit: 'mL', system: 'http://unitsofmeasure.org', code: 'mL' }`. The 4 fields are MANDATORY — emitting `{ value, unit }` only is technically valid FHIR but unparseable by Epic's UCUM-strict pipeline.

- **D-06 — Double-void volume combined for the void Observation.** Per RESEARCH §Code Examples (`buildVoidObservation`): emit `value: v.volumeMl + (v.doubleVoidMl ?? 0)`. This matches the IPC clinical convention that a double-void's combined volume is one "void event" with the combined volume — see memory `ipc-calculations.md`.

- **D-07 — Bundle is `type: 'collection'`, not `transaction` or `document`.** Per CONTEXT FHIR-EX-02 + RESEARCH §Bundle assembly: collection is the right type for "loose set of resources sharing a topic" with no atomic-commit semantics. Transaction would require server-side processing semantics; document would require a Composition resource as the root.

- **D-08 — `crypto.randomUUID()` for `fullUrl`.** Per RESEARCH §'Don't Hand-Roll': `urn:uuid:` prefix + the result of `crypto.randomUUID()` (available in all target browsers per Assumption A6 — iOS 14.5+, all modern Chrome/Firefox/Edge). No polyfill needed.

- **D-09 — `types.ts` is hand-rolled minimal R4 subset.** Per RESEARCH §Pattern 2 + PATTERNS.md §'Why diverges' point 4 + Anti-Pattern: do NOT import `@types/fhir` (~10MB of types). Define ONLY the resources Phase 13 emits — `FhirBundle`, `FhirBundleEntry`, `FhirPatient`, `FhirIdentifier`, `FhirObservation`, `FhirCoding`, `FhirCodeableConcept`, `FhirReference`. `FhirQuestionnaireResponse` is added in 13-02 (when it's actually built); 13-01 leaves a `// see 13-02` placeholder import-spot in `types.ts`.

- **D-10 — `QuestionnaireResponse` is deferred to 13-02.** Per the 5-plan structure: 13-01 ships Patient + Observations + Bundle skeleton. 13-02 adds `QuestionnaireResponse` (which encodes IPC clinical metrics per FHIR-EX-02) AND the AJV validation suite. Splitting this way keeps 13-01 self-contained at ~30-40% context and gives 13-02 a single coherent concern (clinical-metrics encoding + schema validation).

  In 13-01, the orchestrator `generateFhirBundle` is structured so that 13-02 can ADD the QR entry without rewriting the orchestrator — see Task 5 below for the exact extension point.

**Out of scope for 13-01 (deferred to later plans):**

- AJV validation, FHIR R4 schema fetch/check (13-02)
- QuestionnaireResponse builder + IPC metrics encoding (13-02)
- README zip composition (13-03)
- `ExportActions` UI reshape + Web Share API two-stage probe (13-04)
- Playwright spec covering hero CTA flow (13-05)
- jszip / ajv dependency installs — 13-02 + 13-03 own those (13-01 has zero new dependencies; types.ts is hand-rolled per D-09)

This plan does NOT modify any existing file. It is purely additive — creates a new directory + 5 new files. Backward compatibility with the existing 3-button export flow is trivially preserved (no existing code path is touched).

**Critical reminders before writing observations.ts (Task 4):**
- VoidEntry uses `doubleVoidMl?: number` (undefined-or-number). The combine expression is `v.volumeMl + (v.doubleVoidMl ?? 0)`. NEVER access `v.doubleVoidMl` assuming `null` semantics — a typeof check would fail.
- VoidEntry has a REQUIRED `note: string` field (may be empty). The empty-trim gate is `v.note.trim() !== ''`. NEVER write `v.note && v.note.trim() !== ''` — TS strict treats `v.note` as definitely a string, so the truthiness guard is redundant and ESLint may flag.
- DrinkEntry similarly has REQUIRED `note: string`. Same trim gate.
- LeakEntry's free-text field is `notes` (PLURAL), NOT `note`. The builder's trigger-text composition reads `l.notes`, and the gate is `l.notes && l.notes.trim() !== ''` because `notes?: string` IS optional.
- LeakEntry has `trigger: LeakTrigger` (required), `urgencyBeforeLeak: boolean | null` (required), `amount?: LeakAmount | null` (optional). 13-01 surfaces ONLY `trigger` in the Observation.note; the other two fields are deferred (they go into the QR in 13-02 or stay un-exported).

The `crypto.randomUUID()` global is available in browsers (Web Crypto API) — TypeScript's lib.dom.d.ts types it as `() => `${string}-${string}-${string}-${string}-${string}``. Per RESEARCH §Assumption A6: all target browsers (iOS 14.5+, modern Chrome/Firefox/Edge) support it. No polyfill needed for this app's browser-support matrix.

     * Build a FHIR R4 collection Bundle from a DiaryState snapshot.

     * Synchronous + pure. Does NOT take a locale: clinical terminology stays

    export function generateFhirBundle(state: DiaryState): FhirBundle {
      const patient = buildPatient(state);

      // 13-02 will insert the QuestionnaireResponse entry here (between Patient
      // and the Observations). The extension point is: build qrEntry above the
      // observationEntries, then concat [patientEntry, qrEntry, ...observationEntries].
      // 13-01 omits the QR entry — Bundle is still a valid collection with just

      const observations = [
        ...state.voids.map((v) => buildVoidObservation(v, patient.id)),
        ...state.drinks.map((d) => buildDrinkObservation(d, patient.id)),
        ...state.leaks.map((l) => buildLeakObservation(l, patient.id)),
      ];

      const patientEntry: FhirBundleEntry = {
        fullUrl: `urn:uuid:${crypto.randomUUID()}`,
        resource: patient,
      };
      const observationEntries: FhirBundleEntry[] = observations.map((o) => ({
        fullUrl: `urn:uuid:${crypto.randomUUID()}`,
        resource: o,
      }));

      return {
        resourceType: 'Bundle',
        type: 'collection',
        timestamp: new Date().toISOString(),
        entry: [patientEntry, ...observationEntries],
      };
    }
