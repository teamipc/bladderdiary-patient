**Key design decisions resolved here (locked, NOT for executor to revisit):**

- **D-01 — JSZip over fflate.** Per RESEARCH §JSZip vs fflate Decision Matrix + Recommendation: jszip 3.10.1, ~50KB gzipped, ecosystem-mature, already-lazy code path mitigates the bundle-size disadvantage. fflate would save ~40KB but lands on a code path that's already dynamic-imported (only fires when patient hits Day 3 export) — marginal value.

- **D-02 — Dynamic-import JSZip from INSIDE `generatePackageBlob`, not from `ExportActions.tsx`.** Both layers could lazy-load:
  - (a) `ExportActions.tsx` dynamic-imports the whole `@/lib/exportPackage` module (the existing pattern for `@/lib/exportPdf` at `ExportActions.tsx:51`). This already keeps the package code out of the entry bundle.
  - (b) Inside `generatePackageBlob`, `await import('jszip')` keeps jszip out of the package-module's static deps, so even within the lazy-loaded `exportPackage` chunk, jszip lands as a separate sub-chunk.
  
  We do BOTH. (a) is implemented in 13-04. (b) is implemented HERE so the module itself stays jszip-import-free at the static level.

- **D-03 — Zip filenames are clinician-sort-order per PKG-02.** Inside the zip:
  - `01-clinical-report.pdf` (not `my-flow-check-<date>.pdf` — the standalone PDF filename) — STRIP and rename when zipping per RESEARCH §Anti-Patterns "Don't reuse generatePdfBlob's filename field in the zip".
  - `02-events.csv` (not `my-flow-check-<date>.csv`).
  - `03-emr-bundle.fhir.json` (new file from `generateFhirBundle(state)` + `JSON.stringify(bundle, null, 2)`).
  - `README.txt` (no number prefix because alphabetical sort puts it last anyway; intentional — the README is the "open this" file).
  
  The OUTER zip filename uses `myflowcheck-` prefix (no hyphens between brand words) per PATTERNS.md §'Filename convention' — the zip name appears in the share-sheet title row and clinician's downloads folder, both benefit from the compact brand form. Existing PDF/CSV standalone filenames stay `my-flow-check-` for backward compatibility.

- **D-04 — README locale resolution: pass resolved messages object as a parameter, not via next-intl async getter.** Reason: `generatePackageBlob` runs in the BROWSER from the summary page (a `'use client'` component). next-intl's CLIENT pattern is `useTranslations()` (hook, returns sync `t()` function). We CANNOT call hooks outside components, so we cannot call `useTranslations` inside `generatePackageBlob` directly. Two options:
  - (a) `buildReadme(state, locale)` calls `next-intl/server` `getTranslations({ locale, namespace })` async. Works in Next.js 16 client components if the static-export config supports it; verify at execution time.
  - (b) `ExportActions.tsx` resolves the translations via `useTranslations('exportPackage.readme')` at component render time, captures the `t()` function, and passes it INTO `generatePackageBlob(state, locale, t)`. `generatePackageBlob` then calls `t('title', { age, timezone, date, clinicCodeNote })` etc. Synchronous, browser-safe, no async-await on the next-intl side.
  
  **CHOSEN: (b) with refinement.** Actually, simpler: `buildReadme(state, locale, messages)` accepts a typed `messages: ReadmeMessages` parameter (a plain object of strings shaped like `{ title, patientLine, completedLine, intro, fileDescriptions, ehrInstructions, footer, clinicCodeDisclaimer }`). The component (13-04) constructs `messages` by calling `useTranslations('exportPackage.readme')` and grabbing the resolved strings. `buildReadme` is then a pure sync function that does ICU placeholder substitution on those strings (or — since the strings are pre-resolved by next-intl's `t()` — `buildReadme` just concatenates them with the dynamic values).
  
  **Even simpler architecture (FINAL):** `buildReadme` takes `(state: DiaryState, t: TranslationFunction)` where `TranslationFunction` is `(key: string, values?: Record<string, string|number>) => string` — the next-intl `t()` shape. The caller (13-04) supplies `t = useTranslations('exportPackage.readme')`. Inside `buildReadme`, every `messages/*.json` lookup is `t('title')`, `t('patientLine', { age, timezone })`, etc. — synchronous, browser-safe, no async.
  
  For `generatePackageBlob`, the orchestrator either takes `t` directly OR takes a `readmeText` string already composed by the component. **DECISION: `generatePackageBlob(state, locale, readmeText)` — the component composes the README via `buildReadme` before calling `generatePackageBlob`.** This is the cleanest separation: the package layer doesn't know about next-intl at all; the README is just a plain string passed in.

- **D-05 — README is ONE big i18n key per locale.** Per RESEARCH §'README Locale-Aware Authoring Decision': prose, not template data. The 6 keys are:
  ```json
  "exportPackage": {
    "readme": {
      "title": "My Flow Check — 3-Day Bladder Diary",  // BUT no em-dash per CLAUDE.md — replace with hyphen-minus or colon
      "patientLine": "Patient profile: age {age}, {timezone} timezone",
      "completedLine": "Diary completed: {date}",
      "intro": "This package contains 4 files for your records...",
      "fileDescriptions": "  01-clinical-report.pdf\n     The 7-page clinical PDF...\n\n  02-events.csv\n     Raw event data...\n\n  03-emr-bundle.fhir.json\n     FHIR R4 Bundle for direct upload...\n\n  README.txt\n     This file.",
      "ehrInstructions": "       Epic: Patient Chart -> Documents -> Upload...\n       Cerner / Allscripts / athenahealth: ...\n       Prompt Health: ...\n       Other EHR / paper: ...",
      "clinicCodeDisclaimer": "The clinic-code field in this patient's profile is a tracking reference (TRACKING type), not a medical record number. See Patient.identifier in 03-emr-bundle.fhir.json.",
      "footer": "Questions about what's in the diary or how to read the metrics?\nSee https://myflowcheck.com/learn/for-clinicians"
    }
  }
  ```
  
  Note: the CONTEXT.md draft uses an em-dash in the title. We REPLACE with a hyphen-minus or just space: `"My Flow Check: 3-Day Bladder Diary"` (colon) — keeps the no-em-dash rule across the codebase.
  
  ICU placeholders inside the strings: `{age}`, `{timezone}`, `{date}`, `{clinicCodeNote}`. The composition logic inside `buildReadme` is:
  ```ts
  return [
    t('title'),
    '',
    t('patientLine', { age: state.age ?? '—', timezone: state.timeZone }),  // — is em-dash, REPLACE
    t('completedLine', { date: today }),

    '',

  Wait — that `'—'` would emit an em-dash. Use `'unknown'` or empty string for missing age. Final: `age: state.age ?? 'unknown'`. (Or just emit nothing for the patient-line item if age is null. Cleanest: emit nothing — see updated logic in Task 2.)

- **D-06 — Conditional clinic-code disclaimer.** Per RESEARCH §Open Questions #4: when `clinicCode` is non-null AND validates against `[A-Za-z0-9-]{1,32}`, emit a paragraph explaining the TRACKING semantics.       state: DiaryState,
      locale: string = 'en',
      readmeText: string,
    ): Promise<{ blob: Blob; filename: string }> {

      // module graph, even within the already-lazy exportPackage chunk.

      zip.file('03-emr-bundle.fhir.json', bundleJson);

     * Trigger a browser download of the package zip.
     *
     * Used as the desktop fallback when navigator.share is unavailable.
     * Mirrors the generatePdf download pattern from src/lib/exportPdf/index.ts:71-87
     * exactly — 500ms cleanup timeout, link.style.display = 'none', document.body
     * appendChild + removeChild.
     */
    export async function downloadPackage(
      state: DiaryState,
      locale: string = 'en',
      readmeText: string,
    ): Promise<void> {
      const { blob, filename } = await generatePackageBlob(state, locale, readmeText);
      const url = URL.createObjectURL(blob);
      const link = document.createElement('a');
      link.href = url;
      link.download = filename;
      link.style.display = 'none';
      document.body.appendChild(link);
      link.click();
      setTimeout(() => {
        document.body.removeChild(link);
        URL.revokeObjectURL(url);
      }, 500);
    }
    ```
