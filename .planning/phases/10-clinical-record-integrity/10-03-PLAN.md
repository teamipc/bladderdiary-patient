### CRI-04 -- `removeWakeTime` does not recompute the FMV anchor

In `src/lib/store.ts:317-320`:
```ts
removeWakeTime: (dayNumber) =>
  set((s) => ({
    wakeTimes: s.wakeTimes.filter((w) => w.dayNumber !== dayNumber),
  })),
```

The action filters out the entry, but does NOT call `reassignMorningVoid` afterwards. Every sibling action that touches FMV-affecting state DOES call it:

- `addVoid` (line 198)
- `updateVoid` (lines 212, 214)
- `removeVoid` (line 224)
- `setBedtime` (lines 292, 294)
- `removeBedtime` (lines 301, 303)
- `setWakeTime` (line 314)

So the FMV invariant documented in `docs/TIME_MODEL.md` ("Every form-side correction must preserve this invariant" + "Wake-time and FMV gating") is broken on exactly one path: `removeWakeTime`. The UI typically prevents reaching this state (you cannot easily remove a wake while a bedtime is still set with the standard flow), but back-edits and the explicit wake-time edit flow can trigger it. Stale `isFirstMorningVoid: true` flags then leak into `calculations.ts:calcNocturnalVolume` (which uses `fmv?.timestampIso ?? wakeTime?.timestampIso` as the nocturnal-period end boundary), producing a non-zero nocturnal volume on what should be an undefined night.

**Fix shape:** Mirror `setWakeTime`'s structure exactly:
```ts
removeWakeTime: (dayNumber) =>
  set((s) => {
    const newWakeTimes = s.wakeTimes.filter((w) => w.dayNumber !== dayNumber);
    const voids = reassignMorningVoid(s.voids, dayNumber, s.startDate, s.bedtimes, newWakeTimes, s.timeZone);
    return { wakeTimes: newWakeTimes, voids };
  }),
```

Note: `reassignMorningVoid` already handles the no-wake case correctly (`src/lib/store.ts:41-42`):
```ts
const wakeEntry = wakeTimes.find((w) => w.dayNumber === dayNumber);
if (!wakeEntry) return allVoids; // No wake time -> don't change anything
```
So calling it after a removal with `newWakeTimes` that has no entry for this day is a documented no-op... which is the wrong behavior here. The current `reassignMorningVoid` early-returns the input voids unchanged, leaving any pre-existing `isFirstMorningVoid: true` flag in place.

**Sub-fix in `reassignMorningVoid`:** The early-return is correct for the "no wake set yet" case (do nothing; let `setWakeTime` set it later), but wrong for the "wake just removed" case (need to CLEAR any stale FMV flag for that day). The cleanest fix is to extend `reassignMorningVoid` so that when no wake entry exists for `dayNumber`, it CLEARS any `isFirstMorningVoid: true` flag on voids that belong to that day, then returns. This makes `removeWakeTime` work correctly via the same call site as `setWakeTime`.

Concrete edit to the helper at `src/lib/store.ts:33-73`:

```ts
function reassignMorningVoid(
  allVoids: VoidEntry[],
  dayNumber: number,
  startDate: string,
  bedtimes: BedtimeEntry[],
  wakeTimes: WakeTimeEntry[],
  timeZone?: string,
): VoidEntry[] {
  const wakeEntry = wakeTimes.find((w) => w.dayNumber === dayNumber);
  if (!wakeEntry) {
    // No wake time for this day: clear any stale FMV flag on voids
    // attributed to this day, then return. This makes removeWakeTime
    // a true inverse of setWakeTime for the FMV invariant.
    return allVoids.map((v) => {
      const vDay = getDayNumber(v.timestampIso, startDate, bedtimes, timeZone);
      if (vDay === dayNumber && v.isFirstMorningVoid) {
        return { ...v, isFirstMorningVoid: false };
      }
      return v;
    });
  }
  // ... (rest of existing implementation unchanged)
}
```

This is the minimal correctness fix. Any existing test that relies on the old "no wake = no-op" semantics would only break if it asserts FMV stays flagged after a wake removal, which is precisely the bug we're fixing.

### CRI-05 -- `observations.ts` caffeine-pattern detection includes Day 1

In `src/lib/observations.ts:74-85`:
```ts
const caffeineDrinks = drinks.filter((d) => CAFFEINE_TYPES.includes(d.drinkType));
if (caffeineDrinks.length >= 2) {
  const followed = caffeineDrinks.filter((d) => drinkFollowedByVoid(d, voids, 120)).length;
  if (followed >= 2 && followed / caffeineDrinks.length >= 0.5) {
    out.push({
      key: 'caffeineToBathroom',
      values: { count: followed },
    });
  }
}
```

`caffeineDrinks` aggregates across ALL diary days. The IPC rule (memory `ipc-calculations.md`, `docs/TIME_MODEL.md`, `calculations.ts`) is "Day 1 is excluded from 24HV / NPi / AVV (adaptation period)" because patient habits in the first day are unrepresentative. The summary observations should follow the same exclusion so the patient does not see "we noticed coffee tends to send you to the bathroom" based on a day the metrics explicitly ignore.

**Fix shape:** Filter caffeine drinks AND the voids array to exclude Day 1 events before counting. The Day-1-exclusion check goes through the canonical `getDayNumber` helper already imported at line 18, so the same delegated-day-attribution logic that powers `isVoidOnDay` (line 155-157) applies here.

Concrete edit:
```ts
const caffeineDrinks = drinks.filter(
  (d) =>
    CAFFEINE_TYPES.includes(d.drinkType) &&
    getDayNumber(d.timestampIso, state.startDate, state.bedtimes, state.timeZone) !== 1,
);
// Voids matched by drinkFollowedByVoid are within 120 min of a caffeine drink;
// the caffeine drink is already known to be Day 2 or 3, but the matched void
// could (in edge cases) be on Day 1 (a caffeine drink in the late evening of
// the calendar day Day 1 could be re-attributed to Day 2 by getDayNumber's
// bedtime-aware bump, and the followup void within 120 min sits on the
// calendar day after, still attributed to Day 2). For tight rigor, also
// gate the void: matched voids must NOT be Day-1-attributed.
const eligibleVoids = voids.filter(
  (v) => getDayNumber(v.timestampIso, state.startDate, state.bedtimes, state.timeZone) !== 1,
);
if (caffeineDrinks.length >= 2) {
  const followed = caffeineDrinks.filter((d) => drinkFollowedByVoid(d, eligibleVoids, 120)).length;
  // ... rest unchanged
}
```

The audit (ME-05) explicitly suggests "Same fix in `eveningFluids` / `morningFluids` if Day 1 should be excluded from those too (open product question -- defer to user before changing)." -- **DEFER** those per the audit's own recommendation. This plan ONLY filters Day 1 from the caffeine-to-bathroom observation. The fluid-timing buckets and the night-wakings counter and the consistency observation are out of scope for this plan and follow their existing semantics.
