# Plan 10-02: Eliminate the two remaining browser-local-time leaks — `NextStepBanner.tsx` (CRI-02) + `reminders.ts:anchorTimeLabel` (CRI-03)

<objective>
CRI-02 + CRI-03 — CODE-REVIEW.md findings **HI-02 (High)** + **HI-03 (High)**. Two regressions of the exact anti-pattern that Stabilization Phases 1–2 spent months eliminating: code outside `src/lib/utils.ts` calling `new Date().getHours()` / `new Date().setHours()` instead of the canonical timezone helpers. A patient whose stored diary timezone differs from their browser timezone (VPN, travel, misconfigured device) sees the wrong "next step" suggestion in the diary banner and the wrong reminder-time label in the Day 1 / Day 2 reminder cards. Neither is a clinical-calculation error, but both undermine the time-model invariant documented in `docs/TIME_MODEL.md` ("The browser tz is never trusted").

### CRI-02 — `NextStepBanner.tsx:79`

```ts
const hour = new Date().getHours();
if (hour >= 20) { ... bedtime branch ... }
else { ... keep-logging branch ... }
```

`new Date().getHours()` reads the **browser**'s local hour. For a patient stored in `Asia/Singapore` browsing from `America/New_York`, the banner thinks it's 4 AM SGT when it's 8 PM SGT (and vice versa), so the "Time for bed" prompt fires at the wrong wall-clock time.

**Fix:** Pull `timeZone` from `useDiaryStore` (already destructured in the component for other fields would be cleanest, but currently the component does NOT use timeZone — add it). Replace the `new Date().getHours()` call with `getHoursInTz(new Date().toISOString(), timeZone)` from `src/lib/utils.ts`. Mirror the exact pattern used by `observations.ts:93` (`getHoursInTz(d.timestampIso, state.timeZone)`) and `utils.ts:351` (`getHoursInTz(timestampIso, timeZone)` inside `getDayNumber`).

### CRI-03 — `reminders.ts:anchorTimeLabel`

```ts
export function anchorTimeLabel(anchor: MorningAnchor | null, locale: string): string {
  const d = new Date();
  d.setHours(anchor === 'coffee' ? 8 : 7, anchor === 'bathroom' ? 15 : 0, 0, 0);
  return d.toLocaleTimeString(locale, { hour: 'numeric', minute: '2-digit' });
}
```

`setHours()` operates in **browser**-local tz. `toLocaleTimeString()` without an explicit `timeZone` option also defaults to browser tz. The function takes only `(anchor, locale)` and is consequently incapable of representing the patient's stored tz. The displayed label (e.g. "8:00 AM") thus reads as 8 AM in whatever timezone the browser happens to be in — for a travelling / VPN'd patient stored in `Asia/Singapore` browsing from `America/New_York`, the card shows "8:00 AM" when the reminder will actually fire at 8 AM Singapore (which is 8 PM EST = 20:00 the prior day in the browser's tz).

The underlying ICS file uses "floating local time" per RFC 5545 and will fire correctly at 8 AM in whatever tz the phone is in at trigger time — **so the actual reminder is fine**. Only the UI **label** is wrong. The fix is to widen the function signature and use canonical helpers.

**Fix:** Add a required `timeZone?: string` parameter to `anchorTimeLabel`. Build a `today-at-anchor-hour-and-minute` ISO via `buildIsoForClockTimeInTz(new Date().toISOString(), h, m, timeZone)` and format with `formatTime(iso, locale, timeZone)` from `src/lib/utils.ts`. **Reuse the existing `anchorHourMinute` private helper** in `reminders.ts:16` instead of duplicating the magic-number table (`anchor === 'coffee' ? 8 : 7, anchor === 'bathroom' ? 15 : 0`). That helper already maps `null` / `'wake'` / `'coffee'` / `'bathroom'` to `{ h, m }`. Export it for reuse if needed (it's currently unexported but co-located).

Mirror the exact pattern used by `notifications.ts:96-108` after Phase 1 — the same fix has already been applied there for the actual reminder-scheduling path, which is why the live reminder fires correctly. This is the matching display-label fix.

### Why these two CRIs are bundled in one plan

They are **the same bug class** (browser-tz leak), they touch **adjacent surfaces** (the banner + reminder UI surfaces shown right next to each other on the diary page), and the fix shape is identical (replace browser-local helper with stored-tz helper). Bundling avoids two PRs that both regress-test the same anti-pattern. No file overlap with plan 10-01 (forms) or plan 10-03 (store + observations).

### What's not in scope

- The actual reminder-scheduling timezone fix in `notifications.ts` already landed in Stabilization Phase 1 (STAB-02). This plan only touches the **display label** (`anchorTimeLabel`) and the **banner suggestion** (`NextStepBanner`), not the underlying schedule.
- The `.ics` file generation (`buildDiaryIcs` in `reminders.ts:54`) uses "floating local time" intentionally — RFC 5545 contract is "fire at this clock time in whatever tz the phone is in at trigger time," which is what we want for travel-resilient morning reminders. **Do NOT change** the ICS path.
- No new i18n keys (the banner text is already keyed via `useTranslations('nextStep')` and the label is rendered by callers in their own keyed contexts).
