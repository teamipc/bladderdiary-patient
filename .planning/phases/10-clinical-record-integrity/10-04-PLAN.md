# Plan 10-04: Phase 10 integration verification + 6-locale regression Playwright spec

<objective>
Wave 2 verification spec covering all 5 CRI fixes from plans 10-01, 10-02, 10-03 at the integration layer. The unit-level guards in those plans pin the contracts at the source-code and store-action layers; this plan exercises the contracts END-TO-END through the live UI on a local production build.

Why this needs its own plan:
- The Phase 6 ConfirmDialog -> Discard flow is integration territory (BottomSheet animation + ConfirmDialog stacking + form unmount sequencing). Unit tests can prove "unmount without Save does not mutate the store" (10-01 Tasks 4-6), but they cannot prove "the user-visible Discard button in the ConfirmDialog wires through to that store-unchanged state."
- CRI-02 (`NextStepBanner`) integration requires rendering the actual DayPageClient with a real diary state plus tz manipulation -- the unit test in 10-02 Task 4 covers the branch selection logic but not the actual DOM banner in context.
- CRI-04 + CRI-05 integration coverage happens implicitly through the existing daily 6-locale production walkthrough -- this plan ALSO adds explicit spec coverage so the regression risk does not depend on a flaky daily-walkthrough cron.

Pattern reference: `e2e/phase5-chrome.spec.ts` (Phase 5 verification spec), `e2e/phase6-keyboard.spec.ts` (Phase 6 verification spec), `e2e/walkthrough.spec.ts` (existing 6-locale production walkthrough). All three run against the local static export (`out/`) via `npx serve out -l 4173`, NOT against production.

This plan also includes a human-verify checkpoint at the end so the developer can sanity-check the discarded-edit behavior across all 3 forms manually before declaring Phase 10 complete.

### Constraints inherited from Phase 5/6 verification patterns

- **Spec target:** `e2e/phase10-clinical-record-integrity.spec.ts` (one new file, no edits to existing specs).
- **Run model:** Build static export -> serve locally -> Playwright spec. Do NOT run against `https://myflowcheck.com` -- that is the daily walkthrough's job; this spec is a pre-merge gate.
- **Locale matrix:** 6 locales (en/fr/es/pt/zh/ar) -- per the CRI-01 ConfirmDialog copy and the CRI-02 NextStepBanner text rendering in each locale. CRI-03 (`anchorTimeLabel`) is exercised via Day1Celebration which is shown only once after Day 1 completion; cover this with a focused test on en + ar (LTR + RTL) rather than 6-way.
- **Verification approach:** assertion-driven, NOT screenshot-driven. Screenshots are flaky across CI runs; DOM + store assertions are deterministic.
</objective>
