# Plan 10-01: Remove autosave-on-unmount from `LogVoidForm` / `LogDrinkForm` / `LogLeakForm` + add discard/save-path regression tests

<objective>
CRI-01 — CODE-REVIEW.md findings **CR-01 (Critical)** + **CR-02 (Critical)** + **ME-04 (Medium)**.

Each of `LogVoidForm`, `LogDrinkForm`, `LogLeakForm` registers a cleanup `useEffect` whose return function calls `updateVoid` / `updateDrink` / `updateLeak` with the form's last in-memory state on unmount, gated only by `savedRef.current`. Phase 6's explicit Discard ConfirmDialog wiring in `DayPageClient.tsx` confirms the user wants their edits dropped — but then `setSheetMode(null)` unmounts the form, the cleanup fires, and the supposedly-discarded edit is silently persisted to the store. **The clinical record reflects state the patient explicitly chose to discard.** The ConfirmDialog copy "Your changes won't be saved" (key: `common.discardEntryMessage`) is contradicted by behavior.

This plan applies **approach (a) from CONTEXT § "Key planning questions" Q1 / CODE-REVIEW.md CR-01 § "Suggested fix"**: remove the autosave-on-unmount `useEffect` entirely from all 3 forms. The forms already have:
- An explicit `Save` button (`handleSave` → `addVoid` / `updateVoid` etc., gated on validation, sets `savedRef.current = true` then calls `onSave()`).
- An explicit `Discard` path (`DayPageClient.tsx:284 handleDiscardConfirm` → `closeSheet` → `setSheetMode(null)` → form unmounts).
- Working dirty-state tracking via `onDirtyChange` → `setActiveFormDirty` → `pendingClose` → ConfirmDialog.

Autosave is now dead UX. Removing it makes Save the only persistence path and aligns the implementation with the documented Phase 6 contract. The ESLint suppression (`react-hooks/exhaustive-deps`) is removed alongside (closes ME-04).

**Rationale for approach (a) over (b) "add `discardedRef` guard":**
- (a) reduces surface area; (b) adds a foot-gun ref that future contributors might forget to flip on a new dismiss path.
- (a) removes the empty-deps + ESLint-disabled cleanup (a documented architectural smell — see ME-04).
- (a) makes the persistence model explicit: Save = persist, anything else = drop. No "hidden" persistence flows.
- The CR-02 race condition (`pendingClose` race with `BottomSheet`'s `inert` toggling) is eliminated: with no autosave there is nothing to race.

**Unit-test mechanics (CONTEXT Q2 — RESOLVED in favor of Option (b) for primary coverage + Option (c) for integration smoke):**

CONTEXT proposed three test approaches:
- (a) Render `<LogVoidForm/>` standalone + invoke its `onSave`/ConfirmDialog parent flow → integration-test territory, fragile against next-intl + BottomSheet plumbing.
- (b) Render the form in isolation in jsdom and exercise the **direct contract**: simulate edit, unmount the form WITHOUT calling Save, assert store was NOT mutated. This is exactly what removing the autosave guarantees, and it's the contract DayPageClient's Discard path depends on.
- (c) Playwright integration test covering the full ConfirmDialog → Discard flow.

This plan uses **(b)** for the tight unit-level regression guard (one test file per form, three files total). The integration coverage from (c) is in plan 10-04 (Wave 2 verification spec). (b) gives:
1. Deterministic, fast vitest coverage that pins the contract per form.
2. Coverage that survives any future ConfirmDialog UI rework (the test is about store mutation, not UI choreography).
3. A separate test asserting the explicit Save path STILL mutates the store (no regression — addresses CONTEXT Q3 "Save-path regression coverage").
4. A static-code-level guard: assert the autosave `useEffect` body pattern no longer exists in any of the 3 forms. This is the strongest possible drift detector — if someone re-introduces an autosave cleanup, the test fails.

CONTEXT Q4 ("DayPageClient interaction") is resolved by inspection: `onDirtyChange` is fired from a separate `useEffect` driven by the `isDirty` memo. It is **independent** of unmount and continues to work unchanged. The autosave-on-unmount removal does not touch dirty-state plumbing — verified by reading the source (`LogVoidForm.tsx:142-144`, `LogDrinkForm.tsx:142-144`, `LogLeakForm.tsx:96-98`).

Note: the `formRef` + the per-render `useEffect` that maintains it (`LogVoidForm.tsx:123-126`, `LogDrinkForm.tsx:110-113`, `LogLeakForm.tsx:78-81`) exist **only** to feed the autosave cleanup with the latest state. With autosave gone, both `formRef` and its maintenance `useEffect` become dead code and MUST be removed alongside. `savedRef` also becomes useless (nothing reads it after the explicit save no-ops the redundant set) but is **left in place** — its current assignments in `handleSave` are harmless and removing it would inflate the diff and re-trigger the long `useCallback` dependency arrays. Document the keep-decision in code.

No new i18n keys. No new ConfirmDialog wiring. No store/types changes.
</objective>
