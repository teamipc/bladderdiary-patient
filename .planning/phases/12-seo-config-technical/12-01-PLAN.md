SEO-M3-01 (SEO-REVIEW.md T-4, High) тАФ `BreadcrumbList` JSON-LD on every learn article currently has two bugs in `src/app/[locale]/learn/[topic]/[slug]/page.tsx:122-127`:

1. **URL inconsistency.** Positions 1-3 use bare paths (`/`, `/learn`, `/learn/${topic}`) that 404 live; position 4 uses `article.urlPath` (locale-prefixed, 200). `BreadcrumbJsonLd` then runs `buildAbsoluteUrl()` on each item, producing `https://myflowcheck.com/` (200 soft-404 shell), `https://myflowcheck.com/learn` (404), `https://myflowcheck.com/learn/${topic}` (404), and `https://myflowcheck.com/<locale>/learn/<topic>/<slug>` (200) inside the same `BreadcrumbList`. Crawlers get inconsistent signal.

2. **Topic name casing.** Position 3 name is `topic.replace(/-/g, ' ')` тЖТ `"nocturia"` (lowercase, raw slug). SERPs render the lowercase pill. The pillar's `_pillar.mdx` frontmatter has a proper Title-Case `title` (e.g. nocturia/_pillar.mdx тЖТ `"Why You Wake Up to Pee at Night: Bladder vs Kidney"`).

This plan fixes both bugs at the assembly site (the article page) and uses the existing `getPillar(locale, topic)` helper in `src/lib/content.ts:258` to pull the localized pillar title. No change to `JsonLd.tsx` itself тАФ `BreadcrumbJsonLd` already prepends `SITE_URL` correctly via `buildAbsoluteUrl()`; the bug is upstream in the data assembly.

**Key design decisions (resolved here, not deferred to executor):**

- **D-01 тАФ Canonical URL form: locale-prefixed for ALL 4 positions.** Per CONTEXT ┬з"Key planning questions" Q2: position 4 already uses `/<locale>/learn/<topic>/<slug>`, the sitemap entries are all locale-prefixed, the canonical `<link>` on every article is locale-prefixed (`/${locale}/learn/...` per page.tsx:54). The bare-path model was the discarded `vercel.json` direction (deleted at Milestone 3 start). All 4 positions in the BreadcrumbList MUST be locale-prefixed to match the live canonical structure. Specifically:
  - Position 1 (Home): `/<locale>` (e.g. `/en`, `/fr`, `/ar`)
  - Position 2 (Learn): `/<locale>/learn`
  - Position 3 (Topic): `/<locale>/learn/<topic>`
  - Position 4 (Article): `article.urlPath` (already locale-prefixed)

- **D-02 тАФ Topic display name source: `getPillar(locale, topic)?.frontmatter.title` with slug-Title-Case fallback.** Per CONTEXT ┬з"Key planning questions" Q3, three options were surfaced:
  - (a) Topic frontmatter `_pillar.mdx` has `title` field тАФ read from there. **CHOSEN** because it's already the source of truth for the topic display name across the rest of the site (the pillar page's `<h1>`, the topic hub heading, the `articleSection` JSON-LD field).
  - (b) Hardcoded map in `src/lib/content.ts` тАФ REJECTED because it duplicates pillar.title and silently desyncs.
  - (c) Slug-to-Title-Case computed from `topic` slug тАФ RETAINED ONLY AS FALLBACK when `getPillar` returns null (defensive: a topic folder with no pillar shouldn't crash breadcrumbs). The fallback maps `'nocturia'` тЖТ `'Nocturia'`, `'bladder-irritants'` тЖТ `'Bladder Irritants'`.

  D-02 is locale-aware: the localized pillar (e.g. `content/articles/fr/nocturia/_pillar.mdx`) has a translated title (`"Pourquoi vous vous levez la nuit pour uriner : vessie ou rein"`), so the FR breadcrumb position-3 name shows in French, the AR in Arabic, etc. This is more aligned with i18n than a hardcoded English map.

- **D-03 тАФ UI Breadcrumbs component mirrors the JSON-LD.** The visible breadcrumb (rendered by `src/components/learn/Breadcrumbs.tsx`) currently shows `topic.replace(/-/g, ' ')` too. After this fix, both the visible breadcrumb AND the JSON-LD show the pillar title тАФ so SERP previews match the rendered page. We update the breadcrumbs array ONCE in page.tsx (the same object feeds both the visible UI and the JSON-LD via `breadcrumbs.map((b) => ({ name: b.label, url: b.href ?? article.urlPath }))`). No `Breadcrumbs.tsx` change needed; the component is data-driven.

- **D-04 тАФ Breadcrumb `href` for the visible component AND the JSON-LD URL: both are locale-prefixed.** The Breadcrumbs component uses `<Link>` from `@/i18n/navigation` which re-prepends the current locale via `localePrefix: 'as-needed'`. Currently the breadcrumb hrefs are bare (`/learn`, `/learn/${topic}`) тАФ for the visible UI this WORKS because `<Link>` adds the prefix back; but for the JSON-LD it's broken because `BreadcrumbJsonLd` does NOT re-prepend the locale, only `buildAbsoluteUrl()`. So we have a fork: keep `href` bare for the visible `<Link>` (which doesn't double-prefix per Phase 9's LP-01 ArticleCard pattern) BUT pass locale-prefixed `url` to the JSON-LD.

  Cleanest approach: build a parallel `jsonLdItems` array alongside the existing `breadcrumbs` array, with locale-prefixed URLs. The visible Breadcrumbs component keeps `href` bare so `<Link>` re-prefixes correctly. The JSON-LD gets explicit locale-prefixed URLs. This avoids changing the visible breadcrumb's Link behavior (which is already correct) while fixing the JSON-LD (which is broken).

- **D-05 тАФ Position 1 ("Home") URL.** SEO-REVIEW reports position 1 currently emits `https://myflowcheck.com/` (the bare root, soft-404 shell). After 12-02 ships the bare-root fix, the bare root WILL be substantive, but to stay consistent with the rest of the BreadcrumbList we still emit `/<locale>` (the canonical home for each locale). This matches what the hreflang map emits as `x-default` for any path: `localizedPath('en', '/')` = `/en`. Locale-consistency wins over "let position 1 stay bare so it benefits from 12-02's fix" тАФ that path would re-introduce mixed forms.

The visible Breadcrumbs component's "Home" link can keep `href: '/'` (renders as `<Link href="/">` which next-intl resolves to the current-locale home, e.g. `/fr/`) тАФ the user-facing UX is unchanged. Only the JSON-LD URL changes to the locale-prefixed form.
