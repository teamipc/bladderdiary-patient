SEO-M3-02 (SEO-REVIEW.md T-2, High) â€” Bare `/` route on production returns a ~8KB JS-only redirect shell with no `<title>`, no `<meta description>`, and no rendered body content. Googlebot reads soft content on the canonical homepage. The local build reproduces exactly this â€” `wc -c out/index.html` returns ~7330 bytes; `grep '<title>' out/index.html` returns nothing; the file is just `<noscript>` meta-refresh + a loading spinner div.

**Root cause investigation (resolved here, not deferred):**

I ran the build mechanics on the local repo to identify the gap before designing the fix (per CONTEXT Â§"Key planning questions" Q1):

1. `out/en.html` (80,590 bytes) â€” the actual EN landing page with `<title>My Flow Check | 3-Day Bladder Diary Tracker</title>`, full body content, JSON-LD schemas. This is what `src/app/[locale]/page.tsx` generates for `locale=en`.
2. `out/fr.html`, `out/es.html`, `out/pt.html`, `out/zh.html`, `out/ar.html` â€” all similarly substantive locale-specific landing pages.
3. `out/index.html` (7,330 bytes) â€” the JS-only shell rendered from `src/app/page.tsx`, which itself just renders `<LocaleRedirect />` (a `'use client'` component that calls `router.replace('/<locale>')` on mount).

This is the **next-intl `localePrefix: 'as-needed'` + Next.js 16 static-export gotcha** that CONTEXT Â§"SEO-M3-02" hypothesized. The plugin does NOT auto-emit the en-locale page to `out/index.html`; it emits `out/en.html` separately. The bare root remains the `<LocaleRedirect>` shell from `src/app/page.tsx`.

**Three fix strategies were evaluated** (per CONTEXT Â§"Key planning questions" Q1's three options):

- **(i) Reconfigure next-intl static-export so `out/index.html` IS the en-locale homepage.** REJECTED. `localePrefix: 'as-needed'` is a routing convention, not a build-output convention. The plugin does NOT have a documented option to emit the default-locale page to the bare-root file. Trying to force this through `next.config.ts` would mean changing the routing model (e.g. to `localePrefix: 'always'`, which would break every bare-path URL the audit confirms works â€” `/learn`, `/diary`, `/summary` rewrites in the Vercel dashboard). Out-of-scope-prevention from CONTEXT: "Don't break the recent SEO wins. Sitemap slim, author bios, audience landing expansion â€” preserve all that."

- **(ii) Vercel-side routing serves `out/en.html` as `/`.** REJECTED. Vercel routing for static exports respects the `out/` directory structure as-shipped. Any rewrite/route config that serves `out/en.html` content under the bare `/` URL re-introduces a canonical-vs-served-URL split (the canonical inside `out/en.html` says `https://myflowcheck.com/en`; if Vercel serves that file at `/` AND `/en`, the content is duplicated under two URLs â€” same problem the deleted `vercel.json` had, just inverted). Per memory `project_canonical_url.md` + audit T-1: routing is owned by the Vercel dashboard, not in-repo, and the canonical model is locked to locale-prefixed URLs. A routing-layer fix here re-litigates a decision Milestone 3 setup already settled.

- **(iii) Post-build copy step that overwrites `out/index.html` with `out/en.html`.** CHOSEN. This is purely a build-output adjustment â€” no routing change, no canonical change. The two files become byte-identical:
  - Both have the same `<title>My Flow Check | 3-Day Bladder Diary Tracker</title>`
  - Both have the same `<link rel="canonical" href="https://myflowcheck.com/en"/>` â€” Googlebot sees a canonical that points away from the apex, consolidating signal to `/en`. The apex no longer competes as a distinct URL because it's an exact byte-copy of the canonical target.
  - Both have the same hreflang map.
  - Both have the same rendered EN landing content (substantive HTML, JSON-LD, body).
  - Crawlers fetching `https://myflowcheck.com/` will see substantive content and a `rel=canonical` to `/en`, which is the textbook canonicalization pattern for bare apex â†’ canonical-locale.

Why (iii) is correct and the canonical-conflict concern is resolved: Google's documented behavior on `<link rel=canonical>` is to consolidate ranking signals to the canonical URL when the bare URL has the canonical tag pointing elsewhere. The bare `/` URL having BYTE-IDENTICAL content with a self-pointing-to-canonical tag is exactly the "duplicate page with canonical" pattern Google handles cleanly â€” far better than the current "soft 404 / JS-only redirect" pattern which the audit flags as the worst-case rendering signature.

**Key design decisions (resolved here):**

- **D-01 â€” Copy strategy: `out/en.html` â†’ `out/index.html` byte-for-byte.** No re-canonicalization, no rewriting of the `<link rel=canonical>` to point to `/`. The bare `/` URL stays in Google's index as a duplicate of `/en` (with canonical pointing to `/en`), and Google consolidates signal correctly. This is the lowest-risk option.

- **D-02 â€” Live the copy script alongside `scripts/check-search-console.mjs` + `scripts/generate-icons.mjs`.** `scripts/` is the established place for Node CLIs in this repo. Use the `.mjs` extension (matches the other scripts; the repo's Node 23+ environment treats `.mjs` as ESM unambiguously).

- **D-03 â€” Wire into `package.json`'s `build` script, NOT a separate `postbuild` script.** Two reasons:
  1. The repo's `package.json` does NOT have a `postbuild` hook today â€” adding one as a separate script means relying on npm's lifecycle-hook machinery, which is opaque to anyone glancing at the package.json. Explicit `&&` chaining in `build` is more discoverable.
  2. Vercel's build pipeline runs `npm run build`. If the copy step is in `postbuild`, Vercel runs it automatically; that works. If it's in `build`, Vercel also runs it. Either works; explicit chaining wins on legibility.

  The new `build` script: `next build && node scripts/post-build-copy-en-root.mjs`.

- **D-04 â€” Idempotency + validation guard.** The copy script:
  - Reads `out/en.html`; if it doesn't exist OR is < 50KB (sanity floor â€” actual file is 80KB+), exit 1 with a clear error message ("aborting copy: out/en.html is missing or suspiciously small â€” did the build fail?"). This prevents a corrupted build from silently re-instating the JS-only shell.
  - Copies `out/en.html` to `out/index.html` via `fs.cpSync`.
  - Logs the source size + destination size to stdout.
  - Exits 0 on success, 1 on any failure.

  The script is idempotent: running it twice (or on an unchanged `out/`) produces the same output, no side effects beyond the overwrite.

- **D-05 â€” Test scope: unit test that runs the copy script end-to-end against a fixture.** The vitest unit test creates a tmpdir, writes a fake `out/en.html`, runs the script (via `spawnSync`), then asserts `out/index.html` exists with matching content + size. This catches:
  - Script silently failing
  - Script copying the wrong file
  - File-size guard triggering on a legitimate build
  - Future regression that re-introduces the JS-only shell

  The test does NOT depend on a real `npm run build` (slow, requires the full Next.js toolchain to install â€” too heavy for a unit-test loop). The build-time integration is verified by plan 12-04's e2e check that asserts `out/index.html` shape after a real build.

- **D-06 â€” `src/app/page.tsx` + `src/app/LocaleRedirect.tsx` are NOT modified by this plan.** The post-build copy overwrites `out/index.html` after Next.js emits it. The LocaleRedirect spinner remains as a defensive layer (if someone runs `next build` without running the copy step, the bare root falls back to the old JS-only shell â€” degraded but not broken; the build script wiring ensures normal builds always include the copy). Modifying `src/app/page.tsx` to render the EN landing content directly would (a) duplicate the LandingContent component, (b) tangle the i18n-detection-redirect logic with the actual EN content rendering, (c) re-litigate the next-intl routing model. The post-build copy is surgical.
